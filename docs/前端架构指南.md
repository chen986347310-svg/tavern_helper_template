# 前端架构指南

> 面向未来开发者的架构导航文档, 快速理解项目结构和添加功能的方式

## 当前权威入口结论（2026-05-22）

- **主数据入口**：组件通过 `useDataStore().data` 读取/写入 MVU 映射数据；跨页面读写必须使用 `store.data` 活引用，不要长期缓存旧对象。
- **交互入口**：有叙事意义的 UI 操作统一走 `composables/usePendingAction.ts`，写入 `系统.待处理交互`，等待下一楼层 AI 承接。
- **即时入口**：商城扣费、背包数量、装备/卸下、按钮动效等可在当前楼层即时更新，但剧情结果不能由前端伪造。
- **验收入口**：前端构建通过不等于闭环通过；必须用 CDP 确认最新楼层输出 `<UpdateVariable><JSONPatch>` 且 MVU 最新变量实际回流。
- **术语入口**：统一使用 `牝奴`、`牝阴诀`、`太极/命轮`、`莲灯前苑 / 醉玉小筑 / 绮梦幽阁`。

## 添加新功能指南

### 新增 NPC

需要同步修改 3 处:
1. _variables.scss: $npc专属色 map 加新颜色
2. schema.ts: z.enum 中加新名字
3. guards.ts: 攻略链数组加新名字

可选: assets/avatars/ 加新图片, 世界书/加新 NPC 设定

### 新增页面

需要联动 3 处:
1. App.vue: v-if 链加新条件
2. PageNav.vue: tabs 数组加新 tab
3. useNavigation: 无需改动 (已支持任意 tab 名)

### 新增数据字段

1. schema.ts: z.object 中加新字段
2. store 自动响应 (defineMvuDataStore 基于 Schema)
3. 组件直接通过 useDataStore().data.xxx 读取

### 新增 SCSS mixin

1. _mixins.scss: 定义新 mixin
2. 组件中 @use 'mixins' as * 后 @include 新 mixin

## 图片资源规范

| 类型 | 路径 | 规格 |

| 仙奴版 | assets/avatars/{npc名}_fallen.png | 同上 |
| 显示 | object-fit: cover + object-position: center 20% | |
| 容器 | 响应式宽度(百分比), 跟随 iframe 容器 | |
| 回退 | @error 时显示首字占位符 | |
  | 路径 | 相对路径(无开头/), 适配iframe子目录 | |

  ### Webpack 图片构建

  webpack.config.ts 中为 avatars 目录配置了专用 asset/inline 规则:

  ```
  include: path.resolve(import.meta.dirname, 'src/雌堕合欢宗/界面/assets/avatars')
  type: 'asset/inline'
  ```

  - 图片以 base64 data URI 内联进 JS bundle, 最终由 HtmlInlineScriptWebpackPlugin 内联进单个 HTML
  - 无外部文件输出, 酒馆 iframe 可直接渲染
  - 产物大小约 6 MB (10 张 512x512 压缩 PNG 的 base64, 原图已从 1254x1254 压缩 83%)

## 关键设计决策

| 决策 | 结论 |
|------|------|
| 导航 | ref + v-if, 不引入 vue-router |
| 数据 | Pinia store + MVU 双向同步 |
| 展开动画 | flex-wrap + width 过渡, 0.35s ease-in-out |
| 收起动画 | 0.55s ease-out (比展开慢 60%), 纯 opacity 动画无跳变 |
| 头像渐融 | ::before 伪元素叠加层, opacity 平滑过渡 |
| 仙奴切换 | 数据驱动, 无过渡动画 |
| 测试 | @vue/test-utils + happy-dom + vi.mock |
| 调试 | 单例 composable, Ctrl+Shift+D, 所有模式可用 |

## v4 交互闭环架构补充（2026-05-22）

### 新增关键 composable

| 文件 | 职责 |
|------|------|
| `composables/usePendingAction.ts` | 将购买、装备、卸下、使用物品、灵识窃取等 UI 操作写入 `系统.待处理交互` |
| `composables/useStatusText.ts` | 将好感度/攻略值数值翻译为玩家可见状态语 |
| `data/whispers.ts` | 提供场景感知心音文案库，供前端测试和默认展示逻辑引用 |

### store.data 活引用原则

`defineMvuDataStore` 会轮询 MVU 并可能整体替换 `store.data`。组件和 composable 不应在 setup 阶段长期缓存 `const data = store.data` 后反复读写，涉及跨页面一致性的读写应尽量读取 `store.data` 活引用，避免出现主栏和商城/背包读到不同对象的短暂分叉。

### 待处理交互原则

- `usePendingAction` 只负责入队，不触发 AI 回复。
- 当前楼层可以即时做库存/扣费等 UI 所需变更，但必须把有叙事意义的动作写入队列供下一楼层承接。
- AI 已承接的交互必须在同一次 `<JSONPatch>` 中清空 `系统.待处理交互`。

### 场景枚举原则

前端当前只接受三类 UI 场景：

- `莲灯前苑`
- `醉玉小筑`
- `绮梦幽阁`

正文地点可以更细，但 MVU 场景字段必须映射到上述三者。新增 UI 场景时，至少同步修改 `schema.ts`、`schema.json`、场景守卫、页面测试、变量列表与世界书变量规则。

### 实机验证原则

前端功能完成后，不只看页面：

1. 用 CDP 确认状态栏 iframe 已渲染。
2. 读取 `Mvu.getMvuData({ type: 'message', message_id: 'latest' })`。
3. 确认 AI 输出有标准 `<UpdateVariable>` / `<JSONPatch>`。
4. 确认变量实际回流后，再判定 UI 显示是否正确。
### CDP 闭环验收流程

1. 确认状态栏 iframe 已渲染，例如 `TH-message--<楼层号>--0`。
2. 触发一次 UI 交互，确认 `系统.待处理交互` 出现对应动作。
3. 用户发送下一条消息后，读取最新楼层正文，必须包含 `<UpdateVariable>` 与 `<JSONPatch>`。
4. 执行 `Mvu.getMvuData({ type: 'message', message_id: 'latest' })`，确认 `soul_whisper`、场景或目标变量实际回流。
5. 确认 `系统.待处理交互` 已清空，避免下一楼层重复执行。
