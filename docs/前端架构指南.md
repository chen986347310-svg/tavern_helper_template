# 前端架构指南

> 面向未来开发者的架构导航文档, 快速理解项目结构和添加功能的方式

## 当前权威入口结论（2026-05-24）

- **主数据入口**：组件通过 `useDataStore().data` 读取/写入 MVU 映射数据；跨页面读写必须使用 `store.data` 活引用，不要长期缓存旧对象。
- **交互入口**：购买、装备、卸下、使用物品、追查风声等会推进剧情的 UI 操作统一走 `composables/usePendingAction.ts`，写入 `系统.待处理交互`，等待下一楼层 AI 承接。
- **即时入口**：商城扣费、背包数量、装备/卸下、按钮动效等可在当前楼层即时更新，但剧情结果不能由前端伪造。
- **心音入口**：普通命盘/心音点击只写 `系统.当前聚焦心声NPC`，不默认写入 `灵识窃取` 待处理交互；旧楼层残留的 `灵识窃取` 仍由世界书兼容承接并清空。
- **验收入口**：前端构建通过不等于闭环通过；涉及 AI/MVU 回流时，需要用 CDP 或用户实机确认最新楼层输出 `<UpdateVariable><JSONPatch>` 且 MVU 最新变量实际回流。
- **术语入口**：统一使用 `牝奴`、`牝阴诀`、`太极/命轮`、`莲灯前苑 / 醉玉小筑 / 绮梦幽阁`。
- **图鉴入口**：`GalleryPage.vue` 是 `因果秘录` 只读归档层；`心音册 / 风声卷` 以时间线方式回看 MVU 数据，不写 `系统.待处理交互`。
- **叙事提示入口**：可增长的规则库禁止蓝灯常驻；采用“当前状态短摘要 + 绿灯扫描文本 + 世界书完整规则库”的按需注入架构控制 token。

## 当前架构总览

当前前端不是独立游戏状态机，而是 SillyTavern MVU 变量的可视化与交互暂存层。前端负责展示 `stat_data`、执行当前楼层可立即确认的 UI 变化、把需要 AI 叙事承接的动作写入 `系统.待处理交互`；下一楼层由 AI 根据世界书规则写出 `<UpdateVariable><JSONPatch>`，再由 MVU 回流到 Pinia store。

### 四层职责边界

| 层级 | 权威文件 | 职责 | 禁止事项 |
|------|----------|------|----------|
| MVU Schema | `src/雌堕合欢宗/schema.ts` | 定义变量结构、枚举、默认可读字段 | 组件私自创造未入 schema 的长期字段 |
| Pinia 映射 | `src/雌堕合欢宗/界面/store.ts` | 通过 `defineMvuDataStore(Schema, { type: 'message' })` 提供 `store.data` | 长期缓存旧 `store.data` 对象后跨页面读写 |
| UI 交互 | `src/雌堕合欢宗/界面/**` | 展示状态、扣费/库存/装备即时反馈、写入待处理交互 | 前端伪造剧情结果或直接调用 AI |
| AI 规则 | `src/雌堕合欢宗/世界书/**` + `src/雌堕合欢宗/脚本/**` | 约束 AI 如何承接交互、回写变量、动态注入当前摘要 | 把持续增长的完整规则表挂为蓝灯常驻 |

### AIRP 闭环

AIRP 是当前开发默认边界：Action 由 UI 记录，Interpretation 由 AI 叙事承接，Result 由 `<JSONPatch>` 回写，Presentation 由前端重新读取 MVU 显示。

| 阶段 | 责任方 | 当前例子 |
|------|--------|----------|
| Action | 前端 | 商城购买、背包装备/卸下、使用物品、主页追查风声写入 `系统.待处理交互` |
| Interpretation | AI + 世界书 | 读取动态场景、风声、心音、道具生命周期、服装叙事规则 |
| Result | AI + MVU | 输出 `<UpdateVariable><JSONPatch>`，更新 `系统.场景上下文`、`系统.心音回响`、`道具.装备` 等字段，并清空已承接队列 |
| Presentation | 前端 | `HomePage`、`SystemBar`、`BackpackPage`、`GalleryPage` 根据新变量重新渲染 |

## 添加新功能指南

### 新增 NPC

需要同步修改 3 处:
1. _variables.scss: $npc专属色 map 加新颜色
2. schema.ts: z.enum 中加新名字
3. guards.ts: 攻略链数组加新名字

可选: assets/avatars/ 加新图片, 世界书/加新 NPC 设定

### 新增页面

需要联动 3 处:
1. App.vue: v-if 链加新条件
2. PageNav.vue: tabs 数组加新 tab
3. useNavigation: 无需改动 (已支持任意 tab 名)

### 新增数据字段

1. schema.ts: z.object 中加新字段
2. store 自动响应 (defineMvuDataStore 基于 Schema)
3. 组件直接通过 useDataStore().data.xxx 读取

### 新增 SCSS mixin

1. _mixins.scss: 定义新 mixin
2. 组件中 @use 'mixins' as * 后 @include 新 mixin

## 动态叙事提示注入原则（2026-05-24）

当某类叙事规则可能随系统扩展持续增长时，不得把完整规则表长期挂为蓝灯常驻。当前服装商城 v3 采用以下分层方案，后续风声、心音、场景、道具效果等系统也应优先复用此模式：

| 层级 | 发送方式 | 作用 | Token 策略 |
|------|----------|------|------------|
| 当前状态短摘要 | 脚本动态 `injectPrompts`，`position: 'in_chat'` | 只告诉 AI 本轮真正相关的状态，例如玩家与在场 NPC 当前装备 | 每轮几十到数百 token，且无相关状态时不注入 |
| 绿灯扫描文本 | 脚本动态 `injectPrompts`，`position: 'none'` + `should_scan: true` | 用逻辑名、显示名、交互类型激活对应世界书资料库 | 不直接进入正文提示，只参与世界书触发 |
| 完整规则资料库 | 世界书绿灯条目 | 保存完整描写重点、边界、场景变化与专属规则 | 按需触发，禁止整表蓝灯常驻 |

### 服装/禁器叙事当前实现

服装叙事规则的完整表位于 `src/雌堕合欢宗/世界书/道具/服装叙事规则.yaml`，在 `src/雌堕合欢宗/index.yaml` 中作为绿灯资料库挂载。关键字只保留触发锚点，例如 `服装叙事规则`、`道具.装备`、`装备道具`、`卸下`。

禁器叙事规则的完整表位于 `src/雌堕合欢宗/世界书/道具/禁器叙事规则.yaml`，同样作为绿灯资料库挂载。禁器动态注入只携带当前玩家与在场 NPC 身上的禁器短摘要，以及用于扫描的逻辑名、显示名、器阶、作用部位和交互类型；完整描写重点仍留在世界书中按需触发。

### 商城场景商品边界（2026-05-24）

商城里的场景商品采用“令牌/钥匙/通行权”表达，避免把地点本身当作普通背包物品出售。前端商品卡展示 `显示名`，例如 `合契浴令`、`暖炉试香令`；购买、解锁与存档仍使用真实场景名，例如 `阴阳池`、`药庐暖阁`。

当前版本不新增 schema 字段。购买成功后只把真实场景名写入 `场景.已解锁`，不写入 `道具.拥有`，也不把令牌显示名写入解锁列表。这样可以保持旧存档兼容，并让动态场景系统继续围绕真实地点工作。

场景叙事规则的完整表位于 `src/雌堕合欢宗/世界书/道具/场景叙事规则.yaml`，在 `src/雌堕合欢宗/index.yaml` 中作为绿灯资料库挂载。规则只记录场景用途、关联 NPC、适配道具、进入理由和风声钩子；AI 承接时应根据 `场景.已解锁`、当前 NPC、装备、丹药、禁器与风声自然调用，不应机械复述商城说明。

### 商城剧情商品边界（2026-05-24）

商城剧情商品采用“剧情信物 / 线索钥匙”表达。前端展示信物名，变量层写入剧情线逻辑名；购买后只写入 `剧情.已解锁`，不写入 `道具.拥有`。第一版不新增 schema 字段，不记录剧情分段进度；AI 后续根据已解锁剧情线、当前场景、风声、心音和 NPC 状态自然承接，不一次性把秘密讲完。

剧情叙事规则完整表位于 `src/雌堕合欢宗/世界书/道具/剧情叙事规则.yaml`，在 `src/雌堕合欢宗/index.yaml` 中作为绿灯资料库挂载（顺序 12，紧随场景叙事规则）。资料库为每条剧情钥匙记录显示名、关联 NPC、秘密主题、推荐场景、关联道具、解锁提示与叙事重点；AI 收到购买剧情钥匙的待处理交互后，先承认信物入手或线索浮现即可，**不打断当前正在发生的剧情**，等到自然过渡点（场景切换、当前剧情段落收束、玩家主动靠近关联 NPC、风声或心音触发）再让伏笔生效。第一版共 5 条剧情钥匙：白芷旧誓线、苏芸错炉线、纪兰禁录线、沈月秋失衡线、柳素衣命契线。

### 商城特殊道具事件种子边界（2026-05-24）

商城特殊板块定位为**显式系统级钩子**——跟剧情/风声/心音的"隐性铺垫"对偶，玩家明确知道按下去会发生什么。整个板块走**事件种子**模型，不走"状态机 buff"：购买在 `系统.待处理交互` 写一条事件种子，AI 立即承接戏剧场景，演到自然收束。持久后果由 AI 作为剧情结果自行写入 JSONPatch，**前端不预设倒计时、不靠后端 buff 清理、不动态注入维持状态**。这是为了保持对话式 AIRP 的内核——AI 是叙事伙伴而不是执行引擎。

第一版共 5 件，前两件是原有锚点，后三件从世界书设定原生挖出：

- **时间延长**（自用消耗，10000 灵石）：节奏修正器，把欲海定位倒计时推后。
- **改变阵法**（购买即生效，500000 灵石）：命运触发器，触发 GOOD END。
- **欲海遮蔽符**（自用消耗，8000 灵石）：青云宗遗散的气息遮蔽符箓，对应世界书设定原文「灵石可购买气息遮蔽换取额外天数」。使用时短时遮蔽气息，欲海核心暂时失嗅；窗口长短由 AI 决定。
- **欲海回声**（购买即生效，30000 灵石）：青铜残钟贴在阴阳池阵眼上接通欲海核心，让玩家听见镇压在内的青云道人残存意识。世界真相揭示级剧情爆点；青云道人可能复仇请求、救赎请求、冷漠拒答或揭示玩家自身因果根源。
- **投欲钥**（购买即生效，100000 灵石）：与「改变阵法」对偶的**主动 BAD END 通道**。玩家主动激活气息让欲海核心定位自己，触发柳素衣依牝奴诀仪轨投阴阳池→粉碎练阳诀→强灌牝奴诀。AI 必须在剧情中写入 `系统.阶段='牝奴期'`；这不是失败、不是被抓，是玩家承认自己是青云道人二代、主动接受被改造的命运。

特殊道具事件规则完整表位于 `src/雌堕合欢宗/世界书/道具/特殊道具事件规则.yaml`，在 `src/雌堕合欢宗/index.yaml` 中作为绿灯资料库挂载（顺序 13，紧随剧情叙事规则）。规则关键字包含 `欲海核心 / 青云道人 / 九天驭奴诀 / 御奴诀 / 阴阳池 / 气息遮蔽` 等世界观锚点，跟 `欲海与印记` 世界书条目咬合触发。

当前装备摘要由 `src/雌堕合欢宗/脚本/服装叙事注入/outfitPrompt.ts` 生成，并由 `src/雌堕合欢宗/脚本/后端校验/index.ts` 在 `GENERATE_BEFORE_COMBINE_PROMPTS` 前刷新注入。脚本会先清理旧注入，再按最新 MVU 重新注入，生成结束或切换聊天时清空，避免提示残留到无关楼层。该摘要只读取：

禁器摘要由 `src/雌堕合欢宗/脚本/服装叙事注入/contrabandPrompt.ts` 生成，与服装摘要在同一个 `GENERATE_BEFORE_COMBINE_PROMPTS` 钩子中刷新，在 `GENERATION_ENDED` 与 `CHAT_CHANGED` 中统一清理。禁器摘要会过滤 `NSFW道具` 以外的装备，避免服装误入禁器提示。

- `道具.装备.玩家`
- `系统.场景上下文.在场NPC`
- 在场 NPC 对应的 `道具.装备[NPC名]`
- 本轮 `系统.待处理交互` 中的 `装备道具` / `卸下`

离场 NPC 的装备不进入当前短摘要，但仍保留在 MVU 中；当 NPC 再入场，脚本会重新把其当前装备摘要注入给 AI。若当前玩家与在场 NPC 均无装备，则不注入任何服装提示。当前实现只有存在可见装备行时才生成可读摘要和扫描文本；单纯卸下到全员无装备时，下一楼层依靠 `系统.待处理交互` 与世界书变量规则承接卸下结果。

### 约束

- 完整规则表禁止蓝灯常驻，避免后续系统叠加后 token 爆炸。
- 动态短摘要只描述当前可见或本轮交互相关状态，不应携带全部服装百科。
- `position: 'none'` 的扫描注入只用于激活绿灯世界书，不应包含需要 AI 直接阅读的长说明。
- 修改 `脚本/后端校验/index.ts` 或其依赖的注入模块后，必须按 esbuild 独立脚本流程重建 `dist/backend_validate.js`；普通 `npm run build` 不会刷新该产物。
- 验收时除单元测试外，还应检查 `dist/backend_validate.js` 是否包含注入标识，例如 `hehuan-current-outfit-summary` 与 `hehuan-current-contraband-summary`。

## 关键设计决策

| 决策 | 结论 |
|------|------|

### store.data 活引用原则

`defineMvuDataStore` 会轮询 MVU 并可能整体替换 `store.data`。组件和 composable 不应在 setup 阶段长期缓存 `const data = store.data` 后反复读写，涉及跨页面一致性的读写应尽量读取 `store.data` 活引用，避免出现主栏和商城/背包读到不同对象的短暂分叉。

### 待处理交互原则

- `usePendingAction` 只负责入队，不触发 AI 回复。
- 当前楼层可以即时做库存/扣费等 UI 所需变更，但必须把有叙事意义的动作写入队列供下一楼层承接。
- 普通命盘点击、图鉴心音聚焦、图鉴风声聚焦属于阅读状态，只写 `系统.当前聚焦心声NPC` / `系统.当前追查风声ID`，不得写入待处理交互。
- 道具类交互的 `道具` 字段始终保留逻辑名作为主键；若道具存在显示名、禁器器阶、作用部位或 AI 短提示，前端可在同一条待处理交互中附加 `道具显示名`、`器阶`、`作用部位`、`AI短提示`，供下一楼层自然承接。普通道具没有叙事元数据时保持旧载荷。
- AI 已承接的交互必须在同一次 `<JSONPatch>` 中清空 `系统.待处理交互`。

### P2 UI 数据与风格边界

P2 牝奴期主控台组件不得直接调用 AI、不得本地编排剧情、不得写入非 MVU 状态。组件只能读取 Pinia/MVU store；玩家点击产生的意图统一进入 `系统.待处理交互`，由动态注入层和世界书规则约束 AI 承接。

P2 视觉不应和 P1 割裂。牝奴期界面应继承攻略期的卷轴底色、透明阵列、铭文字体、太极/樱花圆环、弱风声涟漪等基础语法，再通过牝印暗纹、朱批命令、日课、羞名风声和烙名痕迹逐步污染原有界面。`SystemBar.vue` 是统一状态栏，不应为 P2 另做突兀的独立皮肤；P2 只替换状态语义，保留 P1 的布局骨架和动效节制。

`Phase2Page.vue` 负责展示牝印核心、日课、支配者、听风羞名和烙名标签；`SystemBar.vue` 只提供常驻弱反馈，例如牝印圆环、当前日课、当前命令和羞名风声涟漪。状态栏不得重复显示风声正文，避免和主控台的追查入口形成双入口误解。

### 场景枚举原则

前端当前只接受三类 UI 场景：

- `莲灯前苑`
- `醉玉小筑`
- `绮梦幽阁`

正文地点可以更细，但 MVU 场景字段必须映射到上述三者。新增 UI 场景时，至少同步修改 `schema.ts`、场景守卫、页面测试、`世界书/变量/变量列表.yaml` 与 `世界书/变量/变量更新规则.yaml`；如需要重新导出结构，再使用 `dump_schema.ts` 或变量结构脚本更新产物。

### 验证原则

前端功能完成后，不只看页面。涉及 AI/MVU 回流、待处理交互、动态提示注入时，需要做完整闭环；纯样式或只读文档变更可由单元测试、构建检查或人工验收覆盖。

1. 用 CDP 确认状态栏 iframe 已渲染。
2. 读取 `Mvu.getMvuData({ type: 'message', message_id: 'latest' })`。
3. 确认 AI 输出有标准 `<UpdateVariable>` / `<JSONPatch>`。
4. 确认变量实际回流后，再判定 UI 显示是否正确。

### 灵石经济边界（2026-05-24）

新版商城扩容后，灵石经济采用“收入放宽、终局保重”的口径。初始 `系统.灵石` 为 `2000`，让玩家开局即可购买低阶服装、禁器或丹药；常规 NSFW 收益公式为 `NPC境界系数 × 攻略值增量 × 4`，对应实现位于 `src/雌堕合欢宗/界面/guards.ts` 的 `calculate灵石获取`。

### CDP 闭环验收流程（需要实机时）

1. 确认状态栏 iframe 已渲染，例如 `TH-message--<楼层号>--0`。
2. 触发一次 UI 交互，确认 `系统.待处理交互` 出现对应动作。
3. 用户发送下一条消息后，读取最新楼层正文，必须包含 `<UpdateVariable>` 与 `<JSONPatch>`。
4. 执行 `Mvu.getMvuData({ type: 'message', message_id: 'latest' })`，确认 `soul_whisper`、`心音回响`、场景或目标变量实际回流。
5. 确认 `系统.待处理交互` 已清空，避免下一楼层重复执行。

### 待处理交互清空验收脚本

AI 已承接购买、使用、装备、卸下、追查风声或旧版灵识窃取后，运行：

```bash
node cdp-pending-action-clearance-check.mjs
```

脚本只读取最新楼层 `系统.待处理交互` 并判定是否为空；不得在验收脚本或 `validate.ts` 中自动清空队列，避免吞掉未被 AI 承接的动作。

## 丹药系统体系（2026-05-24）

丹药分为 `临时丹药`、`永久丹药`、`仙奴丹`。临时丹药制造短期场景和剧情失衡，其中温药用于试探，烈药用于推动护法、质问、风声和心音变化；永久丹药固化体态、生理、思维和羞耻机制；仙奴丹只在仙奴/牝奴后期开放，直接服务性、取悦、身体可玩乐化和归属展示。
