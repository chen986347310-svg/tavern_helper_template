# PRD: 雌堕合欢宗 后端校验与测试系统

## Problem Statement

雌堕合欢宗是一个 SillyTavern 角色卡项目，包含复杂的游戏机制：好感度系统、攻略链、粘滞计数、装备门槛、双阶段切换等。AI 输出的变量更新（通过 MVU 框架的 <update> 块）可能产生不合法的数值（溢出、负数、违反攻略链顺序等），需要后端校验来兜底修正。

此前缺少：
1. 后端校验逻辑与事件处理器耦合，无法独立测试
2. 没有自动化测试覆盖游戏规则
3. 粘滞阈值在 guards（区分行为类型）和后端校验（统一阈值）之间存在不一致
4. 剩余天数没有取整，导致显示小数（如 29.998 天）

## Solution

构建三层测试体系，确保游戏规则在实际游玩中正确执行：

- L1 纯函数单元测试：提取后端校验为纯函数 validateVariables()，配合 guards 纯函数，实现 100% 场景覆盖
- L2 CDP 注入集成测试：通过 Chrome DevTools Protocol 在酒馆页面直接注入变量变更，验证从 MVU API 到 Zod schema 到后端校验到最终变量状态的完整链路
- L3 实际消息端到端测试：在酒馆中真实发送消息，AI 回复后验证状态栏渲染和变量正确性

## User Stories

1. 作为玩家，我希望好感度增加后数值正确保留，以便我能追踪与 NPC 的关系进展
2. 作为玩家，我希望非当前攻略链 NPC 的好感度也能自由提升，以便我可以在等待攻略进度时提前培养关系
3. 作为玩家，我希望好感度达到 30 后才能提升攻略值，以便亲密行为有合理的情感基础
4. 作为玩家，我希望好感度低于 30 时攻略值自动归零，以便防止数值错乱
5. 作为玩家，我希望攻略值只能增长不能倒退，以便我的攻略进度不会丢失
6. 作为玩家，我希望只有当前攻略链 NPC 的攻略值可以增长，以便攻略链顺序被强制执行
7. 作为玩家，我希望粘滞计数达到 3 时自动触发攻略值增长并归零，以便防爆机制正常工作
8. 作为玩家，我希望粘滞计数超过 3 时被强制归零，以便防止异常累积
9. 作为玩家，我希望好感度不能超过 100 或显示负数，以便数值在合理范围内
10. 作为玩家，我希望攻略值不能超过 100 或显示负数，以便数值在合理范围内
11. 作为玩家，我希望灵石不能为负，以便我不会欠系统钱
12. 作为玩家，我希望装备道具时检查好感度门槛，以便低好感度 NPC 不能装备高级道具
13. 作为玩家，我希望玩家自身的装备不受好感度门槛限制，以便 Phase 2 装备机制正常
14. 作为玩家，我希望混合装备有效和无效道具时只移除无效的，以便有效道具不被误删
15. 作为玩家，我希望购买改变阵法需要柳素衣攻略值等于100，以便真结局有前置条件
16. 作为玩家，我希望不满足条件购买改变阵法时自动退款，以便我不会损失灵石
17. 作为玩家，我希望剩余天数归零时自动切换到 Phase 2（牝奴期），以便游戏阶段正确转换
18. 作为玩家，我希望 Phase 2 中 NPC 的好感度、攻略值、粘滞计数被冻结，以便 Phase 1 的成果被保留
19. 作为玩家，我希望 Phase 2 中堕落度只能升不能降，以便堕落进程是单向的
20. 作为玩家，我希望剩余天数显示为整数（向下取整），以便界面清晰易读
21. 作为玩家，我希望牝阴决层数不超过 9，以便数值在设计范围内
22. 作为开发者，我希望后端校验逻辑提取为纯函数，以便可以独立编写单元测试
23. 作为开发者，我希望 guards 函数和后端校验的粘滞阈值保持一致（统一为 3），以便行为可预测
24. 作为开发者，我希望通过 CDP 注入测试验证 MVU 事件链路，以便确认后端校验在正确时机触发
25. 作为开发者，我希望所有测试自动运行并报告 PASS/FAIL，以便快速发现回归

## Implementation Decisions

### 模块架构

已构建的模块：

1. guards.ts（纯函数层）
   - 纯业务逻辑函数，无副作用
   - 包含：好感度门槛检查、攻略值增量计算、灵石获取计算、攻略链状态查询、粘滞触发判断、Phase 2 初始化、GOOD END 检查、堕落度阶段判断
   - 可独立测试，不依赖 MVU 运行时

2. validate.ts（后端校验纯函数层）- 新增
   - 从事件处理器中提取的 validateVariables(new_data, old_data) 纯函数
   - 使用 lodash.get/set 操作嵌套对象
   - 包含全部 15 条校验规则
   - 可独立测试，不依赖 MVU 运行时

3. 后端校验 index.ts（事件接线层）- 简化
   - 仅包含事件监听和 StatusPlaceHolderImpl 注入
   - 调用 validateVariables() 执行实际校验
   - 不包含业务逻辑

### 关键设计决策

1. 粘滞阈值统一为 3
   - 之前：guards 区分亲密接触(3)/NSFW行为(2)，后端校验统一用 3
   - 现在：guards 和后端校验统一使用阈值 3
   - 原因：简化逻辑，避免 guards 和后端行为不一致

2. 剩余天数 Math.floor 取整
   - 29.998 变 29，28.998 变 28
   - 取整在后端校验中执行，位于 Phase 2 切换检查之后
   - 0.5 天不会触发 Phase 2（因为 0.5 > 0），但显示为 0 天
   - 原因：用户要求整数天数显示

3. 校验执行顺序（在 validateVariables 内部）
   1. Phase 2 切换检查（剩余天数 <= 0 且旧阶段为攻略期）
   2. Phase 2 NPC 变量冻结
   3. 攻略链顺序强制（好感度<30 则攻略值归零、攻略值不能倒退、非当前NPC攻略值回退）
   4. 装备门槛校验
   5. 改变阵法购买前置校验
   6. 粘滞触发兜底（>=3 且好感度>=30 则自动增长+归零）
   7. 灵石负数修正
   8. 堕落度只升不降（Phase 2）
   9. 牝阴决层数上限
   10. 剩余天数取整+范围校验
   11. 好感度/攻略值范围校验
   12. 堕落度范围校验

4. 双阶段变量策略
   - Phase 1（攻略期）：好感度、攻略值、粘滞计数均可变
   - Phase 2（牝奴期）：NPC 变量冻结（从旧值恢复），堕落度只能升不能降

### Schema 变更

- 剩余天数：Zod 层使用 clamp(0, 30)，后端校验层增加 Math.floor
- 好感度/攻略值：Zod 层使用 clamp(0, 100)，后端校验层二次确认

## Testing Decisions

### 好测试的标准

- 只测试外部行为（输入到输出），不测试实现细节
- 每个测试用例对应一个具体的游戏场景
- 测试名称使用中文描述游戏场景，便于理解
- 使用 makeData() 辅助函数构建测试数据，保持测试简洁
