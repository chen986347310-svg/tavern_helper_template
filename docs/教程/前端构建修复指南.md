# 前端构建修复指南

## 背景

本指南总结了从一次实际白屏修复中积累的经验，覆盖 webpack 构建错误的排查思路、常见陷阱和验证方法。

## 排查流程

遇到白屏时，按以下顺序排查：

### 1. 确认 dist 是否有构建错误

dist 产物中的 `throw new Error("Module parse failed...")` 是 webpack 编译失败的标志。webpack 会把错误信息内联到 HTML 中，导致 iframe 加载时立即崩溃。

```powershell
# 检查 dist 中是否有构建错误
Select-String -Path dist\雌堕合欢宗\index.html -Pattern 'Module parse failed'
```

**关键区分**：`throw new Error` 有两种：
- **构建错误**：包含 `Module parse failed`、`SyntaxError` 等 → 需要修复
- **运行时断言**：如 `injectLocal must be called in setup` → 正常，不影响启动

### 2. 回溯到 src 找根因

如果 dist 有错误，不要在 dist 里修——回到 src 找原因。常见根因：

| 症状 | 根因 | 修复方法 |
|------|------|---------|
| SCSS 编译失败 | 文件末尾残留 `*** End Patch` | 删除残留行 |
| 二进制文件解析失败 | webpack 正则或 loader 配置错误 | 检查 `webpack.config.ts` 的 `module.rules` |
| 模块找不到 | `import` 路径错误或依赖缺失 | 检查 `import` 语句和 `node_modules` |

### 3. 修复后验证

修复 src 后，必须清理并重建 dist：

```powershell
# 清理旧产物
Remove-Item dist\雌堕合欢宗 -Recurse -Force

# 重新构建
pnpm build
```

## 常见陷阱

### 陷阱 1：apply_patch 工具残留

`apply_patch` 工具可能在文件末尾留下 `*** End Patch` 标记。这行会被 SCSS/TS 编译器当作语法错误。

**检查方法**：
```powershell
Get-ChildItem src -Recurse -Include '*.scss','*.ts','*.vue' | Select-String '\*\*\* End Patch'
```

**预防**：每次使用 apply_patch 后，检查被修改文件的末尾。

### 陷阱 2：正则转义错误

在 TypeScript/JavaScript 的正则字面量中，`$` 是字符串结尾锚点，`\$` 是匹配字面量 `$` 字符。两者含义完全不同。

```typescript
// 错误：匹配 .png$、.jpg$ 等（永远不会匹配任何文件）
test: /\.(png|jpe?g|gif|webp)\$/i,

// 正确：匹配 .png、.jpg 等结尾
test: /\.(png|jpe?g|gif|webp)$/i,
```

**症状**：正则不匹配 → webpack 尝试用默认 JS 解析器处理二进制文件 → `Unexpected character`。

### 陷阱 3：asset/resource vs asset/inline

webpack 的 `type` 配置决定文件如何打包：

| type | 行为 | 适用场景 |
|------|------|---------|
| `asset/resource` | 输出为独立文件，JS 中得到 URL | 传统 Web 应用，文件由服务器提供 |
| `asset/inline` | 内联为 base64 data URI | 单文件 HTML 应用（如酒馆 iframe） |
| `asset` | 自动判断（8KB 为界） | 通用场景 |

**本项目规则**：所有 entry 都使用 `HtmlInlineScriptWebpackPlugin` + `HTMLInlineCSSWebpackPlugin` 产出自包含单文件 HTML。因此图片、字体等资源必须用 `asset/inline`，否则 iframe 中无法访问外部文件。

**验证方法**：对比 dist 中其他 entry 是否有外部文件：
```powershell
Get-ChildItem dist -Directory | ForEach-Object {
    $files = (Get-ChildItem $_.FullName -Recurse -File).Count
    Write-Output "$($_.Name): $files 个文件"
}
```

## Vue Composable 陷阱

### 陷阱 4：composable 非单例导致状态隔离

Vue composable 普通函数每次调用都创建新的局部变量。如果两个地方各自调用 `useDebug()`，它们拿到的 `visible` 是不同的 ref，互不影响。

```typescript
// ❌ 错误：每次调用创建新的 visible
export function useDebug() {
  const visible = ref(false);  // 每次都是新的
  return { visible, toggle };
}

// ✅ 正确：模块级变量，所有调用方共享
const visible = ref(false);  // 模块顶层
export function useDebug() {
  return { visible, toggle };
}
```

**症状**：快捷键 toggle 了 `visible`，但面板没有响应——因为面板绑的是另一个 `visible`。

**排查方法**：检查 composable 函数体内是否有 `ref()`/`reactive()` 创建状态。如果有，且该 composable 被多处调用，就需要提升到模块顶层。

### 陷阱 5：环境守卫导致功能被 tree-shake

`process.env.NODE_ENV === "development"` 在 webpack production 模式下会被替换为 `"production" === "development"`（即 `false`），TerserPlugin 会将整个 `if` 分支作为死代码删除。

```typescript
// ❌ production 构建时这段代码被完全移除
if (process.env.NODE_ENV === "development") {
  debug.initShortcut();
}
```

**解决**：将初始化逻辑移到 Vue 组件的 `onMounted` 中，让组件自身负责注册，而非依赖外部条件编译。

### 陷阱 6：happy-dom 的 KeyboardEvent 不支持修饰键

happy-dom 的 `KeyboardEvent` 构造器不正确处理 `ctrlKey`/`shiftKey`/`altKey` 等初始化选项。即使传入 `{ ctrlKey: true, shiftKey: true }`，事件对象的这些属性仍为 `false`。

```typescript
// ❌ 在 happy-dom 中不生效
document.dispatchEvent(
  new KeyboardEvent("keydown", { ctrlKey: true, shiftKey: true, key: "D" })
);

// ✅ 方案 A：用 Object.defineProperty 强制设置
const e = new KeyboardEvent("keydown", { key: "D" });
Object.defineProperty(e, "ctrlKey", { value: true });
Object.defineProperty(e, "shiftKey", { value: true });
document.dispatchEvent(e);

// ✅ 方案 B（推荐）：捕获 handler 后直接调用，mock 事件属性
const handler = vi.fn();
vi.spyOn(document, "addEventListener").mockImplementation((type, cb) => {
  if (type === "keydown") handler.mockImplementation(cb);
});
initShortcut();
const e = new KeyboardEvent("keydown");
vi.spyOn(e, "ctrlKey", "get").mockReturnValue(true);
vi.spyOn(e, "shiftKey", "get").mockReturnValue(true);
vi.spyOn(e, "key", "get").mockReturnValue("D");
handler(e);
```

## 测试断言陷阱

### 陷阱 7：webpack asset/resource 的中文文件名 URL 编码

webpack 的 `asset/resource` 在 happy-dom 测试环境中返回的路径会经过 URL 编码：`苏芸.png` → `%E8%8B%8F%E8%8A%B8.png`。直接用中文字符串做 `toContain` 断言会失败。

```typescript
// ❌ 失败：实际值是 URL 编码的
expect(img.attributes("src")).toContain("苏芸.png");
// 实际: /src/%E9%9B%8C.../%E8%8B%8F%E8%8A%B8.png

// ✅ 正确：先解码再断言
expect(decodeURIComponent(img.attributes("src"))).toContain("苏芸.png");
```

**排查方法**：测试失败时先打印实际值 `console.log(img.attributes("src"))`，检查是否为 URL 编码形式。
