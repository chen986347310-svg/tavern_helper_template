# 前端构建修复指南

## 当前权威边界（2026-05-22）

本指南只负责说明**前端状态栏产物与独立运行脚本**的构建、排错和验证。接手者必须先区分三类产物：

- **状态栏 UI 产物**：`dist/雌堕合欢宗/index.html`，由 `pnpm build` 生成，证明 Vue/SCSS/资源内联打包成功。
- **独立 MVU 脚本产物**：`dist/var_structure.js` 与 `dist/backend_validate.js`，webpack 不处理，必须按本文 esbuild 流程单独构建。
- **角色卡/世界书产物**：SillyTavern 实际启用的角色卡、世界书与变量提示词；这不是 `dist/雌堕合欢宗/index.html`，不能用前端构建成功代替同步成功。

当前 `package.json` 的 `sync` 指向
ode tavern_sync.mjs`，但工作区缺少 `tavern_sync.mjs`。在恢复或替代同步链路前，`pnpm build` 通过也不得宣称世界书已进入酒馆。

## 背景

本指南总结了从一次实际白屏修复中积累的经验，覆盖 webpack 构建错误的排查思路、常见陷阱和验证方法。

## 排查流程

遇到白屏时，按以下顺序排查：

### 1. 确认 dist 是否有构建错误

dist 产物中的 `throw new Error("Module parse failed...")` 是 webpack 编译失败的标志。webpack 会把错误信息内联到 HTML 中，导致 iframe 加载时立即崩溃。

```powershell
# 检查 dist 中是否有构建错误
Select-String -Path dist\雌堕合欢宗\index.html -Pattern 'Module parse failed'
```

**关键区分**：`throw new Error` 有两种：
- **构建错误**：包含 `Module parse failed`、`SyntaxError` 等 → 需要修复
- **运行时断言**：如 `injectLocal must be called in setup` → 正常，不影响启动

### 2. 回溯到 src 找根因

如果 dist 有错误，不要在 dist 里修——回到 src 找原因。常见根因：

| 症状 | 根因 | 修复方法 |
|------|------|---------|
| SCSS 编译失败 | 文件末尾残留 `*** End Patch` | 删除残留行 |
| 二进制文件解析失败 | webpack 正则或 loader 配置错误 | 检查 `webpack.config.ts` 的 `module.rules` |
| 模块找不到 | `import` 路径错误或依赖缺失 | 检查 `import` 语句和
ode_modules` |

### 3. 修复后验证

修复 src 后，必须清理并重建 dist：

```powershell
# 清理旧产物
Remove-Item dist\雌堕合欢宗 -Recurse -Force

# 重新构建
pnpm build
```

## 常见陷阱

### 陷阱 1：apply_patch 工具残留

`apply_patch` 工具可能在文件末尾留下 `*** End Patch` 标记。这行会被 SCSS/TS 编译器当作语法错误。

**检查方法**：
```powershell
Get-ChildItem src -Recurse -Include '*.scss','*.ts','*.vue' | Select-String '\*\*\* End Patch'
```

**预防**：每次使用 apply_patch 后，检查被修改文件的末尾。

### 陷阱 2：正则转义错误

在 TypeScript/JavaScript 的正则字面量中，`$` 是字符串结尾锚点，`\$` 是匹配字面量 `$` 字符。两者含义完全不同。

```typescript
// 错误：匹配 .png$、.jpg$ 等（永远不会匹配任何文件）
test: /\.(png|jpe?g|gif|webp)\$/i,

// 正确：匹配 .png、.jpg 等结尾
test: /\.(png|jpe?g|gif|webp)$/i,
```

**症状**：正则不匹配 → webpack 尝试用默认 JS 解析器处理二进制文件 → `Unexpected character`。

### 陷阱 3：asset/resource vs asset/inline

webpack 的 `type` 配置决定文件如何打包：

| type | 行为 | 适用场景 |
|------|------|---------|
| `asset/resource` | 输出为独立文件，JS 中得到 URL | 传统 Web 应用，文件由服务器提供 |
| `asset/inline` | 内联为 base64 data URI | 单文件 HTML 应用（如酒馆 iframe） |
| `asset` | 自动判断（8KB 为界） | 通用场景 |

**本项目规则**：所有 entry 都使用 `HtmlInlineScriptWebpackPlugin` + `HTMLInlineCSSWebpackPlugin` 产出自包含单文件 HTML。因此图片、字体等资源必须用 `asset/inline`，否则 iframe 中无法访问外部文件。

**验证方法**：对比 dist 中其他 entry 是否有外部文件：
```powershell
Get-ChildItem dist -Directory | ForEach-Object {
    $files = (Get-ChildItem $_.FullName -Recurse -File).Count
    Write-Output "$($_.Name): $files 个文件"
}
```


### 陷阱 8：Vue 模板 v-if 中变量名缺少引号

**现象**：组件条件渲染完全失效，v-if 条件永远为 false。

**根因**：Vue 模板中 `currentTab === home` 的 `home` 被当作 JavaScript 变量引用（值为 undefined），而非字符串字面量 `'home'`。

```html
<!-- 错误 -->
<SystemBar v-if="currentTab === home && ..." />

<!-- 正确 -->
<SystemBar v-if="currentTab === 'home' && ..." />
```

**排查方法**：在 dist 产物中搜索编译后的渲染函数，检查 `_ctx.home`（错误）vs `'home'`（正确）。

**修复记录**：2026-05-20 App.vue 第 18、26 行 SystemBar v-if 条件修复。

## Vue Composable 陷阱

### 陷阱 4：composable 非单例导致状态隔离

Vue composable 普通函数每次调用都创建新的局部变量。如果两个地方各自调用 `useDebug()`，它们拿到的 `visible` 是不同的 ref，互不影响。

```typescript
// ❌ 错误：每次调用创建新的 visible
export function useDebug() {
  const visible = ref(false);  // 每次都是新的
  return { visible, toggle };
}

// ✅ 正确：模块级变量，所有调用方共享
const visible = ref(false);  // 模块顶层
export function useDebug() {
  return { visible, toggle };
}
```

**症状**：快捷键 toggle 了 `visible`，但面板没有响应——因为面板绑的是另一个 `visible`。

**排查方法**：检查 composable 函数体内是否有 `ref()`/`reactive()` 创建状态。如果有，且该 composable 被多处调用，就需要提升到模块顶层。

### 陷阱 5：环境守卫导致功能被 tree-shake

`process.env.NODE_ENV === "development"` 在 webpack production 模式下会被替换为 `"production" === "development"`（即 `false`），TerserPlugin 会将整个 `if` 分支作为死代码删除。

```typescript
// ❌ production 构建时这段代码被完全移除
if (process.env.NODE_ENV === "development") {
  debug.initShortcut();
}
```

**解决**：将初始化逻辑移到 Vue 组件的 `onMounted` 中，让组件自身负责注册，而非依赖外部条件编译。

### 陷阱 6：happy-dom 的 KeyboardEvent 不支持修饰键

happy-dom 的 `KeyboardEvent` 构造器不正确处理 `ctrlKey`/`shiftKey`/`altKey` 等初始化选项。即使传入 `{ ctrlKey: true, shiftKey: true }`，事件对象的这些属性仍为 `false`。

```typescript
// ❌ 在 happy-dom 中不生效
document.dispatchEvent(
  new KeyboardEvent("keydown", { ctrlKey: true, shiftKey: true, key: "D" })
);

// ✅ 方案 A：用 Object.defineProperty 强制设置
const e = new KeyboardEvent("keydown", { key: "D" });
Object.defineProperty(e, "ctrlKey", { value: true });
Object.defineProperty(e, "shiftKey", { value: true });
document.dispatchEvent(e);

// ✅ 方案 B（推荐）：捕获 handler 后直接调用，mock 事件属性
const handler = vi.fn();
vi.spyOn(document, "addEventListener").mockImplementation((type, cb) => {
  if (type === "keydown") handler.mockImplementation(cb);
});
initShortcut();
const e = new KeyboardEvent("keydown");
vi.spyOn(e, "ctrlKey", "get").mockReturnValue(true);
vi.spyOn(e, "shiftKey", "get").mockReturnValue(true);
vi.spyOn(e, "key", "get").mockReturnValue("D");
handler(e);
```

## 测试断言陷阱

### 陷阱 7：webpack asset/resource 的中文文件名 URL 编码

webpack 的 `asset/resource` 在 happy-dom 测试环境中返回的路径会经过 URL 编码：`苏芸.png` → `%E8%8B%8F%E8%8A%B8.png`。直接用中文字符串做 `toContain` 断言会失败。

```typescript
// ❌ 失败：实际值是 URL 编码的
expect(img.attributes("src")).toContain("苏芸.png");
// 实际: /src/%E9%9B%8C.../%E8%8B%8F%E8%8A%B8.png

// ✅ 正确：先解码再断言
expect(decodeURIComponent(img.attributes("src"))).toContain("苏芸.png");
```

**排查方法**：测试失败时先打印实际值 `console.log(img.attributes("src"))`，检查是否为 URL 编码形式。

## 主题系统陷阱

### 陷阱 9: CSS 自定义属性与 SCSS mixin 硬编码色冲突

SCSS mixin（如 gold-book-bg、gold-foil）在编译时生成硬编码颜色值。CSS 自定义属性（var(--theme-xxx)）无法覆盖这些编译后的值。

**解决方案**: 在 html[data-theme=light] 选择器中直接覆写 background、border、box-shadow 等属性，使用 !important 或更高特异性选择器。

### 陷阱 10: Vue scoped 样式特异性竞争

Vue scoped 样式给选择器添加 [data-v-xxx] 属性，提升特异性到 (0,2,0)。全局覆写 html[data-theme=light] .class 也是 (0,2,0)，scoped 在源码顺序中胜出。

**解决方案**: 使用 html[data-theme=light] .class 特异性 (0,2,1) 胜过 scoped 的 (0,2,0)。或在组件内使用 SCSS 嵌套 html[data-theme=light] & 生成 (0,4,1)。

### 陷阱 11: color-mix() 与 var() 组合不可靠

color-mix(in srgb, var(--npc-color) 55%, #2c1a08) 在部分浏览器中解析异常，尤其当 var() 引用的值来自内联样式时。

**解决方案**: 避免运行时颜色混合。改为在 JS 层定义明暗双色映射，通过 CSS 自定义属性切换：

- JS 层: NPC_COLORS（暗色背景用）+ NPC_COLORS_DARK（亮色背景用）
`- 内联 style: --npc-color + --npc-color-dark
`- CSS 层: :root { --npc-accent: var(--npc-color); } [data-theme=light] { --npc-accent: var(--npc-color-dark); }

### 陷阱 12: 全局字符串替换破坏暗色模式

对 SCSS 文件做全局 str.replace() 时，替换模式可能匹配到非目标区域。例如替换 #8b6914 时如果该值出现在暗色模式样式中，会导致暗色模式文字不可见。

**预防**: 替换前用正则确认匹配位置，限定替换范围（如仅在 html[data-theme=light] 块内）。

## esbuild 独立脚本构建（2026-05-21）

### 背景

MVU 框架在运行时动态加载两个独立 JS 文件，这些文件不在 webpack bundle 中，需要单独用 esbuild 构建：

| 文件 | 源码路径 | 用途 |
|------|---------|------|
| `dist/var_structure.js` | `src/雌堕合欢宗/脚本/变量结构/index.ts` | 注册 Zod Schema 到 MVU 框架 |
| `dist/backend_validate.js` | `src/雌堕合欢宗/脚本/后端校验/validate.ts` | 后端变量校验（coerceNumeric + 游戏规则） |

### 构建命令

```powershell
# var_structure.js（标记 https://* 为 external，浏览器运行时解析）
npx esbuild src/雌堕合欢宗/脚本/变量结构/index.ts --bundle --format=esm --outfile=dist/var_structure.js --external:https://*

# backend_validate.js（lodash 在 SillyTavern 中是全局 window._）
npx esbuild src/雌堕合欢宗/脚本/后端校验/validate.ts --bundle --format=esm --outfile=dist/backend_validate.js --external:lodash
```

### Post-build 修补（backend_validate.js 必须）

esbuild 输出的 `backend_validate.js` 需要两处手动修补：

1. **替换 lodash import**：`import _ from "lodash";` → `const _ = window._;`
2. **移除 export**：`export { validateVariables };` → 注释掉或删除（浏览器 script 上下文不支持 ESM export）

可用 Python 脚本自动完成：

```python
with open("dist/backend_validate.js", "r", encoding="utf-8") as f:
    c = f.read()
c = c.replace('import _ from "lodash";', 'const _ = window._;')
c = c.replace("export {
  validateVariables
};", "// validateVariables available in scope")
with open("dist/backend_validate.js", "w", encoding="utf-8") as f:
    f.write(c)
```

### 未构建时的症状

- 控制台 `:5500/dist/var_structure.js` 404 错误
- 控制台 `:5500/dist/backend_validate.js` 404 错误
- Zod 验证错误（支配次数 NaN ×5、NPC undefined ×5）— 变量结构未注册导致默认值未应用

### 验证成功标志

控制台出现 `mvu_zod.ts:186 变量结构注册成功`，且无 404 和 Zod 验证错误。
## 世界书与角色卡同步断点排查（2026-05-22）

### 背景

`pnpm build` 只构建前端状态栏相关产物，不等于世界书提示词已经进入 SillyTavern。`dist/雌堕合欢宗/index.html` 只能证明 Vue 状态栏已打包，不能证明 `变量输出格式.yaml` 已进入 SillyTavern 正在使用的角色卡/世界书。

### 必查清单

1. **确认同步脚本存在**：`package.json` 的 `sync` 当前指向
ode tavern_sync.mjs`，若 `tavern_sync.mjs` 缺失，`pnpm sync` 链路不可用，必须先恢复该文件或确认替代同步工具。
2. **确认源文件已更新**：检查 `src/雌堕合欢宗/世界书/变量/变量输出格式.yaml` 是否包含 `MVU解析硬约束`。
3. **确认酒馆实际启用版本**：仅本地源文件更新不代表 SillyTavern 已导入；需要重新 bundle/import 角色卡或世界书。
4. **确认实机输出格式**：CDP 读取最新楼层正文，必须出现 `<UpdateVariable>` 与 `<JSONPatch>`，不能只有 `变量更新情况` 或裸 JSON Patch。
5. **确认 MVU 回流**：使用 `Mvu.getMvuData({ type: 'message', message_id: 'latest' })` 读取最新楼层变量，确认目标字段实际存在。

### 典型症状

- `dist/雌堕合欢宗/index.html` 搜不到 `MVU解析硬约束`：正常，因为这是前端状态栏产物，不是世界书文本产物。
- AI 回复末尾有裸 JSON Patch，但最新 MVU 没有 `soul_whisper`：提示词/世界书仍未强制标准 `<UpdateVariable>` 包裹。
- AI 写入 `外门广场` 等 schema 外场景：世界书约束未生效或场景映射规则不够硬。

### 推荐复验命令

```powershell
# 检查源世界书规则
Select-String -Path "src/雌堕合欢宗/世界书/变量/变量输出格式.yaml" -Pattern "MVU解析硬约束|<UpdateVariable>|<JSONPatch>"

# 检查同步脚本是否存在
Test-Path .\tavern_sync.mjs

# 前端构建只能验证状态栏，不等于世界书已导入
pnpm build
```
## v4 闭环构建验收口径（2026-05-22）

完成前端构建后，按以下顺序判定，不得跳级：

1. **前端构建通过**：`pnpm build` 退出码为 0，`dist/雌堕合欢宗/index.html` 不含 `Module parse failed`。
2. **独立脚本通过**：如修改 `schema.ts`、`validate.ts` 或 MVU 注册逻辑，必须单独构建并检查 `dist/var_structure.js`、`dist/backend_validate.js`。
3. **世界书已同步**：确认 SillyTavern 实际启用版本包含 `MVU解析硬约束`、`<UpdateVariable>` 与 `<JSONPatch>` 示例；若 `tavern_sync.mjs` 缺失，必须记录替代同步方式。
4. **新楼层回流通过**：用户发送下一条消息后，最新楼层正文必须有标准变量块，且 `Mvu.getMvuData({ type: 'message', message_id: 'latest' })` 能读到目标变量变化。

只有第 1 步通过，结论只能写“前端构建通过”；只有 1-4 全部通过，才能写“v4 UI → AI → MVU 闭环通过”。
