### 1.6 Codex CLI 文件写入规范

在 Codex CLI 环境中编写文件时，**禁止使用 PowerShell 的 `Set-Content`/`Add-Content` 写入文件内容**。PowerShell 字符串转义是最高频的低级错误来源。

#### 根本原因

| 原因 | 表现 | 说明 |
|------|------|------|
| `$var` 插值 | 内容中的 `$f`、`$match` 等被当作变量替换 | PowerShell 双引号内自动插值，破坏正则表达式、Shell 脚本或 Vue 模板 |
| 反引号冲突 | 内容中的反引号被解释为命令替换符 | PowerShell 特有语法，导致 SCSS 嵌套或 ES6 模板字符串结构错乱 |
| 中文路径截断 | Python sys.argv 中中文变成 `??` 乱码 | 编码不一致，跨语言调用含中文路径时触发 FileNotFoundError |
| 长字符串截断 | 命令行超过长度限制，代码被拦腰斩断 | Windows 命令行单条 Command 存在硬性长度限制，大文件整块覆盖时丢失后半部分 |
| 换行符冲突 | 补丁引擎提示 Diff 块匹配失败 | Windows 默认 CRLF 与项目 Git 强制 LF 冲突，导致 apply_patch 无法精准定位行 |
| UTF-8 BOM 污染 | 编译工具抛出未知解析错误 | Windows 默认编辑器或流写入自动加 BOM 头，引发前端编译灾难 |

#### 推荐方案

**方案 A：`apply_patch` 编辑已有文件（首选）**

增量修改，不涉及字符串转义，只描述差异。格式：`Begin Patch` -> `Update File` -> `@@ hunk @@` -> `+/- 行` -> `End Patch`。

**方案 B：`apply_patch` 创建新文件（大文件分多次追加）**

第一次创建骨架，第二次追加中间章节，第三次追加最后章节。每次只处理几十行，完全避免转义问题。

**方案 C：Python 脚本编辑文件（适用于 SCSS 含 `$` 变量时）**

当文件内容包含大量 `$` 符号（如 SCSS 变量）时，apply_patch 的 `$` 行首会被 PowerShell 或 patch 解析器误判。此时用 Python 直接编辑文件：

```powershell
python -c "f='path/to/file'; t=open(f,'r',encoding='utf-8').read(); t=t.replace('old','new'); open(f,'w',encoding='utf-8').write(t)"
```

对于多行复杂替换，先写 Python 脚本到临时文件再执行。

#### 注意事项

- `apply_patch` 内容必须以 `*** Begin Patch` 开头、`*** End Patch` 结尾
- 每个修改块：`@@ 旧内容起始行 @@` -> `- 删除行` -> `+ 新增行`
- 创建新文件用 `*** Create File: path`，修改用 `*** Update File: path`
- patch 内容中的特殊字符无需额外转义（核心优势）
- 写入后检查文件末尾是否残留 `*** End Patch`（参见七、陷阱 1）

#### 什么时候可以使用 PowerShell

仅限简单的只读查询：`Get-Content`、`Select-String`、`Get-ChildItem`、查文件属性等。涉及写入文件内容时，一律使用 `apply_patch`。

## 二、开发流程规范

### 2.1 标准开发流程（6 步）

```
需求理解 → 编码实现 → 本地测试 → 构建验证 → 代码审查 → 提交交付
```

#### 第 1 步：需求理解

- 阅读相关 PRD 文档（`docs/PRD/`）
- 阅读架构文档（`docs/架构报告-*.md`、`docs/前端架构指南.md`）
- 明确修改范围：是新增功能、样式调整还是 bug 修复
- 确认涉及的文件和模块（参考 `docs/前端架构指南.md` 的组件依赖图）

#### 第 2 步：编码实现

- 遵循下面「三、代码质量标准」中的所有规则
- 使用现有 SCSS 变量和 mixin，不硬编码颜色/间距
- 新增组件时同步编写对应的 `.test.ts` 文件
- 修改 schema 时确保幂等性（`Schema.parse(Schema.parse(x)) === Schema.parse(x)`）

#### 第 3 步：本地测试

```powershell
# 运行所有测试
npx vitest run

# 运行特定模块测试
npx vitest run src/雌堕合欢宗/界面/guards.test.ts
```

#### 第 4 步：构建验证

```powershell
# 清理旧产物
Remove-Item dist\雌堕合欢宗 -Recurse -Force

# 生产构建
pnpm build

# 验证产物（见 1.4 节）
```

#### 第 5 步：代码审查

按照「四、代码审查清单」逐项检查。

#### 第 6 步：提交交付

- 确保 dist 产物无构建错误
- 编写或更新 handoff 文档（`docs/handoff/`）
- 记录已知问题和下一步建议

---

### 2.2 热重载开发流程（推荐）

日常开发推荐使用热重载模式，无需每次手动 `pnpm build`：

```
1. 启动监听模式: pnpm watch
2. 启动带调试端口的 Chrome（参见 docs/教程/chrome调试启动指引.md）
3. 通过 MCP chrome-devtools 连接酒馆网页
4. 确认「酒馆助手-实时监听-允许监听」开关已启用
5. 修改 src/ 代码后，酒馆网页自动热重载
```

验证热重载是否生效：打开酒馆网页后，检查 `$("#extensions_settings")` 中的「酒馆助手-实时监听-允许监听」开关是否启用。

### 2.3 MVU 角色卡开发流程（额外步骤）

当开发 MVU 角色卡相关功能时，在标准流程基础上增加以下步骤：

```
需求理解 → schema 设计 → 变量同步 → 界面实现 → 脚本实现 → 世界书编写 → 测试 → 构建
```

#### 角色卡文件结构

```
src/雌堕合欢宗/
  index.ts              # 入口文件
  index.html            # HTML 模板（仅 <body> 静态内容）
  App.vue               # 根组件
  schema.ts             # MVU 变量结构定义（zod 4）
  schema.json           # 自动生成的 JSON Schema
  index.yaml            # 角色卡元数据（头像、描述、世界书引用）
  界面/                 # 前端界面项目
    components/         # Vue 组件
    composables/        # 组合式函数
    pages/              # 页面组件
    styles/             # SCSS 样式
    assets/avatars/     # NPC 立绘图片
    data/               # 静态数据
    store.ts            # Pinia store
    guards.ts           # 游戏逻辑函数
  脚本/                 # 后台脚本项目
    MVU/                # MVU 相关脚本
    变量结构/            # 变量结构脚本
    后端校验/            # 后端校验脚本
  世界书/               # 世界书条目
    变量/               # 变量相关条目
    NPC/                # NPC 专属条目（每人一个 yaml）
    玩家/               # 玩家相关条目
    *.yaml              # 世界设定、宗门文化等
  第一条消息/           # 角色卡首条消息
    0.txt
```

#### schema 设计检查

- [ ] `schema.ts` 使用 zod 4（不是 zod 3）
- [ ] 所有字段有 `.prefault()` 默认值
- [ ] 数字字段用 `z.coerce.number()` + `_.clamp()` 限制范围
- [ ] 使用 `z.record()` 而非 `z.array()` 管理动态集合
- [ ] `Schema.parse(Schema.parse(x)) === Schema.parse(x)` 幂等性成立
- [ ] 修改后执行 `pnpm dump` 更新 `schema.json`

#### 三处同步检查

新增 NPC 时必须同步修改 3 处：
1. `_variables.scss`：`$npc专属色` map 加新颜色
2. `schema.ts`：`z.enum` 中加新名字
3. `guards.ts`：攻略链数组加新名字

可选同步：
4. `世界书/NPC/`：新增 NPC 专属世界书条目
5. `界面/assets/avatars/`：新增正道版 + 仙奴版立绘图片
6. `index.yaml`：更新角色卡元数据

## 三、代码质量标准

### 3.1 TypeScript 规范

| 规则 | 说明 |
|------|------|
| 禁止 `any` | 除非有充分理由并加注释 |
| 禁止未使用变量 | `noUnusedLocals` + `noUnusedParameters` 已在 tsconfig 启用 |
| 禁止隐式返回 | `noImplicitReturns: true` |
| 优先使用 `const` | ESLint `prefer-const: warn` |
| 优先使用酒馆助手接口 | `@types/function/*.d.ts` 优先于 `exported.sillytavern.d.ts` |
| 使用 `klona()` 脱 proxy | 写入酒馆变量前必须 `klona()` 去除 Vue 响应式代理 |

### 3.2 Vue 组件规范

- 使用 `<style lang="scss" scoped>`，禁止全局样式
- 禁止使用 `vh` 单位（iframe 环境下不可靠），使用 `width` + `aspect-ratio`
- 禁止 `position: absolute` 使主体脱离文档流
- 禁止在 `index.html` 中使用 `<link>` 或 `<script>` 引入外部文件
- 图片使用相对路径（无 `/` 开头），适配 iframe 子目录环境

### 3.3 SCSS 规范

- 颜色值使用 `_variables.scss` 中定义的 token
- 可复用视觉效果使用 `_mixins.scss` 中的 mixin
- 禁止在 mixin 中硬编码可被 CSS 变量覆盖的颜色（如需暗/亮双模式）
- 需要主题感知的元素使用 `var(--theme-*)` 或 `var(--npc-accent)`
- 需要覆写 mixin 硬编码色时，在 `_global.scss` 的 `html[data-theme=light]` 块中使用 `!important`

### 3.4 测试规范

- 测试框架：Vitest + @vue/test-utils + happy-dom
- **环境配置**：`vitest.config.ts` 全局设为 `environment: "node"`，各测试文件通过顶部 `// @vitest-environment happy-dom` 指令覆盖
- 只测外部行为，不测实现细节
- 每个组件/模块的 `.test.ts` 文件与源文件同目录
- `beforeEach` 中重置 composable 共享状态（模块级 `ref`/`reactive`）
- happy-dom 限制：修饰键需用 `Object.defineProperty` 强制设置，中文文件名需 `decodeURIComponent()`

### 3.5 ESLint 配置要点

项目 `eslint.config.mjs` 启用的规则：

- `import-x/no-cycle: error`：禁止循环依赖
- `no-var: error`：禁止 `var`
- `yoda: error`：条件表达式常量在左侧
- `pinia` 插件：Pinia store 规范
- `better-tailwindcss`：类名规范（已安装但主要项目不用 Tailwind）
- `vue` 插件：Vue SFC 规范

## 五、智能体协作开发指南

### 5.1 何时使用智能体

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 新功能开发（涉及多文件） | 并行子智能体 | 拆分为独立模块并行开发 |
| Bug 排查 | 单智能体 + `diagnose` skill | 系统化排查流程 |
| UI 样式调整 | 单智能体 + `ui-ux-pro-max` skill | 专业样式建议 |
| 代码审查 | 子智能体 + `review` skill | 独立视角审查 |
| PRD → 代码 | 单智能体 + `writing-plans` + `executing-plans` | 规划后再执行 |
| 截图验证 | `browser-use` skill | 自动化 UI 截图对比 |

### 5.2 并行子智能体模式

当任务可拆分为多个独立模块时，使用 `dispatching-parallel-agents` 模式：

```
主智能体
  子智能体 1: 新增 ShopPage 样式
  子智能体 2: 新增 BackpackPage 样式
  子智能体 3: 新增 GalleryPage 样式
```

拆分原则：
- 每个子智能体拥有独立的文件写入范围（不重叠）
- 共享文件（如 `_variables.scss`、`_global.scss`）由主智能体统一处理
- 子智能体之间不需要互相等待

### 5.3 信息传递格式（Handoff）

当一个智能体需要将工作交接给另一个时，使用 `docs/handoff/` 下的 handoff 文档：

```markdown
# Handoff

## Context
- 技术栈、运行环境、当前测试状态

## Done
- 已完成的工作列表

## BUG / 未解决问题
- 问题描述 + 已排查的线索

## Files
| 文件 | 路径 |
|------|------|

## Next
- 下一步建议

## Skills
- 推荐使用的 skill
```

---
> **与 ``handoff`` skill 的关系**：上面的模板是本项目的扩展版 handoff 格式，比 ``handoff`` skill 自带的通用模板多了 ``Files`` 和 ``Skills`` 两个章节。``handoff`` skill（位于 ``C:/Users/98634/.codex/skills/handoff/SKILL.md``）会生成一份精简的交接文档到临时文件，适合快速交接；而项目模板保存在 ``docs/handoff/`` 下，适合需要留存记录的正式交接。两种方式可以结合使用：先用 ``handoff`` skill 快速生成草稿，再手动补充 Files 和 Skills 信息后保存到 ``docs/handoff/``。

### 6.2 Skill 调用规范

触发规则：当用户明确提到 skill 名称，或任务明显匹配 skill 描述时，必须使用该 skill。

加载方式：
1. 根据 skill 列表找到对应的文件路径
2. 读取 `SKILL.md` 文件了解工作流程
3. 按照流程执行

常用 Skill 快速参考：

```
代码审查:  $review → 读取 SKILL.md → 分析 diff → 给出 pass/fail
Bug 排查:  $diagnose → 读取 SKILL.md → 复现 → 最小化 → 假设 → 修复 → 回归
UI 截图:   $browser-use:browser → 打开 localhost → 截图 → 对比
上下文保存: $context-save → 保存 git 状态 + 决策 + 剩余工作
上下文恢复: $context-restore → 恢复上次会话状态
```

### 6.3 Skill 组合工作流示例

## 七、常见陷阱速查

| 陷阱 | 症状 | 解决 |
|------|------|------|
| `apply_patch` 残留 | SCSS/TS 编译失败 | 检查文件末尾删除 `*** End Patch` |
| 正则转义错误 | 二进制文件解析失败 | 检查 `\$` vs `$` |
| `asset/resource` vs `asset/inline` | iframe 无法加载外部文件 | 使用 `asset/inline` |
| composable 非单例 | 快捷键/面板状态不同步 | 提取 `ref` 到模块顶层 |
| `process.env.NODE_ENV` 守卫 | 功能在生产构建消失 | 移至 `onMounted` |
| happy-dom 修饰键 | 快捷键测试失败 | `Object.defineProperty` 强制设置 |
| CSS 特异性竞争 | 样式覆盖不生效 | 提高选择器特异性或用 `!important` |
| `color-mix()` + `var()` | 部分浏览器异常 | 用 JS 双色映射替代 |
| 全局字符串替换破坏主题 | 暗/亮模式文字不可见 | 替换前用正则确认匹配位置 |
| 中文文件名 URL 编码断言 | 测试中文件名匹配失败 | 先 `decodeURIComponent()` 再断言 |
| PowerShell `$` 插值 | apply_patch 中 `$variable` 被替换为空 | 使用 Python 脚本编辑含 `$` 的文件（SCSS/Shell） |
