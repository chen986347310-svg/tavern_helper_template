# 开发规范与智能体协作指南

> 面向所有开发者（人类和 AI 智能体）的项目开发规范手册。涵盖开发流程、代码质量审查、智能体协作和 Skill 使用指南。

---

## 当前权威协作口径（2026-05-26）

本节优先级高于后文早期通用流程，所有接手者必须先读完再开工：

- **项目定位**：本仓库当前主项目是 `src/雌堕合欢宗/` 角色卡，包含 Vue 状态栏、MVU schema、运行时脚本、世界书、第一条消息与打包/同步配置。
- **当前发布口径**：交付版加载 `https://cdn.jsdelivr.net/gh/chen986347310-svg/tavern_helper_template@main/public/hehuan` 下的入口页、状态栏和三条脚本；`testingcf` 仅保留给第三方模板资源或历史引用，本项目运行资源不要再切回 `testingcf @main`。
- **角色卡格式口径**：当前 `头像: null`，打包结果是 JSON 内容；若产物被命名为 `.png`，必须改名为 `dist/雌堕合欢宗.json` 后再发布和导入。
- **缓存刷新口径**：更新 `public/hehuan/**` 并推送后，用 `https://www.jsdelivr.com/tools/purge` 输入标准 `cdn.jsdelivr.net` URL 清缓存，再验证入口页不含 `document.currentScript.closest` 旧代码。
- **前端定位**：前端是 MVU 变量的显示端、交互端与少量即时反馈执行端；它可以写入 `系统.待处理交互`，但不得直接代替 AI 生成下一楼层剧情。
- **AIRP 闭环**：Action 由 UI 记录，Interpretation 由 AI + 世界书承接，Result 由 `<UpdateVariable><JSONPatch>` 回写，Presentation 由前端重新读取 MVU 显示。
- **心音边界**：普通命盘/心音点击只写 `系统.当前聚焦心声NPC`，不默认写入 `灵识窃取`；旧楼层残留的 `灵识窃取` 队列由世界书兼容承接并清空。
- **动态提示边界**：服装叙事、风声、心音、场景等会持续增长的规则，不得整表蓝灯常驻；优先使用“当前短摘要 + 绿灯扫描文本 + 世界书完整资料库”的按需注入模式。
- **同步口径**：`pnpm sync` 当前指向 `node tavern_sync.mjs`，且 `tavern_sync.mjs` / `tavern_sync.yaml` 均存在；交付时仍必须说明实际同步方式，并确认 SillyTavern 中启用的是最新 bundle/import 结果。
- **验收红线**：`pnpm build`、Vitest 通过只代表本地代码与前端产物通过；涉及 AI/MVU 回流时，必须额外确认最新楼层输出格式、MVU 最新变量回流和 `系统.待处理交互` 清空。
- **术语红线**：统一使用“牝奴”“太极/命轮”“莲灯前苑 / 醉玉小筑 / 绮梦幽阁”，不得回退为“牡奴”“八卦”或旧场景名。
- **事实红线**：报告、指南、PRD 中的数量、路径、产物大小、测试数必须来自实际命令输出，不得按记忆估算。

## 一、开发环境与工具链

### 1.1 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| UI 框架 | Vue 3 + Pinia | 组件化界面 + 状态管理 |
| 样式 | SCSS（scoped） | 预处理器，组件私有样式 |
| 校验 | Zod 4 | MVU 变量结构定义与运行时校验 |
| 构建 | Webpack 5 | 打包为自包含单文件 HTML |
| 测试 | Vitest + @vue/test-utils + happy-dom | 单元测试 + 组件测试 |
| 代码质量 | ESLint + Prettier | 静态分析 + 格式化 |
| 包管理 | pnpm | 依赖管理 |

> **注意**：项目安装了 Tailwind CSS 4，但 `tailwind.css` 注释明确标注"仅激活语法高亮，禁止实际使用"。组件样式一律使用 `<style lang="scss" scoped>`。

### 1.2 常用命令

```powershell

pnpm build:dev

# 生产构建（混淆压缩）
pnpm build

# 监听模式（配合热重载使用）
pnpm watch

# 运行测试
npx vitest run

# 单文件测试
npx vitest run src/雌堕合欢宗/界面/components/NpcCard.test.ts

# ESLint 检查 / 自动修复
pnpm lint
pnpm lint:fix

# Prettier 格式化
pnpm format

# 生成 JSON Schema（用于 schema.json 与 initvar.yaml 校验）
pnpm dump

# 同步/打包角色卡与世界书（执行后仍需确认 SillyTavern 实际启用最新结果）
pnpm sync

# 一键启动开发环境（Chrome 调试 + webpack watch）
.\start-dev.ps1
.\start-dev.ps1 -ChromeOnly   # 仅启动 Chrome
.\start-dev.ps1 -NoKill         # 不关闭现有 Chrome
```

### 1.5 Schema → initvar.yaml 工作流

```
schema.ts (zod 4 定义)
    ↓ pnpm dump
schema.json (JSON Schema)
    ↓ 引用
世界书/变量/initvar.yaml (# yaml-language-server: $schema=schema.json)
```

修改 `schema.ts` 后必须重新执行 `pnpm dump` 以更新 `src/雌堕合欢宗/schema.json`，否则 `initvar.yaml` 的自动校验和角色卡变量结构可能与源码脱节。若改动会影响 MVU 运行时，还要重建 `dist/var_structure.js`；若改动触及 `脚本/后端校验/**` 或其依赖，还要按 esbuild 独立脚本流程重建 `dist/backend_validate.js`。

### 1.5.1 前端 / 脚本 / 世界书三类改动的产物边界

| 改动类型 | 需要检查或重建 | 说明 |
|----------|----------------|------|
| Vue/SCSS 前端 | `pnpm build` -> `dist/雌堕合欢宗/index.html` | webpack 只负责状态栏前端和内联资源 |
| Schema | `pnpm dump`，必要时重建 `dist/var_structure.js` | `schema.ts` 是 MVU 结构权威源 |
| 后端校验脚本 | esbuild 重建 `dist/backend_validate.js` | webpack 不会刷新该文件 |
| 世界书 / index.yaml | `pnpm sync` 或手工 bundle/import | 本地文件变更不等于 SillyTavern 已启用 |
| 动态提示注入 | 单元测试 + `dist/backend_validate.js` 标识检查 | 例如服装叙事应能找到 `hehuan-current-outfit-summary` |

### 1.6 Codex CLI 文件写入规范

在 Codex CLI 环境中写文件时，首选**小块、可复查、可回滚**的增量修改。不要为了省事整文件覆盖，尤其不要用默认编码写中文路径或中文正文。

#### 推荐优先级

| 优先级 | 方式 | 适用场景 | 注意事项 |
|------|------|------|------|
| A | `apply_patch` | 小范围改动、可稳定匹配上下文的代码/文档 | patch 必须只描述差异，写后检查是否残留 `*** End Patch` |
| B | PowerShell 原生 UTF-8 + `.NET` API | 中文文档大段替换、当前环境确认 PowerShell UTF-8 正常时 | 使用 `[System.IO.File]::ReadAllText/WriteAllText` 与 `[Text.UTF8Encoding]::new($false)`，禁止默认编码 |
| C | 临时脚本 | 多处结构化替换、需要正则定位时 | 脚本执行后删除临时文件，保留验证命令输出 |

#### PowerShell UTF-8 安全模板

```powershell
$OutputEncoding = [Console]::OutputEncoding = [Text.UTF8Encoding]::new($false)
[Console]::InputEncoding = [Text.UTF8Encoding]::new($false)
$path = 'docs/教程/example.md'
$utf8 = [Text.UTF8Encoding]::new($false)
$text = [IO.File]::ReadAllText($path, $utf8)
$text = $text.Replace('旧内容', '新内容')
[IO.File]::WriteAllText($path, $text, $utf8)
```

#### 禁止做法

- 禁止用 `Set-Content` / `Add-Content` 默认编码写中文文档。
- 禁止用超长单行命令整文件覆盖。
- 禁止把 `apply_patch` 文本通过 shell 当普通命令执行。
- 禁止在含 `$`、反引号、Vue 模板或 SCSS 变量的内容里使用未转义的 PowerShell 双引号。

#### 写后复查

每次写入后至少执行一条定位复查命令，例如：

```powershell
Select-String -Path 'docs/教程/开发规范与智能体协作指南.md' -Pattern '当前权威协作口径|tavern_sync.mjs|UpdateVariable|待处理交互'
```
## 二、开发流程规范

### 2.1 标准开发流程（6 步）

```
需求理解 → 边界定位 → 编码实现 → 本地验证 → 实机/同步确认 → 交付留痕
```

#### 第 1 步：需求理解

- 阅读相关 PRD 文档（`docs/PRD/`），优先看用户指定的 PRD。
- 阅读架构文档（`docs/前端架构指南.md`、本指南、必要时再看 `docs/架构报告-*.md`）。
- 明确修改范围：是新增功能、样式调整还是 bug 修复
- 确认涉及的是前端、schema、后端校验脚本、世界书、还是构建/同步链路。

#### 第 2 步：边界定位

- UI 操作是否只需要即时反馈，还是必须写入 `系统.待处理交互`。
- 命盘/心音/图鉴类操作是否只是阅读焦点，避免误写待处理交互。
- 规则是否会持续增长；若会增长，优先使用动态短摘要 + 绿灯世界书，不要蓝灯常驻整表。
- 是否需要更新 `schema.ts`、`schema.json`、`世界书/变量/*.yaml` 或独立 dist 脚本。

#### 第 3 步：编码实现

- 遵循下面「三、代码质量标准」中的所有规则
- 使用现有 SCSS 变量和 mixin，不硬编码颜色/间距
- 新增组件时同步编写对应的 `.test.ts` 文件
- 修改 schema 时确保幂等性（`Schema.parse(Schema.parse(x)) === Schema.parse(x)`）并同步 `pnpm dump`

#### 第 4 步：本地验证

```powershell
# 运行所有测试
npx vitest run

# 运行特定模块测试
npx vitest run src/雌堕合欢宗/界面/guards.test.ts

# 清理旧产物
Remove-Item dist\雌堕合欢宗 -Recurse -Force

# 生产构建
pnpm build

# 验证产物（见 1.4 节）
```

如果修改了 `脚本/后端校验/**`、`脚本/服装叙事注入/**` 或 `validate.ts`，还必须按 `docs/教程/前端构建修复指南.md` 重建 `dist/backend_validate.js`。普通 `pnpm build` 不会刷新该文件。

#### 第 5 步：实机/同步确认

- 只改纯文档、纯样式或只读展示时，可由测试、构建和用户人工验收覆盖。
- 改了世界书、角色卡索引或第一条消息时，执行 `pnpm sync` 或说明手工 bundle/import 方式，并确认 SillyTavern 使用的是最新结果。
- 改了 AI/MVU 闭环时，用 `pnpm check:mvu`、CDP 或用户实机反馈确认 `<UpdateVariable><JSONPatch>`、变量回流与队列清空。

#### 第 6 步：交付留痕

- 确保相关 dist 产物无构建错误
- 需要跨会话时编写或更新 handoff 文档（`docs/handoff/`）
- 架构、变量、世界书、验收口径改变时同步更新 `docs/前端架构指南.md` 或本指南
- 记录已知问题和下一步建议

### 2.2 热重载开发流程（推荐）

日常开发推荐使用热重载模式，无需每次手动 `pnpm build`：

```
1. 启动监听模式: pnpm watch（或用 .\start-dev.ps1 一键完成步骤 1-2）
2. 启动带调试端口的 Chrome（参见 docs/教程/chrome调试启动指引.md）
3. 通过 MCP chrome-devtools 连接酒馆网页
4. 确认「酒馆助手-实时监听-允许监听」开关已启用
5. 修改 src/ 代码后，酒馆网页自动热重载
```

验证热重载是否生效：打开酒馆网页后，检查 `$("#extensions_settings")` 中的「酒馆助手-实时监听-允许监听」开关是否启用。

### 2.3 MVU 角色卡开发流程（额外步骤）

当开发 MVU 角色卡相关功能时，在标准流程基础上增加以下步骤：

```
需求理解 → schema 设计 → 变量同步 → 界面实现 → 脚本实现 → 世界书编写 → 测试 → 构建/同步 → 实机确认
```

#### 角色卡文件结构

```
src/雌堕合欢宗/
  index.ts              # 入口文件
  index.html            # HTML 模板（仅 <body> 静态内容）
  App.vue               # 根组件
  schema.ts             # MVU 变量结构定义（zod 4）
  schema.json           # 自动生成的 JSON Schema
  index.yaml            # 角色卡元数据（头像、描述、世界书引用）
  界面/                 # 前端界面项目
    components/         # Vue 组件
    composables/        # 组合式函数
    pages/              # 页面组件
    styles/             # SCSS 样式
    assets/avatars/     # NPC 立绘图片
    data/               # 静态数据
    store.ts            # Pinia store
    guards.ts           # 游戏逻辑函数
  脚本/                 # 后台脚本项目
    MVU/                # MVU 相关脚本
    变量结构/            # 变量结构脚本
    后端校验/            # 后端校验脚本
    服装叙事注入/        # 当前服装短摘要 + 绿灯扫描文本
  世界书/               # 世界书条目
    变量/               # 变量相关条目
    NPC/                # NPC 专属条目（每人一个 yaml）
    道具/               # 道具与服装叙事规则
    玩家/               # 玩家相关条目
    *.yaml              # 世界设定、宗门文化等
  第一条消息/           # 角色卡首条消息
    0.txt
```

#### schema 设计检查

- [ ] `schema.ts` 使用 zod 4（不是 zod 3）
- [ ] 所有字段有 `.prefault()` 默认值
- [ ] 数字字段用 `z.coerce.number()` + `_.clamp()` 限制范围
- [ ] 使用 `z.record()` 而非 `z.array()` 管理动态集合
- [ ] `Schema.parse(Schema.parse(x)) === Schema.parse(x)` 幂等性成立
- [ ] 修改后执行 `pnpm dump` 更新 `src/雌堕合欢宗/schema.json`
- [ ] 若 schema 字段影响世界书变量规则，同步更新 `世界书/变量/变量列表.yaml` 与 `世界书/变量/变量更新规则.yaml`

#### 三处同步检查

新增 NPC 时必须同步修改 3 处：
1. `_variables.scss`：`$npc专属色` map 加新颜色
2. `schema.ts`：`z.enum` 中加新名字
3. `guards.ts`：攻略链数组加新名字

可选同步：
4. `世界书/NPC/`：新增 NPC 专属世界书条目
5. `界面/assets/avatars/`：新增正道版 + 仙奴版立绘图片
6. `index.yaml`：更新角色卡元数据

## 三、代码质量标准

### 3.1 TypeScript 规范

| 规则 | 说明 |
|------|------|
| 禁止 `any` | 除非有充分理由并加注释 |
| 禁止未使用变量 | `noUnusedLocals` + `noUnusedParameters` 已在 tsconfig 启用 |
| 禁止隐式返回 | `noImplicitReturns: true` |
| 优先使用 `const` | ESLint `prefer-const: warn` |
| 优先使用酒馆助手接口 | `@types/function/*.d.ts` 优先于 `exported.sillytavern.d.ts` |
| 使用 `klona()` 脱 proxy | 写入酒馆变量前必须 `klona()` 去除 Vue 响应式代理 |

### 3.2 Vue 组件规范

- 使用 `<style lang="scss" scoped>`，禁止全局样式
- 禁止使用 `vh` 单位（iframe 环境下不可靠），使用 `width` + `aspect-ratio`
- 禁止 `position: absolute` 使主体脱离文档流
- 禁止在 `index.html` 中使用 `<link>` 或 `<script>` 引入外部文件
- 图片使用相对路径（无 `/` 开头），适配 iframe 子目录环境

### 3.3 SCSS 规范

- 颜色值使用 `_variables.scss` 中定义的 token
- 可复用视觉效果使用 `_mixins.scss` 中的 mixin
- 禁止在 mixin 中硬编码可被 CSS 变量覆盖的颜色（如需暗/亮双模式）
- 需要主题感知的元素使用 `var(--theme-*)` 或 `var(--npc-accent)`
- 需要覆写 mixin 硬编码色时，在 `_global.scss` 的 `html[data-theme=light]` 块中使用 `!important`

### 3.4 测试规范

- 测试框架：Vitest + @vue/test-utils + happy-dom
- **环境配置**：`vitest.config.ts` 全局设为 `environment: "node"`，各测试文件通过顶部 `// @vitest-environment happy-dom` 指令覆盖
- 只测外部行为，不测实现细节
- 每个组件/模块的 `.test.ts` 文件与源文件同目录
- `beforeEach` 中重置 composable 共享状态（模块级 `ref`/`reactive`）
- happy-dom 限制：修饰键需用 `Object.defineProperty` 强制设置，中文文件名需 `decodeURIComponent()`

### 3.5 ESLint 配置要点

项目 `eslint.config.mjs` 启用的规则：

- `import-x/no-cycle: error`：禁止循环依赖
- `no-var: error`：禁止 `var`
- `yoda: error`：条件表达式常量在左侧
- `pinia` 插件：Pinia store 规范
- `better-tailwindcss`：类名规范（已安装但主要项目不用 Tailwind）
- `vue` 插件：Vue SFC 规范

## 七、常见陷阱速查

| 陷阱 | 症状 | 解决 |
|------|------|------|
| `apply_patch` 残留 | SCSS/TS 编译失败 | 检查文件末尾删除 `*** End Patch` |
| 正则转义错误 | 二进制文件解析失败 | 检查 `\$` vs `$` |
| `asset/resource` vs `asset/inline` | iframe 无法加载外部文件 | 使用 `asset/inline` |
| composable 非单例 | 快捷键/面板状态不同步 | 提取 `ref` 到模块顶层 |
| `process.env.NODE_ENV` 守卫 | 功能在生产构建消失 | 移至 `onMounted` |
| happy-dom 修饰键 | 快捷键测试失败 | `Object.defineProperty` 强制设置 |
| CSS 特异性竞争 | 样式覆盖不生效 | 提高选择器特异性或用 `!important` |
| `color-mix()` + `var()` | 部分浏览器异常 | 用 JS 双色映射替代 |
| 全局字符串替换破坏主题 | 暗/亮模式文字不可见 | 替换前用正则确认匹配位置 |
| 中文文件名 URL 编码断言 | 测试中文件名匹配失败 | 先 `decodeURIComponent()` 再断言 |
| PowerShell `$` 插值 | shell here-string 中 `$variable` 被替换为空 | 优先用 `apply_patch`；大段中文替换用 `.NET` UTF-8 API，避免未转义双引号 |

## 十、收尾阶段经验总结（2026-05-21）

### 10.1 测试文件 mock 模式规范

所有测试文件中 mock store 时，**禁止使用 `import { __mockData } from '../store'`**。正确模式：

```typescript
// ✅ 正确：mockData 定义在模块顶层，vi.mock 通过闭包引用
const mockData = reactive({
  系统: { ... },
  牝奴: { ... },
  NPC: { ... },
  道具: { 拥有: {}, 装备: { 玩家: [] as string[], ... } },
});

vi.mock('../store', () => ({
  useDataStore: () => ({ data: mockData }),
}));
```

```typescript
// ❌ 错误：TS2305 - __mockData 未导出
vi.mock('../store', () => {
  const mockData = reactive({ ... });
  return { useDataStore: ..., __mockData: mockData };
});
import { __mockData as mockData } from '../store';
```

### 10.2 数组类型推断陷阱

空数组 `[]` 会被 TypeScript 推断为 `never[]`。mock 数据中的数组必须显式标注：

```typescript
道具: { 装备: { 玩家: [] as string[], 白芷: [] as string[], ... } }
```

遍历固定 NPC 列表时使用 `as const`：

```typescript
for (const npc of ['白芷', '苏芸', '纪兰', '沈月秋', '柳素衣'] as const) {
  mockData.NPC[npc].状态 = '已完成'; // ✅ 类型安全
}
```

### 10.3 DOM 类型断言

`document.querySelector()` 返回 `Element | null`，不支持 `.style`。需要断言为 `HTMLElement`：

```typescript
const el = document.querySelector(selector) as HTMLElement | null;
if (!el) return;
el.style.setProperty('--gradient-center-x', centerX + '%');
```

### 10.4 子 Agent 编码陷阱

- 子 agent 使用 Python Unicode 转义时，`牝`(U+7260) 和 `牡`(U+7261) 容易混淆
- PowerShell heredoc 中的正则表达式容易被转义破坏
- 推荐方案：用 `[System.IO.File]::WriteAllText()` 写 Python 脚本文件，再 `python fix.py` 执行


### 11.1 esbuild 独立脚本构建

本项目有 2 个独立 JS 文件需要用 esbuild 单独构建（webpack 不处理它们）：

- `dist/var_structure.js` — 注册 Zod Schema 到 MVU 框架
- `dist/backend_validate.js` — 后端变量校验

**构建步骤：**
1. 运行 esbuild 命令（见 `前端构建修复指南.md`）
2. `backend_validate.js` 需要 post-build 修补：替换 lodash import + 移除 export
3. 验证：控制台出现 `变量结构注册成功` 且无 404

**未构建的症状：** Zod 验证错误（支配次数 NaN、NPC undefined）

## 十二、v4 实机闭环协作规范（2026-05-24）

### 12.1 UI 交互与 AI 回复边界

- 前端 UI 是变量显示端和交互端，只能写入 `系统.待处理交互`、阅读焦点字段或执行已确认的即时本地表现，不允许直接调用 `generate()`。
- 用户发送下一条消息后，AI 才读取 `系统.待处理交互`，在新楼层自然叙事承接，并通过 JSON Patch 更新 MVU 变量。
- AI 不得重复执行前端已经完成的数据变化，例如重复扣灵石、重复增加背包库存、重复插入装备。
- 普通命盘/心音点击只负责聚焦 `系统.当前聚焦心声NPC`，不默认写入 `灵识窃取`；图鉴页聚焦心音/风声也不得写入待处理交互。
- 前端的“心防反震”“按钮抖动”“归零湮灭”等只属于视觉反馈；实际好感度扣减、心音揭示、剧情解锁仍由下一楼层 AI Patch 决定。

### 12.2 变量输出格式硬约束

AI 变量更新必须使用完整标签块，裸 JSON Patch 无法保证被 MVU 解析器正式应用：

```xml
<UpdateVariable>
<Analysis>Handled pending UI interactions and updated MVU state.</Analysis>
<JSONPatch>
[
  { "op": "replace", "path": "/NPC/白芷/soul_whisper/text", "value": "……" },
  { "op": "replace", "path": "/NPC/白芷/soul_whisper/is_revealed", "value": true },
  { "op": "replace", "path": "/系统/待处理交互", "value": [] }
]
</JSONPatch>
</UpdateVariable>
```

禁止只输出 `变量更新情况`、YAML 摘要或裸数组。实机已验证：裸数组会留在正文里，但 `soul_whisper` 不会回流到最新 MVU 状态。
### 12.2.1 v4 系统字段 Patch 操作口径

- 对 `schema.ts` / `initvar.yaml` 已定义的 v4 系统字段，统一使用 `replace` 写入完整值，不再使用 `insert`。
- 必须使用 `replace` 的字段包括：`/系统/当前场景`、`/系统/时辰`、`/系统/待处理交互`、`/系统/场景上下文`、`/系统/风声列表`、`/系统/当前追查风声ID`。
- 实机排查结论：`NPC.*` 旧字段的 `replace` 能落地，但 v4 系统字段使用 `insert` 时可能不会进入最新 `stat_data`；因此世界书和额外解析模型必须按 `replace` 口径输出。
- 若旧聊天快照缺少 v4 字段，`backend_validate.js` 的 `ensureV4SystemFields` 会在 `VARIABLE_UPDATE_ENDED` 后补齐默认值；前提是酒馆脚本库实际加载了当前发布脚本 `https://cdn.jsdelivr.net/gh/chen986347310-svg/tavern_helper_template@main/public/hehuan/scripts/backend_validate.js`，或本地调试时加载等价的本地构建产物。

### 12.3 开放动态场景协作约束

- 当前 v4 已从“只有三处地点”升级为开放动态场景：`系统.场景上下文.地点 / 子区域 / 氛围 / 故事钩子` 可以写入任意贴合合欢宗世界观的具体场景。
- `莲灯前苑`、`醉玉小筑`、`绮梦幽阁` 仍是前端底部快捷入口、默认文案和旧数据回退锚点；不得把它们理解为世界可发生地点的数量上限。
- AI 写入新场景时必须同步写入完整 `系统.场景上下文`，至少包含 `地点`、`子区域`、`公开度`、`在场NPC`、`氛围`、`故事钩子`。
- 角色栏只根据 `系统.场景上下文.在场NPC` 显示当前在场核心 NPC；不在场 NPC 零占位、零卡片。

### 12.4 文档留痕要求

任何实机验收失败都必须记录三件事：
1. CDP 读取到的最新楼层号与 iframe id。
2. 正文输出格式是否包含 `<UpdateVariable>` / `<JSONPatch>`。
3. `Mvu.getMvuData({ type: 'message', message_id: 'latest' })` 中对应变量是否实际回流。
### 12.5 世界书/角色卡同步口径（当前必须显式确认）

当前仓库中 `package.json` 声明 `"sync": "node tavern_sync.mjs"`，且 `tavern_sync.mjs` / `tavern_sync.yaml` 均存在。同步链路可用性仍需要按本机酒馆环境确认，因此：

- 不得把 `pnpm build` 误认为世界书/角色卡已进入 SillyTavern。
- 本地 `src/雌堕合欢宗/世界书/**`、`index.yaml`、`第一条消息/**` 的修改，只有在重新 sync、bundle/import 到酒馆实际启用角色卡或世界书后，才算进入实机环境。
- 交付记录必须写清“同步方式”：`pnpm sync`、手工导入、替代脚本，或用户已在酒馆中验证。
- 若 `pnpm sync` 失败，先记录错误输出和使用的配置名，不得继续宣称世界书已同步。

### 12.6 v4 实机验收最小三件套

每次验证 UI → AI → MVU 闭环，至少记录：

1. 最新楼层号与状态栏 iframe id，例如 `#6` / `TH-message--6--0`。
2. 最新正文是否包含完整 `<UpdateVariable>` 与 `<JSONPatch>`，而不是 `变量更新情况`、YAML 摘要或裸 JSON Patch。
3. `Mvu.getMvuData({ type: 'message', message_id: 'latest' })` 中目标变量是否真实变化，并确认 `系统.待处理交互` 已被清空。

只有三件套全部通过，才能判定前端交互真正影响了下一楼层游戏进程。

### 12.7 v4 MVU CDP 自动验收脚本

为避免每个新楼层都手工拼 CDP 代码，已新增 `cdp-v4-mvu-check.mjs` 作为实机验收工具。

常用命令：

```powershell
# 自动探测 9222 / 9223，仅读取最新楼层 MVU，不写变量
pnpm check:mvu

# 指定 Chrome CDP 端口
node .\cdp-v4-mvu-check.mjs --port 9222

# 输出 JSON，方便贴进报告或继续脚本处理
node .\cdp-v4-mvu-check.mjs --port 9222 --json

# 显式触发一次空校验；只用于验证 backend_validate.js 钩子，不作为日常必跑项
node .\cdp-v4-mvu-check.mjs --port 9222 --apply-empty-validation
```

验收脚本检查内容：

- 本地热更新入口：`dist/var_structure.js`、`dist/backend_validate.js`、状态栏 `index.html` 是否 HTTP 200。
- 最新楼层 `stat_data.系统` 是否包含 v4 字段：`时辰`、`当前场景`、`待处理交互`、`场景上下文`、`风声列表`、`当前追查风声ID`。
- `系统.当前场景` 是否为开放字符串，`系统.场景上下文` 是否包含地点、子区域、公开度、在场 NPC、氛围与故事钩子。
- `系统.风声列表` 是否为数组且不超过 3 条。
- `__TEST_applyValidatedUpdate` 是否能在酒馆助手脚本 iframe 上下文中被定位。

判定口径：`Result: PASS` 才能继续下一轮 UI→AI→MVU 闭环验收；若失败，优先看输出中的 `[FAIL]` 项，不要先猜 UI 渲染问题。

## MVU 楼层人工验收流程（v4.1）

当 CDP 或额外模型解析无法稳定读取最新楼层变量时，使用 `docs/变量MVU.md` 作为临时收件箱：

1. 用户把当前楼层完整 `<UpdateVariable>` 原文粘贴到 `docs/变量MVU.md`。
2. 开发者读取并检查：是否存在 `JSONPatch`、是否有多个块、是否清空 `系统.待处理交互`、是否同步 `系统.当前场景` 与 `系统.场景上下文`。
3. 若同一楼层有多个块，按出现顺序合并判断；同路径以后块为准。
4. 验收完成后立即清空 `docs/变量MVU.md`，保留文件标题和说明，等待下一楼层。
5. `docs/变量MVU.md` 不作为长期证据；长期结论必须回填 PRD 或验收记录。

### 中文文档写入要求

在 PowerShell 中写入中文 Markdown/YAML，优先使用：

```powershell
[System.IO.File]::WriteAllText($path, $content, [System.Text.UTF8Encoding]::new($false))
```

避免使用会破坏中文、反引号或 Markdown 表格的复杂 here-string 与跨 shell 转义。
