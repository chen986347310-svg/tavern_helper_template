# 世界运行核心与 P2 主控台阶段验收报告

日期：2026-05-25

## 1. 报告结论

本阶段已完成“世界运行核心”和“P2 牝奴期主控台”的主要技术闭环：玩家行为可以进入待处理交互队列，AI 下一楼层会被动态注入层要求承接，并在同次 JSONPatch 中回写时间、欲海、事件后果、P2 日课、命令、支配者、调教记录或羞名风声等结构化字段。

P2 不再只是失败结局展示页，而是具备第二阶段玩法雏形：牝印核心、当前日课、当前支配者、听风羞名、烙名标签、状态栏牝印反馈和 DebugPanel 可观测性已经接入。

当前自动化验收通过，构建通过。下一步重点不建议继续盲目加功能，而应先做 CDP 实机闭环验收和变更分组交付。

## 2. 为什么要做这次升级

原系统的核心问题是：玩家点击、购买、使用道具或进入 P2 后，世界容易停留在“前端状态变化”或“AI 自由发挥”层面，缺少强制性的运行账本。这样会导致三个问题：

1. **AI 不一定承接玩家操作**：玩家买了剧情钥匙、追查风声或触发羞名事件后，AI 可能继续普通聊天。
2. **世界缺少时间和后果**：一次行动到底过去多久、欲海是否更接近、事件是否留下记录，没有统一闭环。
3. **P2 容易变成终局页面**：玩家进入牝奴期后，如果没有日课、命令、支配者和羞名风声，玩法会停止。

本阶段的升级目标就是把这些行为纳入统一链路：

```text
玩家点击/购买/使用/追查
-> 前端写入 系统.待处理交互
-> 动态注入层生成强制承接提示
-> AI 正文承接
-> 同次 JSONPatch 回写变量
-> 时间/欲海/事件/P2 后果落账
-> 队列清空
```

### 3.1 世界时间与后果账本

已完成：

- 新增世界时间、欲海状态和事件记录字段。
- 实现 `worldTime.ts` 纯函数，支撑时间流逝、欲海侦测和事件留痕。
- 动态注入提示已要求 AI 承接待处理交互时同步判断时间和后果。
- DebugPanel 可查看时间、欲海和事件记录。

价值：

- 世界开始拥有“行动之后会发生什么”的结构化记忆。
- 后续商城、剧情钥匙、P2 日课、NSFW 欲海侦测都能接在同一条时间轴上。

### 3.2 商城、道具、剧情钥匙入口账本

已完成：

- 待处理交互可携带道具显示名、器阶、作用部位、丹药分类、作用线、AI 短提示等叙事辅助字段。
- 已形成剧情钥匙、场景令牌、特殊道具和商城整改 PRD 与计划文件。
- 商城/背包/道具相关文件已进入同一轮工作区，自动化测试通过。

价值：

- 玩家购买或使用特殊物品后，不只是库存变化，而是能成为下一楼层剧情入口。
- 为“钥匙风声”“场景追寻”“特殊道具触发剧情”提供了统一承接方式。

### 3.3 P2 牝奴期运行闭环

已完成：

- `phase2Runtime.ts` 已实现 P2 日课、调教结算、调教记录和世界事件生成。
- `phase2Rumor.ts` 已实现从 P2 事件生成羞名风声。
- `牝奴.羞名标签` 已进入 schema、initvar、变量列表和 DebugPanel。
- P2 羞名风声点击后会写入 `系统.待处理交互`。
- 动态注入层会要求 AI 把 P2 羞名风声承接为传唤、日课异动、公开凝视或支配事件。

价值：

- P2 具备了“日课 -> 调教事件 -> 后果账本 -> 羞名风声 -> 再次承接”的闭环。
- 羞耻感不只来自单次文本，而来自宗门持续传播和凝视。

## 5. 仍未完成的工作

### 5.1 CDP 实机闭环验收

自动化测试已经通过，但还需要真实 SillyTavern 环境验证：

- 点击 P2 羞名风声后，`系统.待处理交互` 是否真实写入。
- 用户发送下一条消息后，AI 是否承接为 P2 事件。
- AI 是否在同次 JSONPatch 中更新 P2 字段并清空队列。
- 时间、欲海、事件记录是否真实回流。

### 5.2 变更分组提交

当前工作区很大，包含多条工作线。还没有按功能包拆分提交。

建议拆成：

1. 时间与后果账本。
2. 商城、道具、剧情钥匙入口账本。
3. P2 运行闭环。
4. P2 UI 主控台与状态栏融合。
5. DebugPanel 与文档留痕。

### 5.3 P2 UI 视觉细化

当前 P2 主控台是最小可用版本，还需要实机截图后再判断：

- 第一屏是否拥挤。
- P1/P2 风格是否自然连续。
- 长命令是否省略自然。
- 羞名风声是否有足够凝视感和羞耻感。
- 支配者、日课、命令之间是否需要更强视觉关联。

### 5.4 剧情钥匙多次购买与触发上限实机验证

需要实测：

- 同一钥匙多次购买是否受限。
- 是否会重复进入待处理队列。
- 风声联动后是否稳定引导玩家追寻。
- 触发后是否写入事件记录并避免重复触发。

## 6. 风险与缓解

### 风险 1：自动化通过但实机 AI 不稳定承接

缓解：下一步必须做 CDP 实机闭环验收，并保存检查命令、变量截图或日志。

### 风险 2：工作区过大，后续难以回滚或交付

缓解：按验收包拆分提交或打包，每包单独跑测试。

### 风险 3：P2 UI 视觉强但可能与 P1 割裂

缓解：已在架构文档中明确 P2 必须继承 P1 骨架；后续通过截图验收微调。

### 风险 4：P2 羞名风声刷屏

缓解：当前风声列表沿用 3 条上限和去重规则；后续实机观察节奏，如有必要再加生成冷却。

### 风险 5：世界书 token 压力继续增长

缓解：下一阶段应继续把长表格和规则拆为按需触发资料库，蓝灯只保留短规则。

## 7. 下一步建议

推荐按以下顺序推进：

1. **CDP 实机闭环验收**：优先验证真实环境中队列、AI 承接、JSONPatch、变量回流是否稳定。
2. **按验收包拆分提交或归档**：避免当前巨大工作区继续膨胀。
3. **生成实机验收记录**：记录触发动作、变量变化、AI 正文承接和队列清空情况。
4. **P2 UI 视觉细化**：基于真实截图调颜色、密度、长文案、省略和凝视感。
5. **世界书瘦身**：把时间规则、P2 日课、道具叙事和钥匙触发规则继续拆成按需触发资料库。
