# PRD: 雌堕合欢宗 · MVU角色卡完整系统

## Problem Statement

用户正在开发一个基于酒馆助手MVU框架的修仙角色卡项目「雌堕合欢宗」。当前项目只有世界书NPC设定和空白前端骨架，缺少MVU变量系统、脚本、前端界面、道具经济系统、双结局机制等核心模块。需要将已有的设计讨论转化为可运行的完整角色卡。

## Solution

构建一个包含双阶段（攻略期 + 牝奴期）的完整MVU角色卡系统，包含：
- 好感度/攻略值双轨制，带粘滞触发防爆数值
- 灵石经济系统，支持购买道具/服装/场景/剧情/时间延长
- 交互式前端界面（首页/商城/背包/图鉴四页面）
- Phase 2牝奴系统，EJS分阶段解析堕落度
- 后端变量校验兜底，防止AI写坏数据

## Implementation Decisions

### 1. MVU变量结构（schema.ts）

```
stat_data:
  系统:
    阶段: "攻略期" | "牝奴期"
    剩余天数: number
    灵石: number
    已使用阵法: boolean

  牝奴:
    堕落度: number (0~100)
    牝阴决层数: number (0~9)
    上次支配者: string
    支配次数: { 白芷/苏芸/纪兰/沈月秋/柳素衣: number }
    改造进度: { 泌乳/肛门/憋尿: boolean }

  NPC:
    白芷/苏芸/纪兰/沈月秋/柳素衣:
      好感度: number (0~100)
      攻略值: number (0~100)
      粘滞计数: number
      状态: "未开始" | "进行中" | "已完成"

  道具:
    拥有: { [道具名]: number }
    装备: { 玩家/白芷/苏芸/纪兰/沈月秋/柳素衣: string[] }

  场景: { 已解锁: string[] }
  剧情: { 已解锁: string[] }
```

使用 `z.coerce.number()` + `z.transform(_.clamp)` 保证数值范围。
NPC结构完全一致，方便循环处理。
布尔值使用肯定语义（`已使用阵法`而非`阵法状态`）。

### 2. 好感度/攻略值双轨制

- 好感度：日常接触提升，NSFW影响微小
- 攻略值：只通过亲密/NSFW提升，受好感度门槛约束
- 解锁门槛：30+/60+/90+
- 攻略值公式：`floor(基础值 × 好感度 ÷ 50)`
- 粘滞触发：亲密3层、NSFW2层，中断归零
- 攻略值=100 → 玄媚仙奴 → 引荐下一NPC

### 3. 灵石经济系统

**收入：**
- 灵石获取 = NPC境界系数 × 攻略值增量
- 境界系数：白芷10 / 苏芸20 / 纪兰40 / 沈月秋60 / 柳素衣100
- 全攻略总收入约23,460灵石
- 攻略柳素衣完成奖励1,000,000灵石

**支出：**
- 时间延长道具：10,000灵石
- 改变阵法（真结局）：500,000灵石
- 女修服装25+件：500~3500灵石
- NSFW道具12种：500~3000灵石
- 永久丹药6种：1500~3000灵石
- 场景解锁4个：1000~2000灵石
- 特殊剧情5条：1500~4000灵石
- 全买约47,600灵石，玩家必须做选择

### 5. 前端架构

四页面Tab切换：首页 → 商城 → 背包 → 图鉴

组件结构：

- components/：SystemBar、NpcCard、ItemCard、EquipPanel、NpcDetail、PageNav

### 6. Phase 2牝奴系统

**触发：** 剩余天数归零，自动进入

**进入时状态变化：**
- 好感度全部清零，攻略值取消
- 灵石全部上缴归零
- 道具和服装随机装备在玩家身上
- 剩余天数 → 堕落度（0~100%）

**七日改造：** 固定剧情模板，AI可根据当前场景微调

**堕落度6阶段：**
- 0-10%：清醒隐忍期
- 10-30%：动摇挣扎期
- 30-50%：本能屈从期
- 50-70%：淫纹发情期
- 70-90%：欲壑难填期
- 90-100%：彻底雌堕期

**实现方式：** 单个世界书条目使用EJS分阶段解析，根据堕落度输出对应阶段的行为指导和必须触发事件清单。

**粘滞触发：** 采补3层、羞辱指令2层、寸止/憋尿2层、多人同时1层

**NPC支配：** AI自由判断时机，但每阶段必须触发清单内所有事件

**结局：** 单一结局——彻底雌堕，无挣脱可能

### 8. 后端校验机制

通过 `VARIABLE_UPDATE_ENDED` 事件实现：
- 好感度<30时攻略值强制归零
- 攻略值不能倒退
- 粘滞计数器不能超过阈值
- 灵石不能为负
- 堕落度只能升不能降（Phase 2）
- 牝阴决层数不能超过9

### 9. NPC互相影响

- 已成为玄媚仙奴的NPC会主动来找玩家
- NPC之间互动产生灵石，按好感度最高比例分配
- 攻略链过渡钩子：白芷→提供苏芸情报→苏芸→干姐姐引荐纪兰→纪兰→阳气异常需长老审→沈月秋→需掌门亲审→柳素衣

## Testing Decisions

### 测试原则

- 测试外部行为（变量更新是否正确、UI显示是否准确），不测试实现细节
- 校验函数（guards.ts）需要单元测试，覆盖边界条件
- Pinia store需要测试变量读写一致性
- 后端校验逻辑需要测试各种非法输入的修正

### 需要测试的模块

1. **guards.ts**：好感度门槛检查、灵石扣款检查、道具装备校验
2. **store.ts**：MVU变量读写、Phase 1/Phase 2状态切换
3. **data/*.ts**：道具常量表完整性、价格和门槛数据正确性
4. **后端校验脚本**：VARIABLE_UPDATE_ENDED事件中的各项校验逻辑

### 不需要测试的部分

- Vue组件渲染（依赖酒馆iframe环境，无法单元测试）
- AI输出内容（由世界书规则约束，非代码控制）
- EJS模板解析（酒馆内置功能）

## Out of Scope

1. **酒馆助手扩展本身的开发** — 假设已安装并可用
2. **AI模型的行为控制** — 通过世界书规则引导，非代码强制
3. **酒馆主题/样式定制** — 使用酒馆默认主题
4. **多人联机** — 单人角色卡
5. **移动端适配** — 针对桌面端酒馆界面
6. **国际化** — 全中文
7. **Phase 2的挣脱结局** — 确认为单一结局
8. **NPC之间的情感关系** — 仅通过攻略链钩子串联，无独立NPC间关系系统

## Further Notes

### 开发顺序建议

1. schema.ts + initvar.yaml（基础）
2. 脚本/变量结构/index.ts（注册schema）
3. 世界书变量系统条目（变量列表、更新规则、输出格式）
4. 道具系统世界书条目

7. Phase 2牝奴系统世界书条目

### 关键约束

- 使用Zod 4（非3.x），遵循酒馆助手MVU框架规范
- 变量结构必须幂等：`Schema.parse(Schema.parse(input))` === `Schema.parse(input)`
- 前端必须在iframe内运行，使用酒馆助手提供的API
- 世界书条目使用YAML格式，支持EJS模板
- 所有中文键名，统一命名规范
