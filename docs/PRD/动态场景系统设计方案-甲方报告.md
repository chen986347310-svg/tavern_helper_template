# 合欢宗 · 灵缘流转 — 动态场景系统设计方案

> **文档性质**：甲方设计报告 | **版本**：v1.0 | **日期**：2026-05-22
> **设计目标**：用一套「活的世界」系统替代原有的固定三场景，让每一次进入场景都是一次未知的缘分。

---

## 一、旧系统的问题

原有的三场景系统（莲灯前苑/醉玉小筑/绮梦幽阁）存在以下根本性缺陷：

| 问题 | 表现 | 影响 |
|------|------|------|
| NPC 静态绑定 | 每个 NPC 固定在一个场景，不会移动 | 玩家切换场景只是切换 NPC 列表，没有探索感 |
| 场景无生命 | 场景本身没有氛围、没有事件、没有变化 | 场景只是 UI 标签，不是修仙世界的一部分 |
| 交互无上下文 | 同一个 NPC 在任何场景的交互方式完全相同 | 无法体现「在药庐帮忙」vs「在厢房独处」的差异 |
| 无法扩展 | 新增场景需要改 schema、改组件、改测试、改世界书 | 每次扩展都是全栈级变更 |
| 对话式游戏脱节 | 对话式游戏的核心是「情境叙事」，但固定场景无法生成情境 | 玩家只是在选 NPC，不是在探索世界 |

---

## 二、新系统设计：「灵缘流转」

### 2.1 核心理念

**场景不是「选单」，是「遇见」。**

在对话式游戏中，场景的意义在于为 AI 提供叙事上下文——告诉 AI「此刻、此地、此人、何事」。新系统用两层架构实现：

- **物理层**：合欢宗的建筑空间（固定的地点，但有子区域）
- **涌现层**：NPC 日程 + 时辰 + 事件 → 动态生成「此刻正在发生什么」

### 2.2 架构总览

`
┌─────────────────────────────────────────────────┐
│                 灵缘流转引擎                      │
│                                                   │

│  │ 时辰系统 │  │ NPC 日程 │  │ 世界事件 │        │
│  │ 晨/午/酉/亥│  │ 行程表   │  │ 触发器   │        │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘        │
│       │              │              │              │
│       └──────────┬───┴──────────────┘              │

## 三、物理层：合欢宗地图

### 3.1 地点体系

原有的 3 个场景扩展为 **8 个核心地点**，每个地点有 2-3 个子区域：

| 地点 | 子区域 | 解锁条件 | 叙事属性 |
|------|--------|----------|----------|
| 莲灯前苑 | 花径 / 回廊 / 莲池畔 | 默认 | 公开社交、灯影交错、日常接触 |
| 醉玉小筑 | 雅座 / 后厨 / 露台 | 默认 | 半私密、酒香微醺、暧昧试探 |
| 绮梦幽阁 | 正厅 / 内室 / 密道入口 | 默认 | 极度私密、深度攻略、心跳加速 |
| 药庐 | 炼丹房 / 药圃 / 白芷工位 | 好感度≥20 | 白芷专属、药香弥漫、日常帮忙 |
| 经阁 | 藏书区 / 禁区 / 纪兰书房 | 好感度≥30 | 纪兰专属、书香肃穆、知识探索 |
| 演武场 | 擂台 / 休息区 / 苏芸练功房 | 好感度≥20 | 苏芸专属、汗气蒸腾、肉体碰撞 |
| 长老殿 | 正殿 / 偏殿 / 沈月秋密室 | 好感度≥50 | 沈月秋专属、权力中心、禁忌诱惑 |
| 掌门殿 | 大殿 / 后殿 / 柳素衣静室 | 好感度≥70 | 柳素衣专属、威压弥漫、极致反差 |

### 3.2 地点属性

每个地点携带以下属性，影响场景生成：

`
地点属性:
  名称: string
  子区域: string[]
  公开度: '公开' | '半私密' | '私密'  // 影响 NPC 行为模式
  氛围关键词: string[]              // 供 AI 参考的氛围描写词
  可用交互: string[]                // 该地点特有的交互类型
  灵气浓度: number                  // 影响修炼/采补效率
  解锁条件: { 好感度?: number, 剧情?: string, 购买?: number }
`

## 四、涌现层：NPC 日程系统

### 4.1 核心理念

**NPC 不是 UI 挂件，是有自己生活的「人」。**

每个 NPC 在不同时辰出现在不同地点，做不同的事。玩家需要「找到」NPC 才能互动，而不是从列表里选。

### 4.2 NPC 日程表

| 时辰 | 白芷 | 苏芸 | 纪兰 | 沈月秋 | 柳素衣 |
|------|------|------|------|--------|--------|
| 晨时 | 药庐·药圃（采药） | 演武场·擂台（晨练） | 经阁·藏书区（整理典籍） | 长老殿·正殿（处理宗务） | 掌门殿·后殿（闭关打坐） |
| 午时 | 莲灯前苑·花径（晒药） | 醉玉小筑·雅座（小酌） | 经阁·纪兰书房（抄录） | 莲灯前苑·回廊（巡视） | 莲灯前苑·莲池畔（散步） |
| 酉时 | 药庐·炼丹房（炼丹） | 绮梦幽阁·正厅（沐浴后） | 醉玉小筑·后厨（独酌） | 长老殿·偏殿（密谈） | 掌门殿·大殿（处理密报） |
| 亥时 | 绮梦幽阁·内室（秘密实验） | 绮梦幽阁·内室（就寝前） | 经阁·禁区（夜读禁书） | 长老殿·沈月秋密室（修炼） | 掌门殿·柳素衣静室（独处） |

### 4.3 日程动态偏移

NPC 日程不是完全固定的，会根据以下因素偏移：

| 偏移因素 | 影响 | 示例 |
|----------|------|------|
| 好感度等级 | 高好感度 NPC 会主动出现在玩家附近 | 白芷好感度≥60时，午时会出现在玩家所在地点 |
| 攻略状态 | 进行中的 NPC 有额外「等待」行为 | 苏芸状态=进行中时，晨时偶尔不去演武场而在醉玉小筑等玩家 |
| 世界事件 | 特殊事件改变所有 NPC 的位置 | 「宗门大比」时所有 NPC 集中在演武场 |
| 时辰偏移 | 某些 NPC 在特定条件下改变行程 | 纪兰好感度≥80时，亥时不再去禁区而是去绮梦幽阁 |

### 4.4 场景上下文生成

当玩家进入一个地点时，系统自动生成「场景上下文」：

`
场景上下文 = {
  地点: '醉玉小筑',
  子区域: '雅座',
  时辰: '酉时',
  在场NPC: [
    { 名: '苏芸', 活动: '沐浴后独酌', 心情: '慵懒微醺', 穿着: '薄纱浴袍' },
    { 名: '纪兰', 活动: '独酌抄录', 心情: '微醺放松', 穿着: '日常儒裙' }
  ],
  氛围: ['酒香', '暮色', '微醺', '纱帘轻垂'],
  可用交互: ['搭讪', '敬酒', '同坐', '灵识窃取', '送礼'],
  特殊事件: null
}
`

这个上下文直接作为 AI 的叙事参考，让 AI 知道「此刻此地正在发生什么」。

## 五、世界事件系统

### 5.1 事件类型

| 事件类型 | 触发条件 | 影响范围 | 示例 |
|----------|----------|----------|------|
| 日常事件 | 每日自动触发 | 单 NPC | 白芷炼出新丹药，请玩家试药 |
| 关系事件 | 好感度/攻略值达到阈值 | 单 NPC | 苏芸攻略值≥50，邀请玩家去演武场密室 |
| 宗门事件 | 天数到达特定日期 | 全局 | 第 15 天「宗门大比」，所有 NPC 集中演武场 |
| 隐藏事件 | 特殊条件组合 | 单 NPC | 白芷好感度≥80 + 亥时 + 药庐 → 触发「禁忌实验」剧情 |
| 玩家事件 | 购买/使用特定道具 | 单 NPC | 使用「媚体丹」→ 当前场景氛围改变 |

### 5.2 事件对场景的影响

`
事件触发 → 修改场景上下文 → AI 根据新上下文叙事

示例：
  原始上下文: 醉玉小筑·雅座，酉时，苏芸独酌
  事件触发: 「苏芸的秘密」（好感度≥50）
  修改后上下文: 醉玉小筑·雅座，酉时，苏芸独酌 + 心事重重 + 欲言又止
  AI 叙事: 「苏芸端着酒杯，目光却不在酒上。她似乎想说什么，又在犹豫……」
`

## 六、玩家交互流程

### 6.1 旧流程 vs 新流程

**旧流程**：
`
玩家选择场景 → 看到该场景的 NPC 列表 → 点击 NPC → 互动
`

**新流程**：
`
玩家选择地点 → 系统生成场景上下文（谁在、做什么、什么氛围）
  → 玩家看到「此刻此地的实时画面」
  → 选择互动对象和方式
  → AI 根据上下文 + 互动选择叙事
`

### 6.2 UI 变化

| 旧 UI | 新 UI |
|-------|-------|
| 3 个固定场景按钮 | 地点选择器（8 个地点，锁定的灰显） |
| 场景下显示 NPC 列表 | 场景实时画面 + NPC 活动描述 |
| 空场景显示静态留白文案 | 空场景显示「此刻此地无人，但氛围描写」 |
| NPC 卡片直接显示 | NPC 卡片附带「此刻活动」标签 |

### 6.3 交互示例

**示例 1：日常偶遇**
`
玩家进入：醉玉小筑（酉时）
场景上下文：苏芸沐浴后独酌，纪兰独酌抄录
氛围：酒香、暮色、微醺
玩家选择：向苏芸敬酒
AI 叙事：（根据苏芸好感度、当前穿着、心情生成对话）
`

**示例 2：深夜密会**
`
玩家进入：绮梦幽阁·内室（亥时）
场景上下文：白芷秘密实验，氛围紧张
玩家选择：灵识窃取
AI 叙事：（根据窃听成功率、白芷心防阶段生成窥心结果）
`

**示例 3：宗门大比**
`
世界事件触发：第 15 天「宗门大比」
所有 NPC 集中在演武场
场景上下文：擂台上苏芸对阵纪兰，白芷在场边观战，沈月秋评判
玩家选择：为苏芸加油
AI 叙事：（根据苏芸好感度生成反应）
`

## 八、新旧系统对比

| 维度 | 旧系统（三场景） | 新系统（灵缘流转） |
|------|------------------|------------------|
| 场景数量 | 3 个固定 | 8 个核心 + 动态扩展 |
| NPC 位置 | 静态绑定 | 日程驱动 + 动态偏移 |
| 场景意义 | UI 标签 | 叙事上下文生成器 |
| 交互上下文 | 无（任何场景交互方式相同） | 有（地点 + 活动 + 氛围决定交互方式） |
| 世界事件 | 无 | 支持日/关系/宗门/隐藏事件 |
| 扩展性 | 全栈级变更 | 新增地点只需注册表 + 日程表 |
| AI 叙事支持 | 粗糙（只有场景名） | 精细（地点 + 子区域 + NPC 活动 + 氛围 + 事件） |
| 玩家体验 | 选 NPC | 探索世界 |
| 对话式游戏适配 | 弱（静态选择） | 强（动态涌现） |

## 十、玩家体验提升

### 10.1 从「选 NPC」到「遇见 NPC」

旧系统：玩家打开首页，看到 3 个场景按钮，点击一个看到 NPC 列表，选择一个互动。
→ 体验：像在选菜单。

新系统：玩家进入醉玉小筑，看到「酉时，暮色渐浓，酒香弥漫。苏芸沐浴后倚在雅座独酌，薄纱浴袍下肌肤若隐若现。纪兰在角落抄录，笔尖微颤。」
→ 体验：像在推开门，走进一个活的世界。

### 10.2 从「固定互动」到「情境互动」

旧系统：对白芷的交互方式在任何场景都一样。

新系统：
- 药庐·晨时：帮白芷采药 → 日常好感提升
- 莲灯前苑·午时：在花径偶遇白芷晒药 → 暧昧互动
- 绮梦幽阁·亥时：撞见白芷秘密实验 → 触发禁忌剧情

### 10.3 从「静态世界」到「活的世界」

旧系统：世界不会变化，NPC 不会移动。

新系统：
- 第 10 天：宗门传来消息，附近发现妖兽 → 所有 NPC 议论纷纷
- 第 15 天：宗门大比 → 所有 NPC 集中演武场
- 第 20 天：柳素衣闭关 → 掌门殿关闭，柳素衣不可互动
- 第 25 天：阵法异变 → 所有 NPC 行为异常，氛围紧张

## 十一、风险与对策

| 风险 | 等级 | 对策 |
|------|------|------|
| AI 无法准确理解复杂场景上下文 | P1 | 场景上下文格式标准化，提供示例模板 |
| NPC 日程过于复杂导致玩家找不到人 | P2 | 提供「灵识感应」功能，提示 NPC 当前位置 |
| 世界事件与主线剧情冲突 | P2 | 事件系统支持优先级，主线事件覆盖日常事件 |
| 性能问题（每次进入场景都要计算上下文） | P3 | 场景上下文缓存，仅在时辰变化时重新计算 |
| 扩展新地点需要修改多个文件 | P3 | 地点注册表模式，新增地点只需添加注册条目 |

---

## 十二、总结

「灵缘流转」系统的核心价值：

1. **活的世界**：NPC 有自己的日程和生活，不是 UI 挂件
2. **情境叙事**：每次进入场景都是一次独特的「遇见」，AI 有丰富的上下文来生成叙事
3. **动态扩展**：新增地点/事件/NPC 只需注册表级变更，不需要全栈重构
4. **对话式游戏适配**：场景系统为 AI 提供结构化的叙事上下文，而不是简单的 NPC 列表
5. **玩家沉浸**：从「选菜单」到「推开门」，真正进入修仙世界

> 这不是一个「改几个字符串」的变更，而是从「静态 UI」到「活的世界」的范式转换。

## 十三、新需求自查与技术可行性验证（v1.1 修订）

> **修订原因**：甲方确认 AIRP 场景不应被固定为三个，也不能假设 AI 只会使用前端预设场景。新系统必须支持不少于 8 个核心世界观场景，同时允许 AI 生成额外场景，并保证角色栏只显示当前在场 NPC。

### 13.1 自查结论

| 检查项 | 当前方案状态 | 结论 | 必须修订点 |
|--------|--------------|------|------------|
| 核心场景数量不少于 8 个 | 已提出 8 个核心地点 | 通过 | 需把 8 地点提升为世界书核心地点，而非仅 UI 选项 |
| 角色栏只显示在场 NPC | 文档有“在场NPC”概念，但未绑定角色栏 | 不完全通过 | SystemBar 角色栏必须读取 `场景上下文.在场NPC` |
| 不在场 NPC 不出现在角色栏 | 当前代码仍可传入完整 npcList | 不通过 | 缺席 NPC 直接不渲染，不灰显、不占位 |
| AI 可创建新场景 | 文档提出动态扩展，但 schema 草案仍需明确兜底 | 不完全通过 | 场景字段必须使用 string + 来源标记，不再使用 enum |
| 每个场景都有故事 | 有世界事件系统，但未规定“无 NPC 也有事件” | 不完全通过 | 每个核心地点必须配置入场事件池与空场景事件池 |
| 世界观贴合 | 8 地点基本贴合合欢宗叙事 | 通过 | 需要世界书为每个地点补充氛围、禁忌、常驻事件 |
| 前端仅为变量显示端/交互端 | 与项目 MVU 约束一致 | 通过 | UI 只写入待处理交互与场景意图，等待下一楼层 AI 回写变量 |

**最终判定**：现有 `灵缘流转` 方案方向正确，但 v1.0 仍偏“场景设计报告”，不足以作为开发验收 PRD。v1.1 必须升级为“开放动态场景协议”：前端显示当前变量，AI 负责叙事扩展，世界书负责约束输出格式。

### 13.2 技术可行性验证

当前代码已经具备部分落地点，但也暴露出三场景遗留约束：

| 文件 | 当前状态 | 技术判断 |
|------|----------|----------|
| `src/雌堕合欢宗/schema.ts` | `系统.当前场景`、待处理交互、NPC `当前场景` 仍是三场景 enum | 需要改为开放字符串与结构化 `场景上下文`，否则 AI 生成新场景会被 Zod 拦截或丢失 |
| `src/雌堕合欢宗/界面/components/SystemBar.vue` | 角色栏遍历 `npcList` | 可改为 `presentNpcList`；这是纯前端数据源替换，不触碰后端核心逻辑 |
| `src/雌堕合欢宗/界面/pages/HomePage.vue` | 已使用 `visibleNpcs` 与 `TransitionGroup` | 可平滑迁移为读取 `场景上下文.在场NPC`，现有入场/离场动效可复用 |
| `src/雌堕合欢宗/界面/guards.ts` | `get场景NPC列表()` 根据 NPC 的固定 `当前场景` 过滤 | 需要替换为 `get在场NPC列表(场景上下文, npcStates)`，保留攻略链排序与状态过滤 |
| `src/雌堕合欢宗/界面/composables/usePendingAction.ts` | 待处理交互携带三场景字段 | 可改为携带 `地点`、`子区域`、`在场NPC`、`故事钩子`，供下一楼层 AI 读取 |
| `src/雌堕合欢宗/界面/data/whispers.ts` | 心里话按三场景矩阵组织 | 后续应迁移为 `公开度 + 地点类型 + 事件标签 + NPC 阶段`，避免场景膨胀导致文案矩阵爆炸 |

**可行性结论**：技术上可行，且不需要重写核心 MVU 架构。主要工作是把“场景 enum”替换成“场景上下文对象”，并把角色栏、首页、交互记录统一改为读取该对象。

### 13.3 开放动态场景协议

为适配 AIRP，场景系统不再试图阻止 AI 创造新地点，而是采用“核心地点注册表 + AI 生成场景适配器”。

```typescript
type 场景来源 = '核心地点' | '世界书扩展' | 'AI生成' | '临时过渡';

type 场景上下文 = {
  地点: string;
  子区域: string;
  场景来源: 场景来源;
  时辰: TimeName;
  公开度: '公开' | '半私密' | '私密' | '禁地';
  在场NPC: NpcName[];
  NPC活动: Partial<Record<NpcName, string>>;
  氛围: string[];
  故事钩子: string[];
  特殊事件: string;
};
```

**协议规则**：

1. `在场NPC` 是角色栏、首页 NPC 卡片、可互动目标的唯一来源。
2. 不在 `在场NPC` 内的 NPC 不允许出现在角色栏，不允许被前端直接选中互动。
3. AI 生成新地点时，不拒绝、不报错；系统将其标记为 `AI生成` 或 `临时过渡`。
4. 若 AI 未给出 `在场NPC`，前端按空数组处理，同时展示该地点的 `故事钩子` 或氛围事件。
5. 若 AI 写出未知 NPC 名，前端不渲染为角色栏 NPC，只保留在叙事文本中，避免污染核心 NPC 数据。
6. 核心地点不少于 8 个；新增世界书地点只能增加，不能删减核心地点。

### 13.4 角色栏动态显示规则

角色栏从“攻略进度展示条”升级为“当前场景命魂显影”：

```typescript
const presentNpcList = computed(() => {
  const names = store.data.系统.场景上下文?.在场NPC ?? [];
  return 攻略链.filter(name => names.includes(name));
});
```

**验收标准**：

- 当前地点有白芷、纪兰时，角色栏只显示白芷、纪兰。
- 当前地点没有 NPC 时，角色栏不显示任何 NPC 太极盘。
- 柳素衣不在场时，即使她已攻略完成，也不在角色栏出现。
- NPC 离场时，太极盘执行离场淡出动画；NPC 入场时，太极盘执行灵气浮现动画。
- 前端调试添加 NPC 到 `在场NPC` 后，角色栏立即响应；移除后立即消失。

### 13.5 八大核心地点与“非空房间”事件池

每个核心地点必须拥有至少 3 类入场反馈：`有人事件`、`无人事件`、`线索事件`。玩家进入任何地点，都必须发生可叙事内容。

| 核心地点 | 世界观定位 | 有人事件示例 | 无人事件示例 | 线索事件示例 |
|----------|------------|--------------|--------------|--------------|
| 莲灯前苑 | 外门前苑、灯影社交 | 路过弟子议论某位女修近日异样 | 莲灯忽明忽暗，似被人刚刚触碰 | 池畔残留一枚带香气的发簪 |
| 醉玉小筑 | 酒宴、小酌、半私密试探 | NPC 倚栏饮酒，语气比平日松动 | 空杯尚温，桌上留着未干酒痕 | 酒盏下压着一张含糊请帖 |
| 绮梦幽阁 | 私密深阁、心防动摇 | NPC 独坐帘后，似在等待又似在躲避 | 帘幕无风自晃，香雾聚而不散 | 暗格露出一角旧日手札 |
| 药庐 | 丹药、医理、身体异变 | 白芷或弟子正在试药、分拣灵草 | 丹炉余温未散，药香过分浓郁 | 药渣中混着不该出现的灵材 |
| 经阁 | 典籍、禁书、心法线索 | 纪兰或值守弟子正在整理卷宗 | 书页自动翻到某个禁忌章节 | 夹页记载一段被抹去的宗门旧事 |
| 演武场 | 修炼、比试、胜负欲 | 苏芸或门人正在演练功法 | 擂台上残留新鲜剑痕与汗气 | 兵器架少了一柄登记在册的短刃 |
| 长老殿 | 权力、审讯、密令 | 沈月秋或执事正在低声议事 | 殿中香灰断裂，像有人匆匆离开 | 案上密令缺了最后一页 |
| 掌门殿 | 宗门核心、威压、禁令 | 柳素衣或近侍正在处理宗门密报 | 殿门半掩，威压残留令人心悸 | 玉案上浮现一行转瞬即逝的灵纹 |

**非空原则**：空场景不是“没人，所以没有内容”，而是“没人，所以留下痕迹”。这条原则必须写入世界书，避免 AI 把地点写成空房子。

### 13.6 世界书输出约束

新增场景世界书条目必须要求 AI 在每次玩家进入地点、切换时辰、完成交互后，尽量回写以下结构化变量：

```yaml
系统:
  当前地点: 莲灯前苑
  当前子区域: 莲池畔
  场景上下文:
    地点: 莲灯前苑
    子区域: 莲池畔
    场景来源: 核心地点
    公开度: 公开
    在场NPC: [白芷]
    NPC活动:
      白芷: 晒药后在池畔净手，似乎发现了什么
    氛围: [莲灯, 水汽, 低声议论]
    故事钩子:
      - 池畔残留一枚带药香的发簪
      - 远处弟子提到今夜经阁禁区异动
    特殊事件: ''
```

**AIRP 兼容规则**：

- AI 可以创造新地点，但必须给出 `场景来源: AI生成`。
- AI 不得因为前端只有 8 个核心地点就拒绝叙事扩展。
- AI 创建的新地点要尽量映射到一个核心地点类型，如“私密”、“公开”、“禁地”、“修炼”、“医药”、“典籍”、“权力”。
- 前端不负责判断剧情合理性，只负责显示变量；世界书负责约束 AI 叙事风格与变量格式。

### 13.7 待处理交互闭环

由于本项目是对话式 AIRP，前端交互不会立即得到 AI 回应。正确闭环如下：

1. 玩家在 UI 中点击“进入地点 / 选择 NPC / 使用物品 / 灵识窃取”。
2. 前端只写入 `待处理交互`，记录 `动作类型`、`地点`、`子区域`、`目标NPC`、`在场NPC`、`道具`、`故事钩子`。
3. 玩家发送下一楼层消息。
4. AI 读取 `待处理交互` 与当前 `场景上下文`，在正文中叙事回应。
5. AI 回写新的 MVU 变量，包括新的 `场景上下文.在场NPC`、NPC 状态、背包变化、灵石变化等。
6. 前端下一次渲染只显示新变量，不自行编造剧情结果。

这保证了前端 UI 能影响游戏进程，但不越权替 AI 写剧情。

### 13.8 商城解锁场景闭环

现有商城已经存在“特殊场景”分类，购买后写入 `场景.已解锁`，且不会进入背包。动态场景系统必须复用这条经济闭环，而不是另起一套解锁逻辑。

**现有技术事实**：

| 文件 | 已验证现状 | 设计含义 |
|------|------------|----------|
| `src/雌堕合欢宗/界面/data/scenes.ts` | `特殊场景` 已定义 `阴阳池`、`经阁密室`、`掌门殿偏殿`、`渊底灵脉` | 这些不是背包物品，而是可解锁地点/子区域 |
| `src/雌堕合欢宗/界面/pages/ShopPage.vue` | `activeCategory === 'scene'` 时写入 `store.data.场景.已解锁` | 商城是场景开放的正式入口 |
| `src/雌堕合欢宗/界面/pages/ShopPage.test.ts` | 已覆盖“购买特殊场景后加入场景解锁列表且不进入背包” | 该行为已有测试保护，应保留并扩展 |

**动态场景系统中的定位**：

1. 八大核心地点是世界地图骨架，不一定全部一开始可进入。
2. 商城购买的“特殊场景”应作为 `核心地点的子区域` 或 `世界书扩展地点` 接入 `地点注册表`。
3. 购买后立即写入 `场景.已解锁`，前端地点选择器解除锁定。
4. 如果玩家进入已购买场景，`场景上下文.场景来源` 应为 `世界书扩展` 或 `核心地点`，而不是 `AI生成`。
5. 已购买场景同样必须遵守“非空房间”原则，至少有 `有人事件 / 无人事件 / 线索事件`。
6. 商城购买行为仍需写入 `待处理交互`，让下一楼层 AI 在叙事中承认“玩家已取得进入资格”。

**特殊场景接入建议**：

| 商城场景 | 推荐归属 | 场景来源 | 叙事用途 |
|----------|----------|----------|----------|
| 阴阳池 | 独立核心禁地 / 长老殿下辖禁区 | 世界书扩展 | 牝奴改造、阴阳灵气、身体异变事件 |
| 经阁密室 | 经阁·密室 | 世界书扩展 | 禁书、心法残页、纪兰深层剧情 |
| 掌门殿偏殿 | 掌门殿·偏殿 | 世界书扩展 | 柳素衣私密会面、宗门密令、掌门反差剧情 |
| 渊底灵脉 | 宗门地下灵脉 / 绮梦幽阁密道后续 | 世界书扩展 | 高浓度灵气、双修效率、隐藏风险事件 |

**验收标准**：

- 未购买的特殊场景不应作为可进入地点显示，或必须显示为锁定态。
- 购买特殊场景后，`场景.已解锁` 包含该名称，背包 `道具.拥有` 不包含该名称。
- 已解锁特殊场景可被地点选择器、世界书输出与 `场景上下文` 引用。
- AI 若把已购买场景写入当前地点，前端应正常显示，不应当作异常 AI 场景处理。
- 特殊场景购买后，下一楼层 AI 需要根据 `待处理交互: 购买物品` 叙事确认入口开启、令牌生效或禁制解封。
### 13.9 风声系统（三风声推荐版 + 高级版接口预留）

> **设计目标**：让世界每回合主动向玩家泄露“下一步可探索的信息”。玩家不再盲点地图，而是根据宗门传闻、灵气异动、NPC 行踪与器物残痕决定下一次对话要追查什么。

### 13.9.1 首发范围：推荐版“三风声模型”

首发采用推荐版：每个回合最多显示 3 条风声，其中 1 条主风声、最多 2 条次级风声。

| 类型 | 数量 | 用途 | 示例 |
|------|------|------|------|
| 主风声 | 1 | 强引导，指向当前最值得追查的事件 | “药庐丹炉彻夜未熄，白芷屏退了所有杂役。” |
| 次级风声 | 0-2 | 弱引导，提供备选探索路线 | “经阁禁区有书页自行翻动。” |
| 空风声 | 0 | 不显示空盒子 | 若 AI 未回写风声，前端保持留白 |

**首发原则**：

1. 每回合最多 3 条，避免信息过载。
2. 风声只提供探索方向，不直接给剧情结果。
3. 点击风声只写入 `待处理交互`，等待下一楼层 AI 叙事回应。
4. 风声地点可以指向 8 核心地点、商城解锁地点或 AI 生成地点。
5. 风声可以没有 NPC，但必须有 `故事钩子`。

### 13.9.2 数据结构

```typescript
type 风声来源 = '弟子传闻' | '灵气异动' | '器物残痕' | 'NPC行踪' | '宗门告示' | '梦兆';
type 风声状态 = '未读' | '已读' | '已追查' | '已失效';
type 风声紧急度 = '低' | '中' | '高';

type 风声 = {
  id: string;
  来源: 风声来源;
  地点: string;
  子区域: string;
  相关NPC: NpcName[];
  紧急度: 风声紧急度;
  风声文本: string;
  故事钩子: string;
  状态: 风声状态;

  // 高级版预留字段：首发不启用逻辑，只保留结构兼容
  过期回合?: number;
  可信度?: number;
  是否陷阱?: boolean;
  反制NPC?: NpcName;
  后续风声ID?: string;
};
```

**Schema 首发建议**：

```typescript
系统: z.object({
  风声列表: z.array(风声Schema).prefault([]),
  当前追查风声ID: z.string().prefault(''),
});
```

### 13.9.3 UI 呈现规则

风声系统是“信息流”，不是“任务列表”。视觉上应像宗门暗流浮现，而不是现代任务面板。

| 状态 | UI 表现 | 交互 |
|------|---------|------|
| 无风声 | 不显示空盒子，仅保留场景氛围 | 不可点击 |
| 未读风声 | 微弱呼吸光，显示来源、地点、短句 | 点击后标为已读并写入待处理交互 |
| 已读风声 | 光效减弱，仍可追查 | 再次点击可覆盖当前追查目标 |
| 已追查风声 | 半透明淡出或移入旧闻 | 不重复作为主风声提示 |
| 已失效风声 | 不在主界面展示 | 进入图鉴 `风声卷` 作为旧闻归档，避免主界面重复推送 |

**首页建议位置**：放在场景实时画面上方或 NPC 卡片流上方，标题可用“风声入耳”“宗门风闻”“灵脉微响”，避免使用“任务”“通知”等现代词。

### 13.9.4 与待处理交互闭环

玩家点击风声时，前端只写入意图，不生成结果：

```yaml
待处理交互:
  - 类型: 追查风声
    风声ID: rumor-20260522-001
    地点: 药庐
    子区域: 炼丹房
    相关NPC: [白芷]
    故事钩子: 丹炉彻夜未熄，白芷屏退杂役
```

下一楼层 AI 必须读取该交互，并完成三件事：

1. 在正文中叙事回应玩家追查行为。
2. 回写新的 `场景上下文`，包括地点、在场 NPC、氛围、故事钩子。
3. 更新 `风声列表`，将已追查风声标为 `已追查`，并可生成后续风声。

### 13.9.5 与商城解锁场景联动

风声系统必须能引导玩家购买或探索已购买场景。

| 场景状态 | 风声表现 | 前端行为 |
|----------|----------|----------|
| 未购买特殊场景 | 风声可暗示入口存在，但不直接开放进入 | 地点选择器显示锁定，提示需解封禁制 |
| 已购买特殊场景 | 风声可直接指向该地点 | 点击可写入追查风声并切换/请求进入该地点 |
| AI 生成临时场景 | 风声标记为 `AI生成` 或由场景上下文兜底 | 前端正常显示，不阻断 |

示例：

- 未购买 `经阁密室`：风声写为“经阁深处有纸页摩擦声，却被一重禁制隔在门后。”
- 已购买 `经阁密室`：风声写为“经阁密室的灯在无人时亮起，纪兰的笔迹出现在门缝下。”

### 13.9.6 高级版预留接口

首发不实现复杂高级逻辑，但必须保留字段与接口，避免后续返工。

| 高级能力 | 预留字段 | 首发处理 | 后续扩展 |
|----------|----------|----------|----------|
| 风声过期 | `过期回合` | 读取但不主动倒计时 | 按回合自动标记 `已失效` |
| 真假情报 | `可信度`、`是否陷阱` | 默认可信，不做判定 | AI 可制造误导与反制 |
| NPC 反制 | `反制NPC` | 仅作为显示/叙事参考 | 高道心 NPC 可诱捕玩家追查 |
| 链式线索 | `后续风声ID` | 可回写但不强制关联 | 形成连续调查链 |
| 旧闻归档 | `状态` | 已追查不重复主推 | 图鉴 `风声卷` 以因果时间线展示历史线索 |

### 13.9.6.1 图鉴归档落地补充：因果秘录时间线

截至 v4 E-UI-21，`图鉴` 已从普通回看页升级为 `因果秘录` 的长期记录中心：

- `心音册`：读取 `系统.心音回响`，以时辰与场景作为归档锚点，展示玩家曾窥见的 NPC 心音残响。
- `风声卷`：读取 `系统.风声列表`，以来源与地点 / 子区域作为归档锚点，展示世界信息流与旧闻线索。
- `场景志`：负责当前所在、已解锁地点、核心地点的阅读筛选联动；点击地点只过滤图鉴内容，不触发场景跳转。
- `因果轴视觉`：心音与风声条目均使用纵向羽化因果线与圆形刻点，形成“追溯 / 归档 / 回看”的卷宗质感。
- `边界约束`：图鉴页是只读整理层，不写入 `系统.待处理交互`，不调用 AI，不改变下一楼层剧情结算。

该设计使“已追查风声 / 已捕获心音 / 已解锁地点”形成闭环：主页负责即时交互，图鉴负责长期回看，二者共享 MVU 数据但职责不重叠。
### 13.9.7 技术可行性结论

推荐版风声系统可以通过技术验证，原因如下：

1. 它只新增 `系统.风声列表` 与 `当前追查风声ID`，不破坏现有 MVU 架构。
2. UI 只读变量并写入 `待处理交互`，符合“前端是显示端/交互端，AI 是叙事端”的项目原则。
3. 风声地点使用 string，天然兼容 8 核心地点、商城解锁地点和 AI 生成地点。
4. 高级版字段采用 optional，不会影响首发校验。
5. 没有风声时前端不显示空容器，不会制造现代任务面板感。

**验收标准**：

- AI 回写 0 条风声时，首页不显示空盒子。
- AI 回写 1-3 条风声时，首页按顺序显示。
- 玩家点击风声后，`待处理交互` 写入 `类型: 追查风声` 与对应 `风声ID`。
- 风声地点为商城已解锁场景时，前端允许追查。
- 风声地点为商城未解锁场景时，前端可提示禁制未开，不直接进入。
- 高级预留字段存在时，前端不报错；不存在时也不报错。

### 13.10 道具生命周期协议（商城—背包—场景—风声闭环）

> **设计目标**：将道具从“背包里的物品数量”升级为“驱动 NPC、场景、风声与下一楼层叙事变化的触发器”。本协议复用现有商城、背包、装备、待处理交互基础，补齐道具分类、生命周期、目标选择与 AI 回写规则。

### 13.10.1 现有实现自查

| 模块 | 已实现能力 | 当前不足 |
|------|------------|----------|
| `data/items.ts` | 已区分 `装备` / `消耗品`，包含道具、丹药、特殊道具、牝奴道具 | 缺少“购买即生效 / 可卸下 / 永久效果 / 场景触发”等生命周期字段 |
| `ShopPage.vue` | 普通物品进入 `道具.拥有`，场景进入 `场景.已解锁`，剧情进入 `剧情.已解锁` | 购买后与风声、场景上下文联动不足 |
| `BackpackPage.vue` | 装备会扣库存，卸下会返还库存；消耗品使用后扣库存 | 丹药默认对玩家使用，缺少目标选择与永久效果记录 |
| `schema.ts` | 已有 `道具.拥有`、`道具.装备`、`场景.已解锁`、`剧情.已解锁` | 缺少 `道具效果` / `已生效道具` / 开放场景上下文字段 |
| `usePendingAction.ts` | 已记录购买、装备、卸下、使用物品 | 仍绑定三场景 enum，缺少地点、风声、故事钩子上下文 |
| 世界书 `道具系统.yaml` | 已列出道具、门槛、价格、效果 | 缺少“前端只记录动作，AI 下一楼层确认结果并回写变量”的强约束 |

**结论**：道具系统不是从零开始，已有可复用代码地基；本阶段需要把它升级为生命周期协议，并接入开放场景与风声系统。

### 13.10.2 道具生命周期分类

| 类别 | 购买后去向 | 使用/装备规则 | 是否可卸下 | 是否消失 | AI 回写重点 |
|------|------------|---------------|------------|----------|-------------|
| 服装类 | `道具.拥有` | 装备到 NPC 或玩家时库存 -1 | 是，卸下库存 +1 | 库存 <1 从背包消失 | 当前穿着、场景反应、NPC 态度变化 |
| 拘束/器具类 | `道具.拥有` | 装备到目标时库存 -1，受好感度/阶段门槛限制 | 是，卸下库存 +1 | 库存 <1 从背包消失 | 行动限制、互动方式、风声或旁人反应 |
| 丹药类 | `道具.拥有` | 使用时必须选择目标，库存 -1 | 否 | 使用后消失 | 目标状态、长期体质/气息/敏感变化 |
| 时间/消耗类 | `道具.拥有` | 使用后库存 -1 | 否 | 使用后消失 | 天数、时辰、状态增减 |
| 场景类 | `场景.已解锁` | 购买即解锁入口 | 不适用 | 不进背包 | 地点注册、禁制开启、风声可指向 |
| 剧情类 | `剧情.已解锁` | 购买即解锁剧情线索 | 不适用 | 不进背包 | 图鉴/世界书可引用新秘密 |
| 阵法/结局类 | 系统字段 | 购买即生效 | 不适用 | 不进背包 | `系统.已使用阵法`、结局条件 |

### 13.10.3 数据结构建议

```typescript
type 道具生命周期 = '可装备' | '可消耗' | '购买即生效' | '解锁场景' | '解锁剧情';
type 道具目标类型 = '玩家' | 'NPC' | '场景' | '全局';

type 道具定义 = {
  名称: string;
  类别: '服装' | '器具' | '丹药' | '时间' | '场景' | '剧情' | '阵法';
  生命周期: 道具生命周期;
  目标类型: 道具目标类型[];
  可卸下: boolean;
  使用后消失: boolean;
  是否永久效果: boolean;
  触发风声: boolean;
  触发场景变化: boolean;
};

type 已生效道具效果 = {
  目标: string;
  道具: string;
  来源交互ID: string;
  生效楼层: number;
  效果标签: string[];
  可被AI覆盖: boolean;
};
```

### 13.10.4 与待处理交互的闭环

道具操作不能由前端直接生成剧情结果，必须写入 `待处理交互` 等待下一楼层 AI 处理。

```yaml
待处理交互:
  - 类型: 使用物品
    目标: 白芷
    道具: 媚体丹
    数量: 1
    地点: 药庐
    子区域: 炼丹房
    在场NPC: [白芷]
    风声ID: rumor-001
    故事钩子: 丹炉彻夜未熄，药香异常
```

AI 下一楼层必须完成：

1. 正文叙事承认该道具操作。
2. 回写目标 NPC 或系统状态。
3. 如产生世界影响，更新 `场景上下文` 与 `风声列表`。
4. 如是永久效果，写入 `已生效道具效果` 或对应 NPC 标签。
5. 道具库存变化以前端操作为准，AI 不得凭空恢复已消耗库存，除非叙事明确返还。

### 13.10.5 与动态场景、风声、商城的联动

| 触发源 | 联动结果 | 示例 |
|--------|----------|------|
| 购买场景类道具 | 解锁地点，后续风声可指向该地点 | 购买 `经阁密室` 后，风声可提示密室灯火异常 |
| 使用丹药 | 改变 NPC 状态，生成后续风声 | 使用丹药后，药庐出现“药香过浓”的风声 |
| 装备服装/器具 | 改变在场 NPC 反应与可用交互 | NPC 入场时活动描述包含当前装备影响 |
| 卸下装备 | 恢复库存，生成关系或场景反应 | 卸下后 AI 可描写目标松动、防备或依赖变化 |
| 购买剧情 | 解锁线索，影响风声池 | 购买纪兰剧情后，经阁风声更容易出现禁书线索 |

### 13.10.6 技术可行性结论

道具生命周期协议可以通过技术验证，原因如下：

1. 现有 `BackpackPage.vue` 已经实现装备扣库存、卸下返还、消耗品扣库存。
2. 现有 `ShopPage.vue` 已经实现普通道具、场景、剧情、阵法的不同购买去向。
3. 现有 `schema.ts` 已有 `道具.拥有`、`道具.装备`、`场景.已解锁`、`剧情.已解锁` 的持久化结构。
4. 需要新增的是分类元数据、目标选择、已生效效果记录、开放场景上下文，不需要推翻现有代码。
5. 该协议能把商城消费转化为世界变化，是新增预算项的核心价值。

**验收标准**：

- 装备类道具装备后库存 -1，库存 <1 时从背包消失。
- 装备类道具卸下后库存 +1，并重新出现在背包。
- 同一件库存不能同时装备给多个目标。
- 丹药/消耗类使用后库存 -1，库存 <1 时从背包消失。
- 丹药使用必须支持选择目标，不再默认只能对玩家使用。
- 场景类购买进入 `场景.已解锁`，不进入背包。
- 剧情类购买进入 `剧情.已解锁`，不进入背包。
- 购买即生效类写入系统字段，不进入背包。
- 所有道具操作都写入 `待处理交互`，供下一楼层 AI 叙事确认。
- 道具可触发风声或场景上下文变化，但结果必须由 AI 回写。

### 13.11 修订后的实施优先级

| 阶段 | 优先级 | 内容 | 验收标准 |
|------|--------|------|----------|
| Phase A | P0 | Schema 增加 `场景上下文`，保留旧 `当前场景` 兼容读取 | AI 生成新地点不会被校验拦截 |
| Phase B | P0 | SystemBar 角色栏改读 `在场NPC` | 不在场 NPC 不显示 |

| Phase E | P1 | 道具生命周期协议：分类、目标、库存、效果回写 | 道具能驱动 NPC、场景、风声变化且库存准确 |
| Phase F | P1 | 世界书补充 8 核心地点、风声输出、道具回写与 AIRP 格式 | AI 能稳定回写地点、在场 NPC、故事钩子、风声列表、道具效果 |

### 13.12 v1.1 验收红线

- 低于 8 个核心地点：不通过。
- 角色栏出现不在场 NPC：不通过。
- AI 生成新场景导致变量校验失败：不通过。
- 玩家进入无 NPC 场景后只有“暂无 NPC / 此处无人”：不通过。
- 前端直接生成剧情结果而不等待下一楼层 AI 回写：不通过。
- 世界书没有约束 `在场NPC` 与 `故事钩子` 输出：不通过。

> v1.1 的核心不是“把 3 个按钮扩成 8 个按钮”，而是建立一套 AIRP 可扩展的场景协议：AI 可以生长世界，前端只显示当下，角色栏只显影在场之人。
