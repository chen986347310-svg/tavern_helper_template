- 三件新道具全部从世界书 `欲海与印记` 设定原生挖出，玩家若读过世界书会感觉「原来这就是那个机制」，不破坏沉浸。
- 购买即生效道具不写入 `道具.拥有`；除「改变阵法」保留 `系统.已使用阵法 = true` 这一旧契约外，其他购买即生效道具前端零持久状态，所有持久后果由 AI 写 JSONPatch。
- 特殊道具事件规则作为绿灯资料库挂载（顺序 13），关键字包含世界观锚点（欲海核心 / 青云道人 / 九天驭奴诀 等），跟 `欲海与印记` 世界书条目自然咬合触发。
- 完整覆盖测试 60 项（含 5 件道具的显示名、生命周期、购买行为、事件元数据写入）；旧 `usePendingAction.test.ts` 与 `BackpackPage.test.ts` 中对「改变阵法 / 时间延长」的 `toEqual` 精确匹配同步收窄为 `toMatchObject`，因为旧道具也补上了 AI 短提示。

### 3.11 待处理交互动态注入（强提示机制）

实机测试发现 AI 有时完全忽略 `系统.待处理交互` 中的购买/使用/装备操作，或擅自将灵石 replace 回初始值 2000。根本原因是世界书绿灯资料库只能被动触发，没有强制注入到 AI 的当前上下文中。本轮新增 `pendingActionPrompt.ts` 模块，复用 `outfitPrompt.ts` / `contrabandPrompt.ts` 的动态注入架构，在 `GENERATE_BEFORE_COMBINE_PROMPTS` 钩子中将待处理交互队列以 `position: 'in_chat'` 强制注入。

关键文件：

- `src/雌堕合欢宗/脚本/服装叙事注入/pendingActionPrompt.ts`：新建，Phase 1+2 注入逻辑
- `src/雌堕合欢宗/脚本/服装叙事注入/pendingActionPrompt.test.ts`：新建，16 项测试
- `src/雌堕合欢宗/脚本/后端校验/index.ts`：集成 `buildPendingActionPrompt` 到刷新/清理链路
- `dist/backend_validate.js`：已重建（~366 KB），含 `hehuan-pending-action-summary` 注入标识

注入架构（与服装/禁器同层）：

| 层级 | ID | position | 作用 |
|------|----|----------|------|
| 可见注入 | `hehuan-pending-action-summary` | `in_chat` | 按交互类型差异化叙事指令，强制 AI 在本楼层正文承接 |
| 扫描注入 | `hehuan-pending-action-scan` | `none` + `should_scan: true` | 激活对应世界书绿灯（特殊道具事件规则、道具AI承接闭环、禁器叙事规则等） |

Phase 1 — 全类型覆盖：

| 交互类型 | 注入指令重点 |
|----------|------------|
| 购买物品（普通） | 最低承认底线：正文至少一句承认道具入手，禁止"队列清空但正文一字未提" |
| 购买物品（购买即生效） | 高优先级承接：标记为事件种子，不得被开场叙事或 NPC 互动压过 |
| 使用物品 | 必须写出使用效果：身体变化、气息波动、场景反应 |
| 装备道具 | 必须写出目标反应：顺从、抗拒、羞恼或沉默 |
| 卸下 | 写出残痕退去、压力消散或目标反应 |
| 追查风声 | 承接为场景遭遇，参考地点与故事钩子 |
| 灵识窃取 | 写出窥探结果：心音、心防反震或心音残片 |

Phase 2 — AI 红线注入：

- 当 `系统.灵石 > 500000` 时注入保护指令："此为玩家调试行为，禁止用 replace 将灵石修正回初始值或任何低值，必须保留当前数值继续叙事。"

Phase 3 — 场景上下文感知：

- 框架已预留（`PendingActionPromptData` 含场景上下文字段），当前版本暂不触发具体场景切换指令；后续可根据剧情钥匙/特殊道具的推荐场景与当前场景不匹配时注入过渡提示。

重要兼容策略：

- 复用 `getItemLifecycle(name)` 判断购买即生效道具，不硬编码道具名单，避免后续增减道具时漏改。
- 清理逻辑与 `clearOutfitPrompt` / `clearContrabandPrompt` 完全对称：`GENERATION_ENDED` 和 `CHAT_CHANGED` 时统一清理，不会残留到无关楼层。
- 扫描文本包含 `待处理交互`、`最低承认底线`、`道具AI承接闭环` 等关键词，激活世界书中对应绿灯资料库，不额外蓝灯常驻。
- 前端零改动：注入完全在后端校验脚本层完成，`usePendingAction.ts` 不需要修改。

开发过程严格遵循 TDD 红→绿→重构：

1. **红灯**：删除 `pendingActionPrompt.ts`，保留测试文件，运行确认 16 项测试全部失败（`Cannot find module`）。
