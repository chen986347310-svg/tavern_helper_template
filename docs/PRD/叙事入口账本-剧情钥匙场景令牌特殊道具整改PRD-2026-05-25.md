# 叙事入口账本：剧情钥匙、场景令牌、特殊道具整改 PRD

日期：2026-05-25

## 1. 背景与问题

当前商城已经把商品分为普通装备、丹药、场景令牌、剧情钥匙和特殊道具，但“购买后如何进入叙事闭环”的稳定性不一致。

已核查现状：

- 剧情钥匙购买后会写入 `剧情.已解锁`，但 `canBuy()` 没有基于已解锁状态禁用重复购买。
- `剧情.已解锁` 使用 `addUnique`，所以列表不会重复，但重复购买仍会再次扣灵石并再次写入 `系统.待处理交互`。
- 场景令牌购买后会写入 `场景.已解锁`，同样缺少完整的重复购买与重复入队防护。
- `系统.风声列表` 已经是首页、状态栏和追查交互的活跃入口，但目前没有专门承载“剧情钥匙风声 / 场景令牌风声”的生命周期规则。
- `剧情` schema 当前只有 `已解锁`，无法记录某把剧情钥匙是否已经生成风声、是否正在追查、是否已经触发。
- `风声Schema.来源` 当前枚举为 `弟子传闻 / 灵气异动 / 器物残痕 / NPC行踪 / 宗门告示 / 梦兆`，没有 `剧情钥匙 / 场景令牌 / 特殊事件` 这类系统入口来源。
- 特殊道具已经在世界书中被定义为“事件种子”，其中 `欲海回声 / 投欲钥 / 改变阵法` 是高优先级购买即生效事件，不适合拖成风声追查链。

核心问题不是“AI 会不会记得”，而是现有变量层缺少一个可校验的叙事入口账本，导致购买、风声、追查、触发、归档之间没有稳定状态轨迹。

## 2. 目标

本 PRD 目标是建立一套轻量、可测试、可验收的“叙事入口账本”，让剧情钥匙、场景令牌和特殊道具有明确生命周期。

目标包括：

1. 防止剧情钥匙、场景令牌和一次性特殊事件重复购买、重复扣费、重复入队。
2. 让剧情钥匙购买后稳定生成一条可追查风声，而不是依赖 AI 在后续场景中“想起来”。
3. 让场景令牌购买后稳定生成一条场景入口风声，引导玩家前往或追查该场景。
4. 让特殊道具按生命周期拆分：购买即生效事件立即承接，自用消耗道具仍按库存和使用链路处理。
5. 让活跃风声不永久占位：触发后移除或标记失效，历史状态沉淀到账本或图鉴归档。
6. 保持现有 AIRP 边界：前端只做即时反馈、线索入口和待处理交互入队；真正剧情推进与持久剧情后果仍由 AI 下一楼层承接并写 JSONPatch。

## 3. 技术验证结论

### 3.1 Schema 可行性

当前 `schema.ts` 已有：

- `系统.待处理交互`
- `系统.风声列表`
- `系统.当前追查风声ID`
- `场景.已解锁`
- `剧情.已解锁`

缺口：

- 需要新增 `剧情.线索状态`，用于记录剧情钥匙、场景令牌和特殊事件入口状态。
- 需要扩展 `风声Schema.来源`，加入 `剧情钥匙`、`场景令牌`、`特殊事件`。
- 可选扩展 `系统.待处理交互` 字段，补齐 `剧情线`、`关联NPC`、`秘密主题`、`入口类型`、`线索ID` 等字段。当前 DebugPanel 已能显示部分扩展字段，但 schema 尚未包含 `剧情线 / 关联NPC / 秘密主题`。

结论：可行，属于小规模 schema 扩展；需要同步更新 schema 测试、世界书变量规则和前端 mock 数据。

### 3.2 商城链路可行性

当前商城链路：

```text
buyItem -> 扣灵石 -> applyPurchaseResult -> 记录购买物品 -> 关闭弹窗
```

生命周期已由 `getItemLifecycle()` 判断：

- `解锁剧情`
- `解锁场景`
- `购买即生效`
- `自用消耗`
- `目标消耗`
- `可装备`

结论：商城已有足够的生命周期入口，整改可以集中在 `canBuy()`、按钮文案、`applyPurchaseResult()` 和新增叙事入口 helper，不需要重写商城架构。

### 3.3 风声链路可行性

当前首页已经支持：

- 最多显示 3 条未失效风声。
- 点击风声写入 `系统.待处理交互`，类型为 `追查风声`。
- 再次点击同一风声可取消暂存。
- 不调用 `generate()`，等待下一楼层 AI 承接。

结论：风声系统已经是合适的叙事入口 UI，只需要新增“系统生成风声”的来源和生命周期规则。

### 3.4 数据源可行性

`特殊剧情` 已有：

- `名称`
- `显示名`
- `NPC`
- `剧情线`
- `秘密主题`
- `推荐场景`
- `关联道具`
- `解锁提示`
- `AI短提示`

`特殊场景` 已有：

- `名称`
- `显示名`
- `关联NPC`
- `进入理由`
- `风声钩子`
- `AI短提示`

特殊道具已有：

- `时间延长`
- `改变阵法`
- `欲海遮蔽符`
- `欲海回声`
- `投欲钥`

结论：剧情钥匙需要补 `风声种子`；场景令牌可复用现有 `风声钩子` 生成场景入口风声；特殊道具不默认生成可追查风声。

## 4. 方案概述

新增“叙事入口账本”能力，统一管理购买后进入叙事系统的入口状态。

推荐状态机：

```text
未生成 -> 可追查 -> 追查中 -> 已触发
                      -> 已失效
```

推荐账本结构：

```ts
剧情: {
  已解锁: string[];
  线索状态: Record<string, {
    类型: '剧情钥匙' | '场景令牌' | '特殊事件';
    状态: '可追查' | '追查中' | '已触发' | '已失效';
    风声ID: string;
    关联名称: string;
    关联NPC?: string;
    推荐场景?: string[];
    触发次数: number;
    最近场景?: string;
  }>;
}
```

说明：

- 账本字段挂在 `剧情` 下，是为了最小化顶层 schema 扩散。
- `场景令牌` 虽然属于场景系统，但它也是“叙事入口”，可以先统一记录在 `剧情.线索状态`。
- 后续如果入口账本扩大，再考虑独立成 `叙事.入口状态` 顶层字段。

## 5. 分类规则

### 5.1 剧情钥匙

剧情钥匙是“可追查的伏笔入口”。

购买流程：

```text
玩家购买剧情钥匙
-> 前端检查不可重复购买
-> 扣灵石
-> 写入 剧情.已解锁
-> 写入 剧情.线索状态[剧情线] = 可追查
-> 生成固定剧情钥匙风声到 系统.风声列表
-> 写入 系统.待处理交互：购买物品
-> AI 下一楼层承认信物入手或线索浮现
```

追查流程：

```text
玩家点击钥匙风声
-> 写入 系统.待处理交互：追查风声
-> 线索状态可切换为 追查中
-> AI 下一楼层推进剧情钥匙内容
-> AI 清空待处理交互
-> 风声移除或标记已失效
-> 线索状态变为 已触发
```

规则：

- 剧情钥匙购买后不立即剧透。
- 推荐场景是触发加权条件，不是唯一触发条件。
- 钥匙风声触发后不应永久留在首页活跃风声中。
- 如剧情线有多段，可在第一段 `已触发` 后生成下一条风声。

### 5.2 场景令牌

场景令牌是“永久解锁地点 + 一次性入口风声”。

购买流程：

```text
玩家购买场景令牌
-> 前端检查不可重复购买
-> 扣灵石
-> 写入 场景.已解锁
-> 写入 剧情.线索状态[场景名] = 可追查
-> 生成场景入口风声到 系统.风声列表
-> 写入 系统.待处理交互：购买物品
-> AI 下一楼层承认令牌入手或地点权限开启
```

触发流程：

```text
玩家追查场景风声 或 玩家主动前往该场景
-> AI 承接该地点入口、异常或相关 NPC 动向
-> 风声移除或标记已失效
-> 场景仍永久保留在 场景.已解锁
```

规则：

- 场景令牌购买后可以生成风声，但风声不是进入场景的唯一方式。
- 玩家直接在对话中前往已解锁场景，也应被 AI 承接。
- 场景风声触发后不继续占用活跃风声位。

### 5.3 特殊道具

特殊道具按生命周期拆分。

#### 购买即生效事件

包括：

- `改变阵法`
- `欲海回声`
- `投欲钥`

规则：

- 不走风声追查链。
- 购买后必须写入 `系统.待处理交互`。
- AI 下一楼层必须高优先级立即承接。
- 一次性事件应防重复购买和重复入队。
- 可选生成“事后余波风声”，但余波不作为二次触发入口，默认不进入活跃风声列表或进入后立即失效。

#### 自用消耗道具

包括：

- `时间延长`
- `欲海遮蔽符`

规则：

- 可重复购买，进入 `道具.拥有`。
- 使用时写入 `系统.待处理交互`。
- AI 承接使用效果。
- 不默认生成风声，避免风声噪音；必要时由 AI 生成短期余波。

## 6. 防重复与幂等规则

### 6.1 解锁剧情

不可购买条件：

- `剧情.已解锁` 已包含该剧情线。
- `系统.待处理交互` 已存在同名 `购买物品`。
- `剧情.线索状态` 中该剧情线已存在且状态不是 `已失效`。

按钮文案建议：

- `因果已入簿`
- `线索待显`
- `正在追查`

### 6.2 解锁场景

不可购买条件：

- `场景.已解锁` 已包含该场景名。
- `系统.待处理交互` 已存在同名 `购买物品`。
- `剧情.线索状态` 中该场景名已存在且状态为 `可追查` 或 `追查中`。

按钮文案建议：

- `场景已开`
- `入口已现`

### 6.3 购买即生效特殊事件

不可购买条件：

- 一次性事件已经触发。
- 一次性事件已经存在待处理交互。
- 对于 `改变阵法`，沿用当前柳素衣攻略值门槛，并在触发后禁用。

按钮文案建议：

- `事件已触发`
- `命运已定`

### 6.4 自用消耗

可重复购买，继续走库存逻辑。

## 7. 活跃风声容量与清理规则

`系统.风声列表` 最大 3 条，新增系统风声必须有稳定替换策略。

优先级：

1. `追查中` 风声不得替换。
2. 剧情钥匙风声优先保留。
3. 场景令牌风声次级保留。
4. 普通动态风声可被替换。

插入规则：

- 同 ID 已存在时不重复插入。
- 未满 3 条时直接追加。
- 已满时替换最旧的普通未读风声。
- 如果 3 条都不可替换，则不插入新风声，但账本仍记录入口状态。

触发后清理规则：

- 剧情钥匙风声：追查并触发后从活跃列表移除或标记 `已失效`。
- 场景令牌风声：追查或进入该场景后从活跃列表移除或标记 `已失效`。
- 特殊事件余波：默认不进入活跃列表；如果进入，必须短期失效。

## 8. 用户故事

1. 作为玩家，我想购买剧情钥匙后获得明确线索入口，所以我知道下一步可以追查什么。
2. 作为玩家，我不想重复购买同一剧情钥匙导致重复扣灵石，所以商城必须显示该钥匙已入簿。
3. 作为玩家，我希望剧情钥匙不是买完立即剧透，而是通过风声、心音、场景逐步展开。
4. 作为玩家，我想购买场景令牌后看到与该地点相关的风声，所以我知道新地点为什么值得去。
5. 作为玩家，我希望已经解锁的场景永久可用，而不是风声消失后场景也消失。
6. 作为玩家，我追查钥匙风声后，希望首页不再反复显示同一条已处理风声。
7. 作为玩家，我希望特殊道具中主动事件能立即被 AI 承接，而不是还要额外追查风声。
8. 作为作者，我希望 AI 不需要凭记忆判断钥匙是否触发，而是读取变量账本和待处理交互。
9. 作为作者，我希望每条剧情钥匙都有固定风声种子，避免 AI 临场生成偏题线索。
10. 作为开发者，我希望叙事入口插入、去重、清理可以用纯函数测试，避免散落在页面组件里。
11. 作为验收者，我希望调试面板能看到购买、风声、追查和清理的状态变化，便于确认闭环。

## 9. 实现决策

### 9.1 新增深模块

建议新增一个纯函数模块负责叙事入口生命周期，避免把逻辑塞进 `ShopPage.vue`。

模块职责：

- 根据商品生命周期判断是否应生成叙事入口。
- 生成剧情钥匙风声。
- 生成场景令牌风声。
- 插入风声时执行去重和容量替换。
- 写入或更新 `剧情.线索状态`。
- 判断商品是否可重复购买。

### 9.2 数据扩展

剧情钥匙数据增加 `风声种子`。

场景令牌复用 `风声钩子` 生成标准风声，不强制新增字段。

特殊道具增加或派生“是否一次性事件”信息。短期可沿用 `getItemLifecycle()` 中的 `购买即生效特殊道具` 集合。

### 9.3 Schema 扩展

必须扩展：

- `剧情.线索状态`
- `风声Schema.来源`

建议扩展：

- `系统.待处理交互[].剧情线`
- `系统.待处理交互[].关联NPC`
- `系统.待处理交互[].秘密主题`
- `系统.待处理交互[].入口类型`
- `系统.待处理交互[].线索ID`

### 9.4 世界书规则更新

需要更新：

- 剧情叙事规则：追查剧情钥匙风声后推进剧情，并更新线索状态。
- 场景叙事规则：购买场景令牌生成入口风声，追查或进入场景后清理风声。
- 特殊道具事件规则：明确购买即生效特殊道具不走追查风声链。
- 变量输出格式：AI 承接后必须清空待处理交互，并按需要更新风声状态与线索状态。

## 10. 测试决策

测试应覆盖外部行为，不测组件内部实现细节。

必须新增或更新测试：

1. `schema.ts` 测试：新字段 prefault、来源枚举、旧存档兼容。
2. 叙事入口纯函数测试：风声生成、去重、容量替换、状态更新。
3. 商城测试：剧情钥匙不可重复购买、场景令牌不可重复购买、一次性特殊事件不可重复触发、自用消耗仍可重复购买。
4. 待处理交互测试：购买剧情钥匙/场景令牌携带入口元数据。
5. 首页风声测试：系统风声可追查、追查中不可重复入队、触发后可失效。
6. 世界书资料测试：所有剧情钥匙均有风声种子，所有场景令牌均能生成入口风声，特殊事件规则保持绿灯挂载。
7. 调试面板回归测试：叙事入口相关待处理交互字段可读显示。

验收命令建议：

```bash
npx vitest run src/雌堕合欢宗/界面/data/itemLifecycle.test.ts src/雌堕合欢宗/界面/data/itemDisplay.test.ts
npx vitest run src/雌堕合欢宗/界面/pages/ShopPage.test.ts src/雌堕合欢宗/界面/pages/HomePage.test.ts
npx vitest run src/雌堕合欢宗/schema.test.ts
npm run build
```

涉及 AI/MVU 闭环时，仍需用 CDP 或人工楼层验收确认：

- 购买后队列出现。
- 下一楼层 AI 承接。
- `系统.待处理交互` 被清空。
- 风声状态或线索状态被正确更新。

## 11. 非目标

本 PRD 不包含：

- 不做完整任务系统。
- 不做多段剧情进度树。
- 不新增复杂倒计时或自动触发器。
- 不让前端直接推进剧情结果、NPC 状态或结局。
- 不让所有特殊道具都变成风声追查链。
- 不改变普通装备、丹药、服装的基础购买与背包逻辑。

## 12. 验收标准

1. 同一剧情钥匙无法重复购买、重复扣费、重复入队。
2. 同一场景令牌无法重复购买、重复扣费、重复入队。
3. 剧情钥匙购买后稳定生成可追查风声。
4. 场景令牌购买后稳定生成场景入口风声。
5. 风声列表容量为 3 时，新入口风声有确定替换策略。
6. 玩家追查钥匙风声后，AI 能推进剧情并清理待处理交互。
7. 钥匙风声触发后不再永久占用首页活跃风声位置。
8. 场景风声触发或玩家进入场景后不再长期占用活跃风声位置。
9. `欲海回声 / 投欲钥 / 改变阵法` 购买后仍作为高优先级立即事件承接，不被改造成风声追查。
10. `时间延长 / 欲海遮蔽符` 仍可作为自用消耗道具重复购买和使用。
11. 世界书、schema、前端数据源、商城测试和首页风声测试全部同步。

## 13. 推荐实施顺序

1. 扩展 schema 与测试，保证旧存档兼容。
2. 新增叙事入口纯函数模块和测试。
3. 给剧情钥匙补风声种子。
4. 改商城幂等与按钮文案。
5. 接入购买剧情钥匙和场景令牌后的风声生成。
6. 更新首页追查后的状态处理。
7. 更新世界书规则和变量输出格式。
8. 更新调试面板可读字段。
9. 跑单元测试、构建和 CDP/MVU 闭环验收。

## 14. 结论

本方案技术可行，且比单纯依赖 AI 在推荐场景中自然触发更稳。

最终设计原则：

```text
剧情钥匙 = 唯一解锁商品 + 固定风声种子 + 可追查线索状态
场景令牌 = 永久地点解锁 + 一次性入口风声
特殊道具 = 按生命周期区分立即事件与自用消耗
活跃风声 = 当前可追查入口，不是永久归档
```
