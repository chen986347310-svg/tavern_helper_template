# 灵缘流转开放动态系统技术验证与实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `subagent-driven-development` 或 `executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将动态场景、在场 NPC 角色栏、风声系统、商城解锁与道具生命周期协议落地为可测试、可回滚、可验收的 AIRP 前端闭环。

**Architecture:** 采用“MVU 变量为唯一事实源”的增量改造方式。前端只读取 `defineMvuDataStore(Schema)` / `useDataStore().data`，UI 点击只写入 `待处理交互`，剧情结果由下一楼层 AI 回写变量后再显示。旧三场景字段保留兼容读，新增开放 `场景上下文`、`风声列表` 与道具生命周期元数据。

**Tech Stack:** Vue 3、Pinia、TypeScript、Zod 4、Vitest、Webpack、SillyTavern MVU、世界书 YAML。

## 二、实施总原则

1. **先兼容，后替换**：保留旧 `当前场景` 与 NPC `当前场景` 字段，新增 `场景上下文` 后逐步切换读取源。
2. **前端不写剧情结果**：所有点击只写入 `待处理交互`，下一楼层 AI 回写变量后前端显示。
3. **每阶段都有测试**：每个 Phase 至少新增或更新一个测试文件，不靠手感验收。
4. **不做无关重构**：除非当前任务必须，否则不重命名页面、不迁移路由、不重排目录。
5. **世界书同步推进**：涉及 AI 回写的字段必须同步写进世界书，否则前端变量会断流。
6. **频繁提交**：每个 Phase 完成测试后单独提交，便于回滚与甲方验收留痕。

## 四、风险清单与兜底策略

| 风险 | 触发条件 | 兜底策略 |
|------|----------|----------|
| AI 未回写新字段 | 世界书未生效或模型忽略格式 | Schema 默认值为空数组/空对象，前端不崩溃 |
| 风声过多 | AI 回写超过 3 条 | Schema `.max(3)` 或前端 `.slice(0, 3)`，首发只显示 3 条 |
| 未知 NPC 污染角色栏 | AI 写入路人名 | `normalizePresentNpcs()` 只保留核心 NPC |
| 新场景与旧场景冲突 | 旧组件仍读 `当前场景` | 保留旧字段兼容，逐步改读 `场景上下文` |
| 丹药目标错误 | 用户未选择目标 | 默认目标为玩家，但丹药 UI 明确要求目标选择 |
| 道具库存错乱 | 同一库存装备多个目标 | 装备前检查 `道具.拥有[name] > 0`，装备后立即扣库存 |
| AI 恢复已消耗库存 | 叙事回写污染数据 | 世界书明确“库存以前端操作为准”，除非叙事明确返还 |
| 体积警告误判失败 | webpack 大包警告 | 记录为既有警告，不作为本次失败条件 |

## 五、执行建议

推荐执行顺序：

1. 先做 `Phase A-B`，建立开放数据地基与角色栏动态显影。
2. 再做 `Phase C`，让甲方可见“世界每回合有风声”。
3. 再做 `Phase D`，把道具系统接进世界变化闭环，用于追加预算验收。
4. 最后做 `Phase E-F`，世界书与总体验证收尾。

每完成一个 Phase 都运行对应测试并提交，避免多个子系统混在一起难以回滚。

## 六、自查结论

- 需求覆盖：开放场景、在场 NPC、8+ 地点、非空房间、商城解锁、风声、道具生命周期均有阶段覆盖。

## Phase E-UI-3 角色栏动态闭环实施计划（2026-05-23）

### 计划定位

E-UI-3 不再只指“灵识锁定”一个点，而是把 `docs/PRD/角色栏角色卡片设计UIux补充参考.md` 中尚未完成的角色栏表现层需求收束为一个组件级闭环：

1. **卡片入场 / 离场细节**：在场 NPC 进入角色栏时具备“命魂凝聚”感，离场时具备“墨迹消散”感。
2. **同化态高级视觉**：攻略值达到 100 或状态为 `已完成` 时，阴阳命轮进入深红流光同化态，避免只靠 class 切换产生廉价感。
3. **角色栏整体动态收拢**：`presentNpcList` 变化时，角色栏整体向中心自然收拢或展开，不出现空位跳动、横向挤压或旧 NPC 残留占位。
4. **交互边界不越权**：本阶段只做组件表现层与测试闭环，不新增剧情结算逻辑，不让前端替 AI 生成结果。

### 当前状态复核

| 子项 | 当前状态 | 缺口 | 本阶段处理方式 |
|------|----------|------|----------------|
| E-UI-1 缩小态命魂显影 | 已落地 | 仍需与整体收拢协同 | 只做回归保护，不重写 |
| E-UI-2 展开态阴阳命轮 | 已落地 | 同化态仍偏基础视觉 | 在 `NpcCard.vue` 增强同化态样式与测试 |
| 卡片入场动效 | 已有 `ink-fade` 基础 | 缺少“形体凝聚 / 命魂显影”验收标准 | 在 `SystemBar.vue` 明确 TransitionGroup class 与测试 |
| 卡片离场动效 | 已有基础 leave | 缺少绝对定位收拢策略验证 | 增加离场 class 与布局收拢测试 |
| 整体动态收拢 | 居中布局已具备雏形 | 缺少 “0 / 1 / 多 NPC” 布局行为测试 | 用组件测试锁定结构和 class，不依赖像素截图 |
| 灵识锁定红线 | 尚未完整实现 | 涉及待处理交互取消与视觉挂载 | 拆到下一小阶段 E-UI-4，避免与入离场动画混做 |

### 本阶段范围

本阶段只做三个闭环，不混入世界书、道具生命周期或 AI 反震回写：

- **闭环 A：角色栏入场 / 离场**
  - `SystemBar.vue` 继续使用 `TransitionGroup`。
  - 为 `npc-active-group` 增加稳定布局 class，确保 NPC 数量变化时居中收拢。
  - 强化 `ink-fade-enter-*` 与 `ink-fade-leave-*`：入场为 blur → clear、scale(0.95) → 1、轻微上浮；离场为 opacity → 0、scale(0.92)、filter blur。
  - 离场元素允许使用 `position: absolute`，但只限 TransitionGroup 离场阶段，避免撑开布局。

- **闭环 B：同化态高级视觉**
  - `NpcCard.vue` 中 `destiny-assimilated` 继续由 `攻略值 >= 100 || 状态 === '已完成'` 驱动。
  - 深红同化态不改变量逻辑，只增强视觉：双环深红流光、慢速 pulse、外环专属色被红光吞没的过渡。
  - 保留“无数字显示”原则，玩家只看状态语、命轮与色彩变化。

- **闭环 C：组件级测试护栏**
  - `SystemBar.test.ts` 增加：在场 NPC 数量变化时只渲染当前列表；容器包含收拢布局 class；TransitionGroup name 为 `ink-fade`。
  - `NpcCard.test.ts` 已覆盖同化态 class；本阶段可追加样式结构断言，不测试 CSS 动画细节。
  - 不做截图测试，不引入浏览器端视觉测试依赖。

### 明确不做

- 不在本阶段新增 `soul_whisper` 反震字段。
- 不在本阶段实现“点击命盘再次取消灵识窥探”。
- 不在本阶段播放音效或震动反馈。
- 不在本阶段引入粒子系统、canvas 或第三方动画库。
- 不改 MVU 数据结构，除非测试发现现有字段无法表达已落地功能。

## Phase E-Fix 实机变量回写诊断记录（2026-05-23）

### 实机现象

- 第 10 层与第 12 层中，AI 已输出 `<UpdateVariable><JSONPatch>`，但 CDP 读取到的原始 MVU 仍只保留旧 `系统` 字段。
- 第 12 层 AI 已按新要求使用 `insert` 补齐 v4 字段，但仍输出了两个 `<UpdateVariable>` 块，且第一块尝试写入 `系统.当前场景 = 外门广场`。
- `系统.当前场景` 已修正为开放字符串字段；三场景只作为前端快捷入口与兼容回退。`外门广场` 等 AI 场景允许写入 `系统.当前场景`，同时必须同步写入 `系统.场景上下文`。

### 根因判断

1. 历史楼层缺少 v4 字段时，对 `/系统/场景上下文` 使用 `replace` 不可靠，必须使用 `insert` 自愈。

3. 重复输出两个 `<UpdateVariable>` 块会增加解析混乱风险，应把自愈、场景推进、数值调整、清空队列全部合并到同一个 JSONPatch 数组。

### 已执行修正

- `initvar.yaml` 已补齐 v4 字段，保证新开局路径存在。
- `变量输出格式.yaml` 已加入 `MVU v4字段自愈硬约束` 与 `MVU开放场景路径硬约束`。
- `动态场景系统.yaml` 已加入开放场景字段规则。
- `变量更新规则.yaml` 已加入 `系统开放场景红线`，明确 `当前场景` 为开放字符串，三场景只是快捷入口。

### 下一轮验收口径

新楼层应满足：

- 只输出一个 `<UpdateVariable>`。
- 若缺少 v4 字段，使用 `insert` 补齐。
- `系统.当前场景` 可以是 AI 创建的开放场景名，例如 `外门广场`、`药庐`、`经阁`。
- AI 创建场景时必须同步写入 `系统.场景上下文`，其中包含在场 NPC、氛围与故事钩子。
- CDP 读取新楼层 MVU 时，`系统` 应包含 `时辰 / 当前场景 / 场景上下文 / 风声列表 / 当前追查风声ID / 待处理交互`。


### 口径修正（开放场景）

用户确认项目已经从三场景系统升级为开放动态场景系统。此前将 `系统.当前场景` 收紧为三选一是错误方向，现已修正：

## Phase E-UI-10 被动灵识常照与心音回响系统方案（2026-05-23）

### 背景与问题

当前命环点击逻辑本质是“玩家主动发起一次灵识窃取”，前端写入 `系统.待处理交互`，下一楼层由 AI 判定并回写 `NPC.*.soul_whisper`。该模型已经能避免同一 NPC 重复入队：同目标再次点击会取消，切换目标会清空旧灵识并锁定新目标。但实机体验仍存在三个风险：

1. **离场不可见**：如果玩家点击命环后，下一楼层 AI 让该 NPC 离开当前场景，则角色卡不再渲染，心声即使回写也缺少展示位置。
2. **点击语义过重**：命环点击被理解为“发起窥心判定”，容易让玩家以为每次点击都应该立刻产生剧情结果。
3. **状态闭环不稳**：心声显示绑在 `NpcCard` 展开区域，缺少跨场景、跨楼层的“已捕获心音记录”。

### 新产品口径

将“主动窥心”升级为：**合欢心典·被动灵识常照**。

- 玩家默认拥有被动灵识感知能力；每个回合都会扫过当前场景的在场命魂。
- AI 每楼层可根据在场 NPC、公开度、好感、攻略值、剧情压力，生成“心声波动 / 反震 / 锁闭 / 无波动”。
- 命盘不再只是发起请求的按钮，而是持续显示 NPC 当前心防状态的“命魂显影器”。
- 玩家点击命盘的语义改为“聚焦 / 翻阅 / 锁定这段心音回响”，而不是前端自行判定成功或失败。
- 真正推动剧情的后续行为仍通过 `系统.待处理交互` 明确提交，例如“追查这段心声”“以心声试探她”。

### 推荐数据模型

| 字段 | 类型建议 | 作用 | 是否 P0 |
| :--- | :--- | :--- | :--- |
| `系统.心音回响` | array | 存放最近捕获/反震/锁闭的心音结果，NPC 离场后仍可展示 | 是 |
| `系统.当前聚焦心声NPC` | string | 玩家当前点击聚焦的 NPC；只影响 UI 展示，不直接推进剧情 | 是 |
| `NPC.*.心声探测态` | enum | `无波动 / 可窥探 / 已捕获 / 反震 / 锁闭`，驱动命盘视觉 | 是 |
| `NPC.*.soul_whisper_candidates` | array | AI 预备的心声候选池；默认不直接展示 | P1 |
| `系统.待处理交互[].心音ID` | string | 当玩家追查某条心音时，明确交互来源 | P1 |

### 心音回响对象建议

```ts
interface SoulEcho {
  id: string;
  npc: '白芷' | '苏芸' | '纪兰' | '沈月秋' | '柳素衣';
  text: string;
  stage: '警戒' | '动摇' | '沉沦';
  result: '捕获' | '反震' | '锁闭';
  scene: string;
  time: '晨时' | '午时' | '酉时' | '亥时';
  floor?: number;
  is_new: boolean;
}
```

### 命盘视觉规则

| 探测态 | 命盘表现 | 点击行为 |
| :--- | :--- | :--- |
| `无波动` | 慢速呼吸，无古字残片 | 可展开卡片，但不显示心音 |
| `可窥探` | 灵气潮汐、断环、古字碎片闪烁 | 点击聚焦该 NPC，显示“心音将显未显”的预备态 |
| `已捕获` | 命盘出现淡金/桃粉光晕，心音回响池出现新条目 | 点击展开/定位心音回响 |
| `反震` | 命盘碎裂、红线逆冲、显示 `心防反震` | 点击查看反震记录，不再重复触发 |
| `锁闭` | 命盘灰败沉寂，本场景不诱导点击 | 点击只显示锁闭提示，不写入待处理交互 |

### 显示层设计

1. **角色在场时**：心声优先显示在 `NpcCard.vue` 展开区，与命盘、灵犀/蚀心语义同层。
2. **角色离场时**：心声转入 HomePage 的“心音回响”区域，作为跨场景残留信息展示，不依赖角色卡是否存在。
3. **无 NPC 场景**：HomePage 可显示最近一条心音回响，与环境留白叙事并列，避免玩家误以为点击白费。
4. **重复点击**：同一 NPC 的命盘点击只切换聚焦/取消聚焦；不得重复向 `待处理交互` 堆叠 `灵识窃取`。
5. **提交后冻结**：当用户发送文本等待下一楼层时，聚焦状态进入只读，避免同一楼层内多次改写因果意图。

### AIRP 边界

- 前端不得自行生成心声正文。
- 前端不得自行随机决定捕获/反震/锁闭结果。
- 前端可以显示 AI 已回写的 `心音回响`，并允许玩家聚焦、折叠、追查。
- 前端只有在玩家选择“追查这段心音”时，才写入 `系统.待处理交互`。
- AI 每楼层应读取当前场景、在场 NPC、心音回响、当前聚焦 NPC 与待处理交互，决定下一次心音状态。

### 技术可行性验证计划

1. **schema 验证**：确认新增 `系统.心音回响`、`系统.当前聚焦心声NPC`、`NPC.*.心声探测态` 不破坏旧数据 prefault。
2. **组件验证**：`NpcCard.vue` 新增探测态视觉映射测试；`HomePage.vue` 新增“离场后仍显示心音回响”测试。
3. **交互验证**：命盘点击只改变聚焦状态，不重复写入 `灵识窃取`；追查心音才写入 `待处理交互`。
4. **世界书验证**：变量输出规则要求 AI 每楼层可更新心音探测态，并把捕获/反震/锁闭写入 `系统.心音回响`。
5. **实机验证**：构造 NPC 在场、点击命盘、下一楼层 NPC 离场三步，确认心音仍在回响区可读。

### 分期建议

| 阶段 | 内容 | 目标 |
| :--- | :--- | :--- |
| E-UI-10A | PRD 与 schema 技术验证 | 锁定字段与 AIRP 边界 |
| E-UI-10B | HomePage 心音回响区 | 解决 NPC 离场不可见 |
| E-UI-10C | NpcCard 命盘探测态 | 让命盘显示被动灵识状态 |
| E-UI-10D | 世界书与变量规则同步 | 让 AI 稳定回写心音结果 |
| E-UI-10E | 调试面板补充心音注入器 | 快速实机验收捕获/反震/锁闭 |

### 当前结论

采用“被动灵识常照 + 心音回响池”方案。它比“每回合自动点击心声”更准确：系统默认感知心防波动，但不污染 `系统.待处理交互`；玩家点击命盘只是聚焦和查看，真正影响剧情的追查行为仍由明确的待处理交互提交。该方案能解决 NPC 离场导致心声不可见的问题，并让命盘成为持续表达世界信息的核心 UI。

## Phase E-UI-19 修复记录：图鉴页按钮文字黑色兼容兜底

  - 第一行使用稳定的 `#hex 硬兜底`、`var(--hh-text-primary / --hh-accent, #hex)`、`color-mix()` 渐进增强。
  - 第二行保留高级 `color-mix()`，在支持的浏览器中继续呈现细腻混色。
- 覆盖范围：
  - `.filter-chip`
  - `.filter-chip:hover / .filter-chip.active`
  - `.scene-group-title`
  - `.scene-archive-item`
  - `.scene-archive-item.active .scene-name`
  - `.scene-name`
  - `.scene-meta`
- 清理 PowerShell 写入过程中误产生的字面量 `` `r`n ``，确保 SCSS 语法干净。

## Phase E-UI-24 修复记录：禁地公开度氛围遮罩去整页滤镜（2026-05-24）

### 问题现象

- 实机切换 `系统.场景上下文.公开度 = 禁地` 后，主页整体颜色出现异常偏移。

- 原 `pulse-forbidden` 动画继续改变 `filter`，使颜色异常随动画持续波动，放大实机观感问题。

### 修复方案

- 移除禁地公开度的整页 `filter` 与 `pulse-forbidden` 动画。
- 将禁地氛围改为 `.scene-context-container::before` 背景遮罩，通过暗紫、幽绿、胭脂暗雾表达压迫感。
- 使用 `isolation: isolate` 与 `z-index: -1` 限定遮罩层，只影响背景氛围，不影响子组件文字、命轮、风声和心音颜色。
- 新增 `forbidden-veil` 只做 `opacity / scale` 呼吸，不再做 `hue-rotate`。

### 架构边界

- 不改 MVU 字段、不改 `公开度` 枚举、不改场景/风声/心音逻辑。
- 不影响 `系统.场景上下文.在场NPC` 的角色栏显影规则。
- 本修复属于纯视觉层回归修复，禁止后续重新在场景根容器上使用整页 `filter: hue-rotate(...)`。

### 技术验证

## Phase Shop-服装系统问题记录：好感度阶层、玩家专属牝奴服与 AI 可读装备闭环（2026-05-24）

### 1. 新发现问题

- 现有服装名称偏直白或偏后台 SKU，缺少从“保守 → 暧昧 → 暴露 → 仪式化 → 命契化”的吸引力递进。
- 当前 `名称` 同时承担逻辑 key 与玩家展示名，若直接改名会影响库存、装备、门槛判断、旧存档与测试。
- `牝奴服装列表` 当前混在普通服装数据中，但业务语义应为“玩家牝奴期专属服装”，不应作为普通 NPC 服装长期展示。
- 商城服装列表目前主要按系列分类，缺少按好感度/阶段分层的楼层式陈列，无法体现攻略进度带来的商品解锁感。
- 服装详情仅展示基础描述，缺少仪式阶位、暴露阶位、适用对象、AI 叙事提示、装备后可见状态等关键信息。
- 装备后的服装虽然写入 `道具.装备[target]`，但 AI 是否稳定读取、角色离场再入场时 UI 是否持续显示，仍需作为闭环规则明确。

### 2. 牝奴服业务边界

- `牝奴素衣 / 牝奴纹衣 / 牝奴链甲 / 牝奴礼服` 应定义为玩家专属服装。
- 仅当 `系统.阶段 = 牝奴期` 时，商城才显示牝奴服装入口或牝奴服装楼层。
- 牝奴服不参与 NPC 好感度门槛，不允许装备给白芷、苏芸、纪兰、沈月秋、柳素衣。
- 背包装备目标选择中，牝奴服只能选择 `玩家`，避免和普通 NPC 装备逻辑混淆。

### 3. 服装商城重排方向

- 服装商城应按门槛形成楼层式结构：
  - `凡衣层`：好感度 0，保守身份服、基础宗门服。
  - `微露层`：好感度 30，轻纱、半遮、场景制服。
  - `诱形层`：好感度 50，显轮廓、束腰、铃链、开背。
  - `缚心层`：好感度 70，符纹、锁链、仪式束缚、高阶法衣。
  - `命契层`：好感度 90，终局华服、掌门/专属/剧情象征服。
  - `牝奴层`：玩家处于牝奴期时显示，只服务玩家自身阶段变化。
- 价格应随楼层递增：0 层 500-800，30 层 800-1500，50 层 1500-2500，70 层 2500-4200，90 层 4000-8000，牝奴/剧情唯一服可单独定价。

### 4. AI 可读装备闭环

- 前端装备行为仍只负责写入 `道具.装备[target]` 与 `系统.待处理交互`，不直接生成剧情结果。
- 世界书与变量规则必须要求 AI 每回合读取：`道具.装备.玩家` 与当前在场 NPC 的 `道具.装备[NPC名]`。
- AI 描写角色入场时，应自然参考其已装备服装，不要求每次机械复述，但关键服装需要影响神态、动作、旁人反应与场景钩子。
- NPC 离场后，其装备状态仍保留在 MVU 中；再次进入 `场景上下文.在场NPC` 后，前端角色卡与 AI 均应能继续读取同一套穿着。
- 若装备卸下，库存返还，AI 下一楼层读取到待处理交互后应承接“衣装已卸/灵器收回”的叙事，不再继续按已装备服装描写。
