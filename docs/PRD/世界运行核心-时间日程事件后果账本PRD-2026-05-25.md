# 世界运行核心：时间、日程与事件后果账本 PRD

日期：2026-05-25

## 1. 背景与问题

当前世界系统已经具备动态场景、待处理交互、风声追查、叙事入口账本、道具生命周期和五名核心 NPC 的人物资料。玩家的前端操作可以进入 `系统.待处理交互`，AI 下一楼层也被要求叙事承接并清空队列。

但世界真正“运行起来”的核心还不完整：时间、NPC 日程、欲海侦测、事件后果仍然分散在多个规则里，没有形成统一闭环。

已核查现状：

- `系统.时辰` 已存在，取值为 `晨时 / 午时 / 酉时 / 亥时`。
- `系统.剩余天数` 已存在，用于攻略期倒计时和牝奴期触发。
- `变量更新规则.yaml` 已写明“每次交互推进时辰”“亥时后过夜减少剩余天数”，但没有强制 AI 每楼层输出结构化时间结算。
- `时间延长` 已是自用消耗品，但使用后主要依赖 AI 自觉增加 `系统.剩余天数`。
- `欲海遮蔽符` 已是自用消耗品，设定为遮蔽九天驭奴诀气息，但没有结构化字段记录遮蔽剩余时间。
- `欲海与印记.yaml` 已定义欲海核心会扫描 NSFW 异常气息，但变量层没有 `欲海状态`、`搜寻进度`、`警戒等级`。
- `pendingActionPrompt.ts` 已能把待处理交互以 `in_chat` 强制注入 AI 上下文，是当前最可靠的强提示通道；但它目前只强制叙事承接和清空队列，没有强制时间结算。
- 世界书有宗门日常，但没有结构化 `NPC日程`；AI 可以写 NPC 出现，却不能稳定知道“谁在这个时辰应该在哪里”。
- 已生效道具、剧情钥匙、场景令牌和特殊事件能留下部分状态，但世界缺少统一的“事件后果账本”，导致重要事件发生后，知情者、余波、风声后续和 NPC 记忆仍偏依赖 AI 临场发挥。

核心问题不是缺少设定，而是缺少一个“世界运行核心”：让所有事件都落在时间河里，让 NPC 根据日程流动，让欲海根据行为侦测，让事件后果被世界记住。

## 2. 为什么要这样做

### 2.1 时间是所有玩法的共同底座

当前玩法都依赖时间：

- 攻略期倒计时依赖 `系统.剩余天数`。
- 普通对话、扫地、追查风声、装备、丹药、NSFW 都应消耗不同时间。
- 亥时后过夜会减少剩余天数。
- NSFW 会让欲海核心更快定位玩家。
- 使用欲海遮蔽符时，NSFW 不应增加欲海搜寻进度。
- 使用时间延长时，应增加可攻略天数。
- 投欲钥应主动触发欲海定位并进入牝奴期。
- 改变阵法应改写或熄灭欲海压力，进入 GOOD END 序幕。

如果没有统一时间系统，AI 只能按剧情感觉处理这些变化，结果会出现：同样一场 NSFW 有时过一刻钟、有时过一天；遮蔽符有时生效、有时被忘记；时间延长有时只是背包少了一个道具；追查风声有时不消耗任何时间。

### 2.2 日程让 NPC 不再像等待触发的卡片

现在 NPC 资料很丰满，但系统层没有“今天谁在哪里”的结构化数据。世界可以响应玩家点击，却缺少自发流动。

日程系统可以让：

- 白芷在晨时洒扫、午时修行、酉时避开苏芸。
- 苏芸在药庐暖阁试药，偶尔去渊底灵脉。
- 纪兰按时登记双修册，掌握许多事件痕迹。
- 沈月秋根据资源调配出现在阴阳池或经阁。
- 柳素衣通常不轻易出现，但在欲海警戒上升后主动介入。

这样世界不是“玩家点谁谁出现”，而是“玩家在一个时辰进入某地，遇见本来就该在那里的人”。

### 2.3 事件后果账本让世界记住玩家造成的变化

玩家购买道具、使用丹药、触发剧情钥匙、追查风声、进行 NSFW、激活欲海回声，这些都不应只是当前楼层的描写。它们应该在世界里留下痕迹。

事件后果账本可以记录：

- 事件发生时间、地点、涉及 NPC。
- 事件公开度：公开、半私密、私密、禁地。
- 知情者：谁看见、谁怀疑、谁被牵连。
- 后果标签：体香、失控、欲海波动、命契、羞耻外显、宗门记录。
- 是否已生成余波风声。
- 是否已被 NPC 心音或日程系统承接。

这样 AI 后续可以读取历史，写出自然反应，而不是每次重新发明因果。

## 3. 这样做有什么好处

1. **可靠性提升**：AI 不再只被要求“承接队列”，还必须结算时间、欲海和后果。
2. **世界连续性提升**：事件发生后有账本，不会因为换楼层而被遗忘。
3. **NPC 行为更自然**：日程让 NPC 的出现有理由，不再完全依赖剧情临场安排。
4. **欲海压力更清晰**：NSFW 不直接粗暴扣天数，而是先增加搜寻进度，进度到阈值后触发警戒或减少剩余天数。
5. **道具价值更明确**：欲海遮蔽符、时间延长、欲海回声、投欲钥、改变阵法都能接入同一套状态机。
6. **调试更容易**：调试面板可展示最近耗时、结算原因、欲海搜寻进度、遮蔽剩余时段和事件后果队列。
7. **可测试可验收**：纯函数模块可以测试时间推进、跨天、遮蔽豁免、欲海进度、事件记录生成。
8. **避免 AI 自由发挥过度**：半固定耗时表给 AI 留叙事空间，但关键数值由规则约束。

## 4. 有没有必要这样做

结论：有必要，而且应作为下一阶段世界系统地基。

理由：

- 项目的核心压力来自“30天内攻略，否则欲海定位进入牝奴期”。如果时间不可靠，主玩法就不可靠。
- 当前世界已经有丰富内容，下一步瓶颈不是再堆设定，而是让设定按照时间和因果运转。
- 待处理交互闭环已经建立，现在最自然的升级就是“处理交互时必须同步结算时间”。
- 叙事入口账本已经解决“线索入口”的生命周期，本 PRD 解决“线索发生后，世界如何流逝并记住”的生命周期。
- 牝奴期不是普通失败页，而是雌堕主题的第二主玩法。如果没有时间、日程和后果账本，P2 只能停留在堕落度展示和零散调教描写，无法成为真正可持续游玩的路线。
- 如果不做，后续任何日程、风声、经济、欲海、牝奴期都容易继续分散修补，维护成本会上升。

不建议做成大型模拟器。我们不需要小时级真实日历、全 NPC 自主 AI 或复杂行动规划。当前阶段最合适的是轻量、可测试、强提示的世界运行核心。

## 5. 目标

本 PRD 目标是建立一套轻量、强制、可测试的世界运行核心，统一管理时间流逝、NPC 日程、欲海侦测和事件后果。

目标包括：

1. 每次 AI 承接 `系统.待处理交互` 时，必须判断事件耗时。
2. 每次承接后必须写入 `系统.时间状态.最近结算`，说明耗时、原因和是否过夜。
3. 需要推进时辰时，稳定更新 `系统.时辰`；跨过亥时后减少 `系统.剩余天数`。
4. NSFW 或亲密事件必须评估欲海侦测影响。
5. 欲海遮蔽符生效期间，NSFW 不增加或极低增加欲海搜寻进度。
6. 时间延长必须增加剩余天数，并留下最近时间结算原因。
7. 投欲钥必须直接将欲海定位状态推满，并进入牝奴期路线。
8. 改变阵法必须改写或关闭欲海压力，并进入 GOOD END 序幕。
9. NPC 日程应能根据时辰、场景和欲海警戒提供合理在场建议。
10. 重大事件应写入事件后果账本，供风声、心音、NPC 反应和后续剧情读取。
11. 调试面板应展示时间状态、欲海状态和近期事件记录，方便验收。
12. 牝奴期应接入同一套世界运行核心，成为“世界围绕玩家牝奴身份运转”的第二主玩法，而不是只在结局后展示静态状态。
13. 牝奴期必须具备入场记录、日课安排、当前支配者、牝印命令、调教事件结算和后果账本，支撑长期可玩性。
14. P1 与 P2 共用时间、日程、动态注入和事件记录模块；区别在于 P1 的核心压力是欲海搜寻与攻略倒计时，P2 的核心压力是牝印支配、日课轮转、改造推进和宗门公共占有。

## 6. 技术验证结论

### 6.1 Schema 可行性

当前 `schema.ts` 已有以下基础字段：

- `系统.时辰`
- `系统.剩余天数`
- `系统.阶段`
- `系统.待处理交互`
- `系统.场景上下文`
- `系统.风声列表`
- `道具.已生效效果`
- `剧情.线索状态`
- `牝奴.堕落度`
- `牝奴.牝阴决层数`
- `牝奴.上次支配者`
- `牝奴.支配次数`
- `牝奴.情欲控制阶段`
- `牝奴.改造进度`

缺口是缺少结构化时间、欲海字段，以及能让牝奴期持续运转的 P2 运行字段。建议新增：

```yaml
系统:
  时间状态:
    当前日: number
    时段进度: number
    最近耗时: string
    最近结算原因: string
    最近事件类型: string
    是否过夜: boolean
  欲海状态:
    搜寻进度: number
    警戒等级: 平静/微动/警觉/锁定/已改写
    遮蔽剩余时段: number
    遮蔽来源: string
    最近暴露原因: string
    已被定位: boolean
  牝奴期状态:
    入场日: number
    当前日课: string
    当前支配者: string
    当前命令: string
    命令强度: number
    今日调教次数: number
    待执行日课: string[]
    最近调教结算: string
```

结论：可行。时间与欲海属于小规模 schema 扩展；P2 字段可优先放入现有 `牝奴` 对象，避免新增顶层结构。当前 `牝奴` 已有堕落度、牝阴决层数、支配次数、情欲控制阶段和改造进度，说明数据边界已经存在，只需要从“展示型状态”升级为“运行型状态”。

### 6.2 动态注入强制性可行

当前 `pendingActionPrompt.ts` 已经能在存在待处理交互时生成 `position: in_chat` 的强提示。它比普通绿灯世界书更可靠，因为它会直接进入 AI 当前上下文。

因此时间系统的强制性不应只写在世界书中，而应加到动态注入提示里：

```text
本楼层处理待处理交互时，必须同步进行时间结算：判断耗时、更新时辰、必要时减少剩余天数、写入时间状态；若涉及亲密或NSFW，必须更新欲海状态；若遮蔽符生效，本次NSFW不得增加搜寻进度。禁止只清空队列不写时间结算。
```

结论：强制路径成立。只要扩展注入提示和测试，就能把“时间结算”绑定到“待处理交互清空”上。

### 6.3 纯函数模块可测试

建议新增纯函数模块，封装世界时间和欲海结算。它不依赖 Vue，不直接写 store，只接收当前状态和事件摘要，返回下一状态建议。

推荐接口：

```ts
estimateEventCost(action, context) -> TimeCost
advanceTime(currentTimeState, cost) -> NextTimeState
settleDesireSea(currentDesireSeaState, event, modifiers) -> NextDesireSeaState
createWorldEventRecord(event, result) -> WorldEventRecord
```

结论：可测试性强。可以用 Vitest 覆盖跨时辰、过夜、遮蔽、NSFW 搜寻进度、时间延长、投欲钥、改变阵法等关键行为。

### 6.4 前端可视化可行

当前 DebugPanel 已是悬浮可拖动小窗，并能显示 `系统.待处理交互`。它适合继续扩展：

- 显示当前日、时辰、最近耗时、最近结算原因。
- 显示欲海搜寻进度、警戒等级、遮蔽剩余时段。
- 显示最近事件后果记录。
- 支持调试修改这些字段。

结论：可行。属于 DebugPanel 小范围扩展，不需要改动主 UI。

### 6.5 P2 牝奴期运行核心可行

当前代码已经具备 P2 的第一层承载能力：

- `schema.ts` 已定义 `系统.阶段 = 攻略期 | 牝奴期`。
- `guards.ts` 已有 `shouldEnterPhase2`、`initializePhase2`、`canEquip牝奴道具`、`canUpgrade情欲控制` 等 P2 相关纯函数。
- `Phase2Page.vue` 已展示堕落度、牝阴决层数、改造进度和玩家装备。
- `牝奴系统.yaml` 已有堕落度分阶段、NPC 支配风格、深度改造、牝阴决和情欲控制规则。
- `DebugPanel.vue` 已能切换阶段、调整堕落度、牝阴决层数和三项改造进度。

现有缺口也很明确：

- P2 没有结构化日课，AI 不知道今天玩家作为牝奴必须经历什么。
- P2 没有当前支配者字段，`牝奴.上次支配者` 只能记录上一轮，不能表达正在发生的支配关系。
- P2 没有命令队列或牝印命令状态，无法让“女修一句命令驱动玩家行动”进入待处理交互闭环。
- P2 没有调教事件账本，堕落度、牝阴决、改造进度与具体事件之间缺少可回看的因果。
- `牝奴系统.yaml` 是绿灯资料库，适合承载设定，但不适合独自承担强制执行；强制执行仍应通过 schema、动态注入和变量更新规则落地。

结论：P2 运行核心技术上可行，而且适合与时间系统合并实施。第一版不需要直接做完整 C 方案大路线，只需要交付 B 方案的稳定运行底座：入场记录、日课、当前支配者、牝印命令、事件结算、DebugPanel 可视化。这样既不透支范围，又为后续 C 方案“牝奴期作为第二主玩法”留下升级空间。

### 6.6 P2 凝视与羞名系统可行

当前风声系统已经具备复用基础：

- `系统.风声列表` 已在 `schema.ts`、`initvar.yaml` 和变量世界书中存在。
- `HomePage.vue` 已能展示风声流，并把点击写入 `系统.待处理交互` 的 `追查风声`。
- `SystemBar.vue` 已有风声涟漪提示，但当前只在 `攻略期` 读取 `rumorList`。
- `GalleryPage.vue` 已把风声作为可回看的 `风声卷`。
- 商城剧情钥匙、场景令牌已经能通过 `narrativeEntry.ts` 生成入口风声。

因此 P2 不需要重造一套“舆论系统”。更稳的方式是复用 `系统.风声列表`，扩展来源、字段语义和 UI 呈现：P1 风声是“线索系统”，P2 风声升级为“凝视、羞名、传唤和公开调教压力”。

建议新增或放宽风声来源：

```yaml
来源:
  - 牝奴日课
  - 牝印命令
  - 调教余波
  - 宗门闲谈
  - 公开示众
  - 支配者传唤
```

建议在不破坏旧风声结构的前提下，为 P2 羞名风声增加可选字段：

```yaml
风声:
  凝视来源: 弟子低语/执事记录/NPC目光/掌门注视/牝印回响
  羞名等级: 微闻/传开/挂牌/示众/烙名
  羞名标签: string[]
  反噬日课: string
  是否可承接: boolean
```

结论：可行。第一版可以只扩展 schema 枚举和可选字段，让 P2 事件后果账本生成 1~3 条羞名风声；UI 先在 Phase2Page 展示“听风羞名”低语流，不必做复杂舆论模拟。这样能快速增强凝视感和羞耻感，同时沿用已有风声队列、归档和动态注入闭环。

## 7. 方案概述

建立“世界运行核心”，由四个账本共同组成：

```text
时间状态 -> 记录世界如何流逝
欲海状态 -> 记录玩家功法如何被侦测
NPC日程 -> 决定每个时辰谁在哪里
事件后果 -> 记录已发生事件及其余波
P2牝奴期状态 -> 记录玩家被谁支配、今日日课、牝印命令和调教后果
```

AI 每楼层处理逻辑：

```text
读取当前变量
-> 若有待处理交互，逐条叙事承接
-> 根据交互类型估算耗时
-> 更新 时辰 / 剩余天数 / 时间状态
-> 若涉及亲密或NSFW，更新欲海状态
-> 若事件重要，写入事件后果账本
-> 根据日程和事件结果更新场景上下文或风声
-> 清空已处理的待处理交互
```

当 `系统.阶段 = 牝奴期` 时，处理逻辑切换为 P2 运行模式：

```text
读取当前变量
-> 读取当前日、时辰、牝奴期状态和牝奴系统资料库
-> 根据日课/场景/NPC日程决定当前支配者或候选支配者
-> 若有待处理交互，优先承接玩家主动行为
-> 若无待处理交互，可由日课或牝印命令生成自然压迫事件
-> 结算耗时、堕落度、牝阴决、情欲控制、支配次数和改造推进
-> 写入事件后果账本，必要时生成风声、心音或后续日课
-> 更新牝奴期状态与场景上下文
```

这里的关键是：P2 不是“时间系统结束后的失败状态”，而是同一套世界运行核心在另一种阶段下的运行方式。P1 追求攻略和逃离欲海定位；P2 追求在被宗门占有、调教和改造中持续推进雌堕可玩性。

## 8. 时间结算规则

采用半固定表型：每类事件有基础耗时，AI 可在范围内微调，但不能完全自由发挥。

| 事件类型 | 基础耗时 | 时辰推进 | 欲海影响 |
|---|---:|---|---|
| 普通对话 | 一刻 | 通常不推进 | 无 |
| 日常杂务/扫地 | 半个时辰到一个时辰 | 推进 | 无 |
| 购买物品 | 片刻 | 通常不推进 | 无 |
| 装备/卸下 | 片刻到一刻 | 通常不推进 | 无 |
| 使用时间延长 | 片刻 | 不推进 | 增加剩余天数 |
| 使用欲海遮蔽符 | 片刻 | 不推进 | 增加遮蔽剩余时段 |
| 追查风声 | 一个时辰 | 推进 | 视内容 |
| 亲密接触 | 半个时辰到一个时辰 | 推进 | 轻微增加 |
| NSFW | 一个时辰起 | 推进 | 增加搜寻进度 |
| 深度NSFW/多人/禁地事件 | 两个时辰以上 | 推进或过夜 | 大幅增加 |
| 遮蔽期NSFW | 正常耗时 | 推进 | 不增加或极低增加 |
| 投欲钥 | 命运事件 | 可强制推进 | 直接定位 |
| 改变阵法 | 终局事件 | 可强制推进 | 欲海改写 |

过夜规则：

- 一天仍保持四个时辰：晨时 -> 午时 -> 酉时 -> 亥时。
- 若事件耗时推进超过亥时，则进入下一日晨时。
- 每次过夜减少 `系统.剩余天数` 1。
- `系统.剩余天数` 到 0 且未改变阵法时，进入牝奴期触发链。

## 9. 欲海侦测规则

欲海不建议直接“每次 NSFW 扣一天”。更稳的方式是搜寻进度：

```text
NSFW事件 -> 欲海搜寻进度增加
搜寻进度达到阈值 -> 触发警戒风声 / 减少剩余天数 / 推进柳素衣介入
遮蔽符生效 -> 本次NSFW不增加或极低增加搜寻进度
投欲钥 -> 搜寻进度推满，已被定位=true
改变阵法 -> 警戒等级=已改写，欲海压力关闭
```

建议数值：

- 亲密接触：+5~10
- 普通 NSFW：+15~25
- 深度 NSFW：+25~40
- 多人、禁地、阴阳池、渊底灵脉：额外 +10~20
- 遮蔽期：+0 或 +1~3
- 搜寻进度 100 时触发定位或强警戒

警戒等级：

- `平静`：0~24
- `微动`：25~49
- `警觉`：50~79
- `锁定`：80~100 或投欲钥触发
- `已改写`：改变阵法后

## 10. NPC 日程规则

新增轻量日程，不做复杂模拟器。

建议结构：

```yaml
世界:
  日程状态:
    白芷:
      当前活动: string
      候选地点: string[]
      下个时辰倾向: string
      可偶遇: boolean
    苏芸: ...
    纪兰: ...
    沈月秋: ...
    柳素衣: ...
```

日程使用方式：

- AI 创建新场景时优先参考当前时辰和日程。
- 风声生成可参考 NPC 行踪。
- 玩家追查风声时，日程提供合理在场 NPC。
- 欲海警戒上升时，可覆盖普通日程，让柳素衣或纪兰介入。

第一版不要求前端主界面展示完整日程，只在 DebugPanel 展示即可。

## 11. 事件后果账本

建议新增：

```yaml
世界:
  事件记录:
    - id: string
      类型: string
      时辰: string
      当前日: number
      地点: string
      公开度: string
      涉及NPC: string[]
      知情者: string[]
      后果标签: string[]
      摘要: string
      是否已生成风声: boolean
      是否已结算: boolean
```

记录哪些事件：

- 剧情钥匙追查完成。
- 场景令牌入口风声被追查或进入。
- 使用长期丹药。
- 欲海遮蔽符使用。
- 欲海回声触发。
- 投欲钥触发。
- 改变阵法触发。
- 明显公开羞耻、禁地失控、多人事件。
- NPC 攻略值达到 100。

事件记录用途：

- 生成后续风声。
- 支撑 NPC 心音回响。
- 决定 NPC 日程变化。
- 让调试面板看见世界因果。

## 12. P2 牝奴期运行核心

P2 的定位应从“失败结局后的状态展示”调整为“第二主玩法运行模式”。玩家进入牝奴期后，不再以攻略 NPC 为唯一目标，而是成为宗门情欲秩序的中心：NPC、日课、牝印命令、服装道具、场景、风声和改造事件都围绕玩家的牝奴身份运转。

第一版采用 B 方案作为交付目标，C 方案作为架构方向：

- **B 方案：P2 稳定运行底座**。实现入场记录、日课、当前支配者、牝印命令、调教事件结算、动态注入和 DebugPanel 可视化。
- **C 方案：P2 第二主路线**。在 B 的底座上扩展长期路线，例如专属日课链、NPC 支配路线、牝阴决技能、公共占有事件、牝奴身份晋阶和隐藏深层结局。

### 12.1 P2 核心循环

```text
当前日/时辰
-> 读取牝奴期日课
-> 确定当前支配者或候选支配者
-> 生成牝印命令/调教事件/服侍任务
-> 玩家选择顺从、拖延、主动索求或利用道具改变局面
-> AI 结算时间、堕落度、牝阴决、支配次数、改造进度
-> 写入事件后果账本
-> 影响下一次日课、NPC 态度、风声和心音
```

P2 仍然使用 `系统.待处理交互` 作为前端到 AI 的执行通道，但队列类型可以逐步扩展为 `执行日课`、`回应牝印命令`、`请求支配`、`服侍NPC`、`接受改造`、`使用牝奴道具`。第一版可以不立刻扩展 UI 按钮，只先让 schema、世界书和动态注入承认这些类型，为后续界面增加按钮留出接口。

### 12.2 P2 日课模型

牝奴期日课不是普通 NPC 日程，而是“宗门如何使用玩家”的运行表。

| 时辰 | 日课方向 | 典型支配者 | 结算重点 |
|---|---|---|---|
| 晨时 | 验身、服装检查、牝印唤醒 | 纪兰/白芷 | 当前命令、羞耻外显、日课开始 |
| 午时 | 服侍、试药、公共差遣 | 苏芸/沈月秋 | 堕落度、支配次数、道具效果 |
| 酉时 | 调教、采补、场景事件 | 白芷/苏芸/纪兰 | 牝阴决、情欲控制、事件记录 |
| 亥时 | 寝役、命令复盘、深度改造 | 柳素衣/当前支配者 | 改造进度、过夜、次日日课 |

日课只提供默认压力，不强制每楼层都自动推进。若玩家主动购买、使用物品、追查风声或进入特殊场景，仍优先处理玩家操作；日课作为背景秩序影响 NPC 出场和后果结算。

### 12.3 P2 状态字段

建议在现有 `牝奴` 对象下新增运行字段，避免创建过大的新顶层：

```yaml
牝奴:
  入场日: number
  当前日课: string
  当前支配者: string
  当前命令: string
  命令强度: number
  今日调教次数: number
  待执行日课: string[]
  最近调教结算: string
  调教记录:
    - id: string
      当前日: number
      时辰: string
      支配者: string
      类型: string
      场景: string
      命令: string
      结果: string
      后果标签: string[]
```

`当前日课` 表示本时辰默认压迫事件，`当前支配者` 表示正在拥有当前处置权的 NPC，`当前命令` 表示牝印正在执行或刚刚下达的命令，`命令强度` 影响顺从难度和堕落度变化，`今日调教次数` 防止一天内无限叠加调教事件，`调教记录` 最多保留最近 10 条；重大事件再同步写入 `剧情.事件记录`。

### 12.4 P2 与事件后果账本联动

P2 所有关键事件都应留下可读痕迹：第一次被牝印命令控制、第一次在公开场景完成日课、当前支配者变更、情欲控制阶段升级、牝阴决层数提升、泌乳/肛门/憋尿改造完成、牝奴专属道具被使用或强制装备、柳素衣介入或重新分配玩家归属。

这些事件不只增加数值，也要影响后续世界：纪兰会记录，白芷会犹豫或照顾，苏芸会安排药物，沈月秋会把它纳入宗门秩序，柳素衣会根据欲海和牝印状态重新安排玩家。

### 12.5 P2 与服装、道具、场景、剧情钥匙联动

服装、特殊道具、场景和剧情钥匙可以共用本 PRD 的机制：

- **服装**：P1 中是装扮与禁器反应；P2 中升级为日课要求和身份标记。例如晨时验身强制检查牝奴装，公开场景服装会影响羞耻和风声。
- **道具**：P1 中多为攻略和控制工具；P2 中成为牝印命令、支配者偏好和改造进度的触发器。例如牝铃、牝链、牝环可写入当前命令或调教记录。
- **场景**：P1 中是探索入口；P2 中变为日课场所和公共占有地点。例如阴阳池用于改造，经阁用于登记，药庐用于试药，女修宿舍用于寝役。
- **剧情钥匙**：P1 中是秘密入口；P2 中可以转化为“支配者专属路线”的钥匙。例如白芷旧誓线在 P2 中变成温柔占有路线，柳素衣命契线变成牝印归属路线。

因此场景和特殊道具也能使用同样方法：先入队，动态注入强制承接，再由时间结算、日课、事件账本和风声系统固定后果。不同的是 P2 的后果不再只问“是否推进攻略”，而是问“这个事件如何改变玩家作为牝奴被世界使用的方式”。

### 12.6 P2 强制性边界

P2 不能只靠 `牝奴系统.yaml` 触发。强制性应由三层共同保证：`schema.ts` 和 `initvar.yaml` 提供稳定字段，`pendingActionPrompt.ts` 在 P2 阶段注入短指令，`变量更新规则.yaml` 明确要求 P2 事件必须同时更新时间状态、牝奴期状态和事件记录。

这能保证 P2 不只是文案气氛，而是被变量、提示和测试一起固定下来。

### 12.7 凝视与羞名系统

P2 需要继承 P1 的风声系统，但语义要升级。P1 风声主要是“世界给玩家线索”；P2 风声应变成“世界正在看见玩家、议论玩家、记录玩家，并把这些议论反过来变成调教压力”。

核心目标不是让文字更露骨，而是让玩家感到：自己已经成为宗门公共秩序中的可观看对象。每次日课、牝印命令、失态、服装变化、调教后果，都可能被弟子低语、执事记录、NPC 目光、掌门注视或牝印回响捕捉，形成新的羞名风声。

#### 12.7.1 凝视来源

P2 羞名风声应标记“谁在看”。第一版建议五类：

| 凝视来源 | 玩法含义 | 典型后果 |
|---|---|---|
| 弟子低语 | 传播最快、压力较轻 | 低强度羞名、地点流言 |
| 执事记录 | 制度化羞耻 | 修改日课、触发验身或登记 |
| NPC目光 | 与当前支配者/候选支配者绑定 | NPC 传唤、专属支配事件 |
| 掌门注视 | 柳素衣级高压凝视 | 牝印命令、归属重分配 |
| 牝印回响 | 身体状态自行暴露 | 命令强度变化、公开失态 |

#### 12.7.2 羞名等级

羞名等级用于表达风声传播深度，不需要在主界面暴露数值，但 UI 可以用视觉强度表现。

| 羞名等级 | 含义 | 处理建议 |
|---|---|---|
| 微闻 | 有人低声提起 | 可忽略，或轻微影响下一次描写 |
| 传开 | 多个地点出现相同低语 | 可生成后续风声 |
| 挂牌 | 进入执事殿/经阁记录 | 影响日课或传唤 |
| 示众 | 公开场景可触发事件 | 可承接为公开日课 |
| 烙名 | 成为长期身份标签 | 进入牝奴长期状态或调教记录 |

#### 12.7.3 羞名标签

羞名标签是 P2 的记忆点，替代单纯数值增长。每次 P2 重大事件可留下 1~3 个短标签，例如：

- `铃声失序`
- `验身迟疑`
- `牝印发热`
- `药香外泄`
- `目光回避`
- `寝役留名`
- `示众未毕`
- `名册朱批`
- `体香难遮`
- `命令迟滞`

这些标签可出现在羞名风声、调教记录、NPC 反应和 DebugPanel 中，让玩家感觉自己不是只涨了堕落度，而是被世界贴上越来越多可回响的羞名。

#### 12.7.4 风声反噬

P2 风声可以反过来追玩家，而不是永远等玩家主动追查。

规则建议：

- 玩家不处理某条 `微闻`，它可以在 1~2 个时辰后升级为 `传开`。
- `传开` 风声若与公开场景、铃声、体香、牝印发光相关，可升级为 `挂牌` 或 `示众`。
- `挂牌` 风声可改变当日或次日 `当前日课`，例如验身、登记、复查、传唤。
- `示众` 风声被点击承接后，不是追查秘密，而是进入对应公开日课或支配事件。
- `烙名` 不应频繁生成，只由重大调教事件、柳素衣介入、改造完成或长期身份变化触发。

最小闭环：

```text
P2事件发生
-> 写入牝奴.调教记录
-> 根据公开度/地点/支配者/后果标签生成羞名风声
-> Phase2Page 展示听风羞名
-> 玩家点击其中一条
-> 写入 系统.待处理交互: 追查风声 或 回应羞名
-> AI 下一楼层承接为传唤/日课/公开场景
-> 更新羞名等级、当前日课、当前支配者和事件后果账本
```

#### 12.7.5 UI 表现

P2 首页建议新增三个视觉区域：

1. **当前凝视**：显示当前主要凝视来源、公开度和凝视强度。例如执事记录、NPC目光、掌门注视。
2. **听风羞名**：显示 1~3 条羞名风声。它们应像低语流，不像任务列表；可点击承接。
3. **烙名痕迹**：显示最近羞名标签，像朱批、铃痕、牝印残光，不做现代徽章感。

UI 文案示例：

```text
◎ 弟子低语 · 听风廊 · 微闻
她走过时铃声乱了一拍，像是牝印先替她认了错。

◉ 执事记录 · 经阁 · 挂牌
今日晨课缺一项验身记录，纪兰已在名册旁留了朱批。

◆ 掌门注视 · 偏殿 · 烙名
柳素衣没有问她为何失态，只让人把她的名字移到今晚的寝役册上。
```

第一版 UI 不需要复杂动画，重点是层级准确：P2 第一眼应同时看到牝印核心、当前日课/支配者、听风羞名。这样玩家会自然感到“自己正被宗门看着”。

## 13. User Stories

1. 作为玩家，我想知道行动会消耗多少时间，所以我能规划攻略节奏。
2. 作为玩家，我想扫地、追查、NSFW 消耗不同时间，所以世界不会机械推进。
3. 作为玩家，我想在使用时间延长后确实增加可攻略天数，所以购买行为有明确价值。
4. 作为玩家，我想使用欲海遮蔽符后在一段时间内避开欲海侦测，所以我能主动安排高风险行动。
5. 作为玩家，我想 NSFW 会增加欲海压力，所以主线倒计时和玩法行为有真实联系。
6. 作为玩家，我想遮蔽期 NSFW 不缩短搜寻时间，所以遮蔽符不是纯文案道具。
7. 作为玩家，我想追查风声会消耗时辰，所以探索选择有重量。
8. 作为玩家，我想 NPC 根据时辰出现在合理地点，所以偶遇更自然。
9. 作为玩家，我想世界记住我使用过的丹药、触发过的事件和被谁看见，所以后续反应更连续。
10. 作为作者，我想事件后果被结构化记录，所以 AI 后续能稳定承接因果。
11. 作为作者，我想欲海状态有搜寻进度和警戒等级，所以主线压力可调试、可验收。
12. 作为作者，我想时间结算被动态注入强制，所以 AI 不会只清空队列却不处理时间。
13. 作为开发者，我想时间和欲海结算是纯函数，所以可以用 Vitest 覆盖关键规则。
14. 作为开发者，我想 DebugPanel 显示最近时间结算和欲海状态，所以实机问题能快速定位。
15. 作为测试者，我想验收脚本能检查队列清空和时间状态更新，所以不会把未结算楼层误判为通过。
16. 作为进入牝奴期的玩家，我想世界围绕我的牝奴身份继续运转，所以 P2 不是游戏结束，而是更深的可玩路线。
17. 作为牝奴期玩家，我想每天有日课、支配者和牝印命令，所以我能感到宗门在持续使用和改造我。
18. 作为作者，我想 P2 调教事件写入账本，所以堕落度、牝阴决、改造进度和 NPC 反应都有因果来源。
19. 作为开发者，我想 P2 也走待处理交互和动态注入，所以 P1/P2 不需要维护两套执行机制。
20. 作为测试者，我想 DebugPanel 能显示 P2 当前日课、支配者、命令和调教记录，所以牝奴期问题可以直观看见。
21. 作为牝奴期玩家，我想看到宗门低语和羞名流传，所以我能感到自己正在被世界凝视和议论。
22. 作为牝奴期玩家，我想某些羞名风声会反过来触发传唤或日课，所以风声不是装饰，而是调教压力的一部分。
23. 作为作者，我想 P2 事件能生成羞名标签，所以每次调教后果都有可回响的记忆点。
24. 作为开发者，我想复用 `系统.风声列表` 承载 P2 羞名风声，所以 P1/P2 不需要两套风声数据结构。
25. 作为测试者，我想验证羞名风声的生成、升级和承接，所以 P2 凝视系统不会只停留在文案。

## 14. 实施方案

### 14.1 第一阶段：结构化变量

新增：

- `系统.时间状态`
- `系统.欲海状态`
- `世界.日程状态` 或 `系统.日程状态`
- `世界.事件记录` 或 `剧情.事件记录`

技术建议：

- 若希望少改顶层结构，可先放入 `系统.时间状态 / 系统.欲海状态`，事件记录放入 `剧情.事件记录`。
- 若希望语义更清晰，可新增顶层 `世界`，但这会扩大 schema、initvar、测试 mock 修改面。

推荐第一版保守方案：

- `系统.时间状态`
- `系统.欲海状态`
- `剧情.事件记录`
- `NPC.${NPC名}.当前场景` 继续作为轻量日程辅助，不新增完整顶层 `世界`。

### 14.2 第二阶段：纯函数结算模块

新增世界时间核心模块，职责：

- 根据待处理交互估算事件耗时。
- 根据耗时推进时辰和当前日。
- 判断是否过夜和是否减少剩余天数。
- 根据 NSFW/亲密/遮蔽状态更新欲海状态。
- 生成事件后果记录建议。

第一版不需要前端自动应用所有结算。它主要服务：

- 测试规则。
- 生成动态注入提示文本。
- 后续实机验收脚本。

### 14.3 第三阶段：动态注入强制提示

扩展 `pendingActionPrompt.ts`：

- 有待处理交互时，提示必须包含“时间结算”。
- 使用 `时间延长` 时，提示必须要求增加剩余天数。
- 使用 `欲海遮蔽符` 时，提示必须要求增加遮蔽剩余时段。
- 涉及 NSFW/亲密时，提示必须要求更新欲海状态。
- 处理后必须清空队列，并禁止只清队列不写时间结算。

### 14.4 第四阶段：世界书同步

更新：

- `变量列表.yaml`
- `变量更新规则.yaml`
- `变量输出格式.yaml`
- `动态场景系统.yaml`
- `欲海与印记.yaml`
- `特殊道具事件规则.yaml`

重点是把时间系统定义成最高优先级规则：所有行动先落时间，再落后果。

### 14.5 第五阶段：世界书瘦身与 Token 预算

时间系统和事件后果账本会增加规则密度，如果直接把全部内容写入蓝灯常驻世界书，会显著增加 token 压力。因此本 PRD 要求同步做世界书精简。

核心原则：

```text
蓝灯只放硬约束和最小字段说明。
绿灯承载长设定、示例和资料库。
动态注入只注入当前楼层必须执行的短指令。
长表格和长叙事示例不常驻。
```

建议分层：

1. **蓝灯核心规则层**
   - 保留变量字段、JSONPatch 红线、待处理交互清空红线、时间结算红线。
   - 每条规则尽量一行表达，避免长篇解释。
   - 不放 NPC 长设定、道具整表、场景整表和完整示例。

2. **绿灯资料库层**
   - `欲海与印记`、`特殊道具事件规则`、`剧情叙事规则`、`场景叙事规则`、`丹药叙事规则`继续做绿灯资料库。
   - 新增或拆分 `世界时间规则`、`NPC日程资料库`、`事件后果规则` 时，默认绿灯，不默认蓝灯常驻。
   - 触发词围绕真实业务词：`时间结算 / 欲海遮蔽 / 过夜 / 搜寻进度 / 日程 / 事件记录 / 余波风声`。

3. **动态注入执行层**
   - `pendingActionPrompt.ts` 只注入当前队列相关的短摘要。
   - 例如队列中没有 NSFW、欲海遮蔽、时间延长，就不注入完整欲海规则。
   - 队列中出现 `时间延长`，只注入“增加剩余天数 + 写时间状态 + 清队列”。
   - 队列中出现 `欲海遮蔽符`，只注入“增加遮蔽剩余时段 + 后续 NSFW 豁免”。
   - 队列中出现 `追查风声`，只注入“消耗一个时辰 + 更新场景上下文 + 记录事件后果”。

4. **压缩摘要层**
   - 对长世界书条目增加 `短摘要` 字段，用于动态注入或蓝灯引用。
   - 长正文保留给绿灯触发，不进入常驻规则。
   - 示例：

```yaml
世界时间规则:
  短摘要: 待处理交互必须结算耗时；跨过亥时则过夜并减少剩余天数；NSFW按欲海状态结算搜寻进度。
  详细规则: ...长表格和示例...
```

5. **重复规则合并**
   - 当前 `动态场景系统.yaml`、`变量更新规则.yaml`、`变量输出格式.yaml`、`道具系统.yaml` 中存在部分重复的待处理交互规则。
   - 后续实施时应保留一个蓝灯主口径，其余文件只做引用或短摘要，避免同一规则多处长篇重复。
   - 最高优先级规则建议放在 `变量更新规则.yaml`；输出样例放在 `变量输出格式.yaml`；资料解释放在绿灯资料库。

Token 验收建议：

- 蓝灯常驻新增内容控制在 600~1000 中文字以内。
- 动态注入单条待处理交互提示控制在 500 字符以内；多条队列只列必要字段。
- 时间系统不把完整耗时表每楼层注入，只注入当前交互命中的规则。
- 世界书资料测试应检查新增世界时间资料库为绿灯，而不是误设为蓝灯。
- 禁止为了“更保险”把完整 PRD 内容搬进世界书。

这样做的好处：

- 保持 AI 强制执行时间结算，同时不把 token 消耗压到每楼层。
- 世界书从“长篇常驻说明”变成“短规则常驻 + 长资料按需触发”。
- 后续新增日程、事件、欲海细节时，可以扩展资料库，不挤爆系统提示。
- 调试更清楚：蓝灯负责硬约束，绿灯负责叙事风味，动态注入负责当前任务。

### 14.7 第七阶段：P2 牝奴期运行底座

在时间、欲海、日程和事件后果账本完成最小闭环后，把 P2 接入同一套世界运行核心。

第一版实施内容：

- 扩展 `initvar.yaml` 和 `变量列表.yaml`：保证 AI 能稳定写入 P2 运行字段。
- 扩展 `变量更新规则.yaml`：当 `系统.阶段=牝奴期` 时，AI 处理事件必须同时结算时间、日课、牝奴期状态和调教后果。
- 扩展 `pendingActionPrompt.ts`：若当前阶段为牝奴期，待处理交互提示必须包含 P2 结算要求；若交互类型与牝奴道具、日课、支配者、改造相关，提示必须要求写入调教记录。
- 扩展 `牝奴系统.yaml`：保留绿灯资料库定位，新增短摘要，避免把长篇 P2 规则放入蓝灯常驻。

P2 第一版不做完整 C 方案路线树，也不做自动调教模拟器。它只交付稳定运行底座，让后续所有牝奴期玩法都能长在这套字段和结算规则上。

第一版不做全宗门舆论模拟，不做复杂传播图。只要求形成最小闭环：P2 事件生成羞名风声，羞名风声可展示、可升级、可承接，并能反过来影响日课或支配事件。

## 17. 风险与缓解

### 风险 1：规则太重，AI 输出负担变大

缓解：第一版只要求最近一次时间结算，不要求写完整时间日志。事件记录只记录重大事件。

### 风险 2：AI 仍然忘记时间结算

缓解：把时间结算要求加入 `pendingActionPrompt.ts` 的 `in_chat` 强提示，并写测试防止被删。

### 风险 3：欲海进度数值太机械

缓解：前端只调试显示，不一定在主界面直接暴露数值。正文用“欲海微动、潮声更近、阵眼发热”等世界观表达。

### 风险 4：事件记录无限增长

缓解：限制最近 20 条，旧事件可由 AI 摘要进剧情状态，不无限保留。

### 风险 5：日程系统过度复杂

缓解：第一版只做候选地点和当前活动，不做 NPC 自主行动模拟器。

### 风险 6：P2 范围膨胀成完整新游戏

缓解：第一版只做 B 方案运行底座，不做完整 C 方案路线树。P2 必须先具备字段、日课、支配者、命令、调教记录和动态注入闭环，再扩展专属路线、技能和结局。

### 风险 7：P2 文案很强但变量不落地

缓解：P2 事件必须写入 `牝奴` 运行字段和事件后果账本；动态注入测试必须覆盖“日课 / 当前支配者 / 调教后果 / 时间结算”，避免只靠绿灯世界书发挥。

### 风险 8：羞名风声刷屏，压过主玩法

缓解：第一版 `系统.风声列表` 仍保留最多 3 条活跃风声，P2 羞名风声只展示 1~3 条。只有公开度较高、后果标签明确或支配者/日课相关的事件才生成羞名风声。普通每楼层描写不自动生成风声。

### 风险 9：羞名系统变成现代任务列表

缓解：UI 命名使用“听风羞名 / 当前凝视 / 烙名痕迹”，不使用任务、成就、通知等现代词。点击羞名风声的含义是“被风声牵引到传唤、日课或公开凝视”，不是接任务。

### 风险 10：羞名等级过度机械

缓解：等级主要服务后台结算和视觉强度，正文表达仍使用世界观语言。主界面可以显示“微闻/传开/挂牌/示众/烙名”，但不显示数值进度条，不把羞耻感做成现代数值榜。

## 18. Out of Scope

本 PRD 不做以下内容：

- 不做真实分钟级日历系统。
- 不做全宗门 NPC 大规模模拟。
- 不做主界面复杂日程表。
- 不自动调用 generate。
- 不让前端直接生成剧情。
- 不把欲海搜寻进度做成玩家必须看到的现代进度条。
- 不在第一版实现完整结局路线系统，但为 GOOD END / BAD END 留出字段联动。
- 不把事件记录做成永久无限归档。
- 不把完整时间规则、NPC日程表、事件后果示例全部放入蓝灯常驻世界书。
- 不用长篇重复规则换取“看起来更保险”的效果；强制性主要依赖结构化变量、动态注入和测试验收。
- 不在第一版实现完整 P2 第二主路线树；本 PRD 只要求 P2 运行底座可用。
- 不让 P2 自动刷出大量不可控调教事件；日课只是默认压力，玩家操作和待处理交互仍优先。
- 不把 P2 调教记录无限归档；近期记录保留在 `牝奴.调教记录`，重大事件再进入事件后果账本。
- 不做全宗门舆论传播模拟，不计算每个弟子的认知状态。
- 不把羞名风声做成现代任务系统或成就系统。
- 不让每一次普通描写都生成羞名风声；只对公开度、后果标签、支配者传唤或日课变化明确的事件生成。

## 19. 推荐实施顺序

1. 写时间/欲海 schema 与默认值测试。
2. 新增世界时间纯函数测试，先红灯。
3. 实现最小 `worldTime.ts`，让测试绿灯。
4. 扩展 `pendingActionPrompt.ts`，强制时间结算提示。
5. 做世界书瘦身设计：蓝灯只保留短规则，长表格拆到绿灯资料库。
6. 更新变量世界书和特殊道具规则。
7. 扩展 DebugPanel 显示。
8. 增加事件记录基础结构。
9. 扩展 P2 牝奴期运行字段和最小日课结算。
10. 扩展 P2 动态注入、牝奴系统短摘要、Phase2Page 和 DebugPanel。
11. 扩展 P2 凝视与羞名系统：风声来源、羞名标签、Phase2Page 低语流、SystemBar 涟漪、风声卷归档。
12. 跑全量测试和构建。
13. 做一次 CDP 实机验收：时间延长、遮蔽符、追查风声、NSFW 欲海进度、P2 日课/支配者/调教记录、P2 羞名风声承接。

## 20. 最终判断

这是必要的地基升级。

叙事入口账本解决了“玩家买了钥匙，世界如何给入口”。本 PRD 解决的是“玩家做了任何事，世界如何流逝、如何侦测、如何记住”。

整合 P2 后，它还解决了另一个核心问题：玩家进入牝奴期后，世界不应停在失败结局，而应切换成以牝奴身份为中心的第二运行模式。P1 是攻略、躲避欲海和争取 GOOD END；P2 是被宗门占有、被日课驱动、被牝印命令和改造事件持续推进的雌堕核心路线。

做完后，世界系统会从：

```text
玩家点击 -> AI承接 -> 清空队列
```

升级为：

```text
玩家点击 -> AI承接 -> 时间结算 -> 欲海侦测 -> 事件留痕 -> 日程/风声/心音后续承接 -> 清空队列
```

这会显著提升世界的连续性、可靠性和可调试性，也能让后续玩法都长在同一根时间轴上。

加入凝视与羞名系统后，P2 还会进一步从“玩家被调教”升级为“玩家被世界看见并反复命名”：调教事件会留下羞名，羞名会进入风声，风声会引来传唤、围观、登记或新的日课。这个闭环会让牝奴期的羞耻感不只来自当下文本，而来自整个宗门持续的凝视。

## 附录：P2牝奴期主控台落地口径

### 数据闭环

- P2 UI 只读 `store.data`，不生成剧情正文。
- 玩家点击羞名风声、日课异动、支配者传唤时，只写入 `系统.待处理交互`。
- AI 必须在下一楼层正文承接，并在同次 JSONPatch 中清队列，同时更新 P2 时间、日课、命令、支配者、调教记录或后果账本字段。
- `Phase2Page.vue` 是 P2 主控台主体，`SystemBar.vue` 是 P2 常驻状态反馈；二者都不得绕过 MVU 或直接调用 AI。

### UI 验收

- P2 第一屏必须出现牝印核心、日课、支配者、羞名风声、烙名标签。
- 旧的堕落度、牝阴决、改造进度、拘束法器不可丢失，但降为次级信息。
- 玩家可见文案避免现代 UI 词：任务、成就、通知、领取奖励。
- P2 状态栏必须继承 P1 的透明阵列、铭文字体、圆环进度和弱风声涟漪骨架，只把语义转为牝印、日课、命令和羞名风声，不能像突然切换到另一套独立皮肤。

### 状态栏融合口径

- `SystemBar.vue` 在攻略期显示太极命魂、时辰、剩余日数、灵石和攻略期风声涟漪。
- `SystemBar.vue` 在牝奴期继续使用同一条状态栏布局，左侧圆环保留原有樱花/堕落进度视觉，并叠加低透明度牝印暗纹。
- P2 右侧数值区使用既有 `stat-item/stat-glyph/stat-value` 结构显示时辰、牝阴决、当前日课和当前命令。
- P2 羞名风声只触发顶部弱涟漪，不在状态栏重复显示风声正文；具体追查入口仍放在 `Phase2Page.vue` 的听风羞名区域。
