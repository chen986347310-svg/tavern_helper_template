## 第一部分：环境准备与基础配置

### 1.1 Windows 启用符号链接策略

编写模板使用了符号链接文件（将一个文件内容映射到其他位置），但 Windows 默认不允许非管理员创建符号链接。

**操作步骤：**
1. `Win+R` 输入 `gpedit.msc` 打开本地组策略编辑器
2. 依次展开：计算机配置 → Windows 设置 → 安全设置 → 本地策略 → 用户权限分配
3. 双击「创建符号链接」，添加你的 Windows 用户名

### 1.2 安装 Git

- 下载地址：https://git-scm.com/downloads/win
- 安装：双击安装包一路 Next

### 1.3 安装 NodeJS

- 下载地址：https://nodejs.org/zh-cn/download
- 要求版本 ≥ 24
- 安装后验证：`node -v`

### 1.4 安装 pnpm

以管理员身份打开 PowerShell，依次执行：

```powershell
set-executionpolicy remotesigned                          # 改变执行策略
npm config set registry https://registry.npmmirror.com/   # 配置 npm 国内镜像
npm install -g pnpm                                        # 安装 pnpm
pnpm config set registry https://registry.npmmirror.com/  # 配置 pnpm 国内镜像
```

### 1.5 安装和配置 Cursor/VSCode

**推荐使用 Cursor**（基于 VSCode，内置 AI 编程助手）：

1. 下载：https://cursor.com/cn/downloads
2. **以管理员身份运行**（以后也要如此）
3. 注册登录 → 勾选数据共享 → 选择 AI 回复语言为中文
4. 设置侧边栏：`Ctrl+Shift+P` → `Open VSCode Settings` → 搜索 `orientation` → 改为 `vertical`
5. **导入配置文件**：下载教程提供的 `sillytavern.code-profile`，从左下角齿轮 → Profiles → Import Profile 导入
6. 安装中文插件：侧边栏 Extensions 搜索 `Chinese` 安装
7. 安装代码依赖：终端执行 `pnpm install`

> **其他 AI 编程助手推荐：**
> - **Kilo Code**（推荐）：有中文，改自 RooCode，支持自定义 API
> - **Claude Code**（推荐）：Anthropic 编程助手，有 claude pro 账号即可使用
> - **Codex**（推荐）：OpenAI 编程助手，据说目前最好用
> - Augment、Cline、RooCode、Gemini CLI 等也都支持

### 1.6 下载编写模板

从 https://github.com/StageDog/tavern_helper_template 下载模板（Code → Download ZIP），用 bandizip 以管理员身份解压，通过 Cursor 打开文件夹。

> **重要：** 解压时请以管理员身份运行解压软件，否则 `.mcp.json` 等文件可能解压异常。

## 第二部分：核心避坑点

### 2.1 环境相关

| 问题 | 解决方案 |
|------|---------|
| 符号链接创建失败 | 必须在组策略中为当前用户启用「创建符号链接」权限 |
| `.mcp.json` 只显示一行 | 解压时未以管理员身份运行，重新解压 |
| Cursor 齿轮没有显示 `SI` | 通过 Profiles 切换到 sillytavern 配置文件 |
| `pnpm install` 失败 | 确认 NodeJS 版本 ≥ 24，确认已配置国内镜像源 |
| Cursor 变蠢了 | 尝试其他 AI 编程助手（Kilo Code、Claude Code、Codex） |

### 2.2 前端界面/脚本相关

| 问题 | 解决方案 |
|------|---------|
| 打包失败 | 让 AI 检查错误原因并修复，或复制终端错误信息发给 AI |
| 前端界面渲染卡顿 | 将 html 转为链接加载，正则代码块里只放链接，减少酒馆渲染负担 |
| `pnpm watch` 的打包结果很大 | **发布时用 `pnpm build`，不要用 `pnpm watch`**（watch 文件大数十倍） |
| AI 不阅读规则 | 在提示词中强调「阅读现有规则」 |
| AI 不看酒馆页面 | 在提示词中强调「通过 MCP 访问酒馆网页」 |

### 2.3 MVU 变量卡相关

| 问题 | 解决方案 |
|------|---------|
| 变量管理器中没有 `stat_data` | 确认角色卡有第一条消息、导入了 MVU 脚本和变量结构脚本、`[initvar]` 条目处于禁用状态、API 选择聊天补全 |
| 日志中没有「变量结构注册成功」 | 变量结构脚本有错误，复制日志发给 AI 重新生成 |
| 出现红色「变量初始化失败」 | `[initvar]` 初始值有错误，复制日志发给 AI 重新生成 |
| AI 回复没有 `<UpdateVariable>` | 检查变量提示词是否已添加且为蓝灯状态 |
| 出现 `<UpdateVariable>` 但变量没更新 | 检查提示词模板是否开启、酒馆助手宏是否禁用、token 数是否正常 |
| 变量重复更新 | 修改正则让 AI 知道以前的更新情况；创建日志变量记录玩家操作 |
| 状态栏没更新但变量正常 | 检查 html 上下是否添加了 ` ``` ` 代码块标记 |
| 不要用 `{{user}}` 等宏作为变量名 | 改用 `主角` 等无需宏的变量名 |

### 2.4 正则相关

- **前端界面正则唯一要做的是定位前端界面应该在的位置，而不是处理输出数据**
- 复杂数据处理应在前端界面代码中通过 `getChatMessages` 获取消息后处理

## 第三部分：核心工具经验——实时编写角色卡、世界书或预设

### 3.1 工具简介

`tavern_sync` 脚本将酒馆角色卡、世界书或预设映射为本地文件，支持：
- **实时监听**：本地文件修改自动同步到酒馆，无须复制粘贴
- **本地打包**：直接将角色卡打包成图片、世界书/预设打包成 JSON 文件

### 3.2 安装

**本地服务端：**
- 需要 NodeJS 24+
- 下载脚本：https://gitgud.io/StageDog/tavern_sync/-/raw/main/dist/tavern_sync.mjs?inline=false

**酒馆客户端：**
1. 安装酒馆助手
2. 在酒馆助手设置 → 脚本 → 新建，内容填写：
```ts
import 'https://testingcf.jsdelivr.net/gh/StageDog/tavern_sync/dist/script.js'
```

**编辑器：** 推荐按教程配置 Cursor，有语法高亮、自动保存、文件历史等功能，还能让 AI 帮你修改整个世界书/预设。

### 3.3 使用流程

1. **配置**：编辑 `tavern_sync.yaml`，指定角色卡名称和本地路径
2. **提取**：`node tavern_sync.mjs pull 配置名称`
3. **修改**：直接编辑本地文件（YAML 格式）
4. **同步**：`node tavern_sync.mjs push 配置名称 -f`
5. **实时监听**：`node tavern_sync.mjs watch 配置名称 -f`
6. **打包**：`node tavern_sync.mjs bundle 配置名称`

### 3.4 在编写模板中使用

编写模板已集成同步脚本：
- `pnpm build` 时自动运行 `bundle all` 打包所有配置
- `pnpm watch` 时自动运行 `watch all -f` 实时监听所有配置

### 3.5 用法提示

- `条目`中每个 `-` 块对应一个世界书条目，顺序即为酒馆世界书显示顺序
- 调整条目顺序：直接移动 `-` 块
- 新建条目：复制粘贴一块 `-`，填入设置和文件路径
- 删除条目：直接删掉对应 `-` 块

### 3.6 特殊功能——AI 直接修改世界书/预设

配置好 MCP 后，AI 编程助手可以直接：
- 读取本地文件中的世界书/预设内容
- 自行修改并保存
- 通过实时监听自动同步到酒馆

## 第四部分：核心工具经验——实时编写前端界面或脚本

### 4.1 脚本 vs 前端界面 vs 流式楼层界面

| 类型 | 特点 | 适用场景 |
|------|------|---------|
| **脚本** | 后台运行，可监听事件 | 修改酒馆行为、添加设置项、监听消息 |
| **前端界面** | 在消息楼层中显示界面 | 状态栏、Galgame 界面、互动组件 |
| **流式楼层界面** | 脚本实现，AI 回复时即可显示 | 需要流式显示的 Galgame 界面 |

### 4.2 完整编写流程

1. **创建项目**：将 `初始模板/前端界面/新建为src文件夹中的文件夹` 复制到 `src` 文件夹，重命名
2. **让 AI 编写**：发送提示词给 AI 编程助手
3. **打包**：`pnpm build` 打包到 `dist` 文件夹
4. **导入酒馆**：将打包结果用代码块包裹（上下加 ` ``` `），复制到酒馆正则的「替换为」部分

### 4.3 实现实时更新

1. **自动打包**：运行 `pnpm watch` 监听文件变化自动打包
2. **实时监听**：酒馆助手 → 开发 → 实时监听 → 启用「允许监听」
3. **Go Live**：点击 Cursor 右下角 Go Live 将文件夹映射为网络链接
4. **正则中使用链接**：
```html
<body>
  <script>
    $('body').load('http://localhost:5500/dist/项目名/index.html')
  </script>
</body>
```

### 4.4 让 AI 看到酒馆页面

**推荐方案：Chrome DevTools MCP**
1. 安装最新版 Chrome
2. 访问 `chrome://inspect/#remote-debugging`，勾选「Allow remote debugging」
3. **完全关闭浏览器或重启电脑**
4. MCP 已在模板中配置好

AI 可以：读取网页显示、控制台日志、网络情况；点击网页、填写表单；像素级复刻网页样式。

**备选方案：Browser MCP**
- 需安装浏览器扩展
- 只能读取不能操控

### 4.5 发布打包结果

**注意：发布用 `pnpm build`，开发用 `pnpm watch`！**

- 前端界面：`.html` 文件内容用代码块包裹，复制到酒馆正则
- 脚本：`.js` 文件内容复制到酒馆助手脚本库的「脚本内容」部分

### 5.2 前端界面正则推荐写法

**核心原则：正则只定位，不处理数据**

```yaml
脚本名称: [界面]状态栏
查找正则表达式: <status>.*</status>  # 正则不需要在意里面具体是什么
替换为: 前端界面的 html 代码块
作用范围:
  - [x] 用户输入
  - [x] AI输出
短暂:
  - [x] 仅格式显示
```

在代码中用 `getChatMessages` 获取整条消息后自行解析：

```ts
const chat_message = getChatMessages(getCurrentMessageId())[0];
const match = chat_message.message.match(/<status>(.*)<\/status>/);
```

### 5.3 避免渲染卡顿

酒馆渲染代码块是性能瓶颈。**解决方案：** 将 html 转为链接，正则代码块里只加载链接。

### 5.4 发布会自动更新的前端界面/脚本

- 上传代码到 GitHub 仓库
- 通过 `https://testingcf.jsdelivr.net/gh/组织名/仓库名/路径` 访问
- 编写模板支持 GitHub 自动工作流，自动打包到 `dist` 文件夹

**缓存刷新方式：**

- 换镜像域名：`testingcf`、`fastly`、`gcore`

### 5.5 发布会自动更新的角色卡

### 5.6 制作流式或同层界面

**同层前端界面：**
- 玩家在一个界面内游玩
- 通过 `generate` 函数触发 AI 回复
- 监听 `STREAM_TOKEN_RECEIVED_FULLY` 获取流式文本

### 5.7 修改发送出的提示词

监听事件（按先后顺序）：
- `GENERATE_AFTER_COMBINE_PROMPTS`
- `CHAT_COMPLETION_PROMPT_READY`
- `GENERATE_AFTER_DATA`（提示词模板、酒馆助手宏发生在这里）
- `CHAT_COMPLETION_SETTINGS_READY`

```ts
eventOn(tavern_events.CHAT_COMPLETION_PROMPT_READY, (event_data) => {
  _.remove(event_data.messages, (message => message.role === 'user'));
});
```

> **注意：** 必须使用原地修改函数（lodash 的原地修改函数、数组的 `splice`/`push` 等），不要用赋值方式替换整个数组。

## 第七部分：MVU 变量卡开发全流程

### 7.1 MVU 变量框架简介

MVU（MagVarUpdate）是一个酒馆助手脚本，用于：
- 定义变量结构（用 zod 库）
- 初始化变量
- 让 AI 通过 `<UpdateVariable>` 命令更新变量
- 提供变量管理器查看和修改变量

### 7.2 安装插件和导入脚本

1. 安装酒馆助手插件
2. 安装提示词模板插件
3. 导入 MVU 脚本（从内置库或 GitHub）

### 7.3 设置和初始化变量

#### 思考需要什么变量

根据角色卡类型设计变量：
- 恋爱卡：好感度、阶段、关系状态
- 冒险卡：背包物品、技能、状态
- 系统卡：任务列表、商城商品、积分

#### 设计变量结构

使用门之主写卡助手的「变量结构设计（脚本）」功能，输入变量需求让 AI 生成。

**变量结构脚本基本结构：**

```js
import { registerMvuSchema } from 'https://testingcf.jsdelivr.net/gh/StageDog/tavern_resource/dist/util/mvu_zod.js';

export const Schema = z.object({
  // 变量定义
});

$(() => {
  registerMvuSchema(Schema);
});
```

#### Zod 类型速查

| 类型 | 用途 | 示例 |
|------|------|------|
| `z.object({...})` | 对象（文件夹） | `白娅: z.object({ 依存度: z.number() })` |
| `z.string()` | 文本 | `z.string()` |
| `z.number()` | 数值 | `z.number()` |
| `z.coerce.number()` | 数值（自动转换文本） | **推荐使用**，防止 AI 把数值写成文本 |
| `z.boolean()` | 真假值 | `z.boolean()` |
| `z.enum([...])` | 枚举文本 | `z.enum(['进行中', '已失败'])` |
| `z.record(键, 值)` | 字段不固定的对象 | 物品栏、能力面板 |
| `z.partialRecord(...)` | 可选字段的 record | 羁绊系统 |
| `z.union([...])` | 多种类型 | 数值或「待初始化」 |
| `z.templateLiteral([...])` | 格式化文本 | `D1.C1.E1.S2` 格式 |
| `.transform(v => ...)` | 转换值 | `_.clamp(v, 0, 100)` |
| `.prefault(值)` | 默认值 | AI 未输出时使用默认值 |
| `.describe('描述')` | 描述字段用途 | 方便理解变量含义 |

#### 初始化变量

在角色卡世界书中新建条目：
- 名称包含 `[initvar]`
- **必须处于禁用状态**
- 内容为 YAML 格式的变量初始值

```yaml
白娅:
  依存度: 35
  着装:
    上装: 整洁的深蓝色校服外套
    下装: 深蓝色百褶裙
```

#### 为不同开局设置不同初始值

在开局消息中用 `<initvar>...</initvar>` 包裹：

```
开局故事...

<UpdateVariable>
<initvar>
世界:
  当前时间: 2025-04-07
白娅:
  依存度: 15
</initvar>
</UpdateVariable>
```

- 有 `<initvar>` 时忽略 `[initvar]` 条目
- 无 `<initvar>` 时使用 `[initvar]` 条目

### 7.4 变量提示词

变量提示词包含三个部分：

1. **变量列表**：告诉 AI 有什么变量、当前值是什么
2. **变量更新规则**：告诉 AI 在什么情况下更新变量、更新成什么值
3. **变量输出格式**：告诉 AI 应该输出什么来更新变量

**变量提示词写法示例（UpdateVariable 格式）：**

```xml
<UpdateVariable>
<Analysis>$(IN ENGLISH, no more than 80 words)
- ${calculate time passed: ...}
- ${decide whether dramatic updates are allowed: yes or no}
- ${what variables should be updated, according only to current reply: ...}
</Analysis>
<JSONPatch>
[
  { "op": "replace", "path": "${/path/to/variable}", "value": "${new_value}" },
  { "op": "delta", "path": "${/path/to/number}", "value": "${delta}" },
  { "op": "insert", "path": "${/path/to/key}", "value": "${new_value}" },
  { "op": "remove", "path": "${/path/to/item}" }
]
</JSONPatch>
</UpdateVariable>
```

**思维链写法选择：**

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| AI 自己搞定 | 不写 check 规则，让 AI 自行分析 | 变量很少 |
| 回忆检查 | 让 AI 回忆 check 规则后分析 | 一般场景 |
| 依次检查 | 让 AI 原封不动列举所有 check | 需要精确 |
| 周全列举 | 将列举和分析隔离 | 防止 AI 偷懒 |

### 7.5 脚本控制变量

可以通过脚本直接修改变量，而不依赖 AI 输出：

```ts
import { getMvuVariable, setMvuVariable } from '酒馆助手接口';

// 获取变量
const value = getMvuVariable('白娅.依存度');

// 设置变量
setMvuVariable('白娅.依存度', 50);
```

### 7.6 状态栏

状态栏是前端界面的一种，用于显示当前变量情况。

**关键点：**
- 状态栏代码需要放在 ` ``` ` 代码块中
- 变量管理器里变量正常更新但状态栏没更新 → 检查代码块标记
- 电脑端用实时编写方案让 AI 自己看网页修改，成功率高
- 手机端不要让 AI 修改原本就有错的状态栏，最好重新生成

### 7.7 迁移旧 MVU 卡

如果从旧版 MVU 迁移到 MVU zod：
1. 用新版变量结构脚本替换旧版
2. 变量初始值格式保持一致
3. 测试变量更新是否正常

### 7.8 MVU 会让 AI 变蠢吗？

三种提示词不可避免地占用 token：
- 变量列表
- 变量更新规则
- 变量输出格式

**减少影响的方法：**
- 用 `_` 前缀标记变量为「AI 不可更新」
- 用 `$` 前缀标记变量为「AI 不可见」
- 省去三种提示词中的任何一种都不可能
- 可以在剧情 AI 输出后，另外调用 AI 来进行变量更新

### 7.9 常见问题排查清单

1. 新开聊天 → 打开变量管理器和日志查看器
2. 从上到下检查：
   - 变量管理器中没有 `stat_data` → 检查脚本导入和 API 设置
   - 没有「变量结构注册成功」→ 变量结构脚本有错
   - 红色「变量初始化失败」→ initvar 有错
   - 没有 `<UpdateVariable>` → 检查变量提示词
   - 有 `<UpdateVariable>` 但没更新 → 检查提示词模板和宏设置
   - 黄色「变量更新错误」→ 变量结构脚本有问题

## 第八部分：提示词个人写法

### 8.1 额外输出格式

AI 输出的额外格式（如状态栏数据、变量更新命令）应该：
- 用明确的标签包裹（如 `<status>...</status>`、`<UpdateVariable>...</UpdateVariable>`）

- 用正则捕获并在需要时替换显示

## 第九部分：酒馆如何处理世界书

### 9.1 激活

世界书条目的激活方式：
- **蓝灯**（常驻激活）：始终发送给 AI
- **绿灯**（条件激活）：仅在关键词匹配时激活
- **禁用**：不发送给 AI（如 `[initvar]` 条目）

### 9.2 插入

- 条目按优先级排序后插入提示词
- 角色卡的世界书条目优先级高于全局世界书
- 同优先级按条目顺序排列

### 9.3 额外处理

- 提示词模板可以对世界书条目进行额外处理
- 酒馆助手宏在 `GENERATE_AFTER_DATA` 事件中展开

## 第十部分：生成立绘、表情差分和 CG

### 10.1 工具准备

- **ForgeUI**：基于 ComfyUI 的 SD WebUI，易于上手
- **Photoshop**：用于换脸和抠图

### 10.2 推荐模型和 LORA

- 模型：waiNSFWIllustrious_v130.safetensors
- LORA：wan_flat_color_1.3b_v2（注意：这是视频 LORA，但立绘效果更好）

### 10.3 生成设置

- 采样方法：Euler a
- 调度类型：Karras
- 迭代步数：28
- 分辨率：1024x1536
- Distilled CFG Scale：3.5
- CFG Scale：4.5

### 10.4 立绘提示词模板

```
<LORA:wan_flat_color_1.3b_v2:1>,flat color,no lineart,...,
[人物与服装提示词],
solo,(upper body),cowboy shot,standing,((green background)),((green screen)),simple background,(((tachi-e))),((((facing viewer)))),light smile,eye contact,looking at viewer,
```

### 10.5 表情差分

1. 将立绘发送到「局部重绘」
2. 用画笔覆盖脸部（注意避开发卡等不变的部分）
3. 修改表情提示词（如 `anger`、`crying`）
4. 重绘幅度：65 左右
5. 勾选「柔和重绘」
6. 修复错误：再次重绘，降低重绘幅度到 0.35

### 10.6 动作差分

与表情差分类似，但需要覆盖更大的区域（如手部），重绘幅度可能需要 0.7-0.85。

### 10.7 换衣服

1. 重绘覆盖脸部，选择「重绘非蒙版区域」
2. 更换服装提示词
3. 在 Photoshop 中将新旧图层叠加，用橡皮擦擦除新图层的脸部

### 10.8 抠图

Photoshop 中使用「选择主体」→「选择并遮住」→ 调整边缘 → 输出为透明背景。

> **提示：** 也可以用 ForgeUI 内置的大模型抠图功能，或在线抠图网站。
